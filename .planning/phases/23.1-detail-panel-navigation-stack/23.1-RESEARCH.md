# Phase 23.1: Detail Panel Navigation Stack - Research

**Researched:** 2026-04-02
**Domain:** React state management (navigation stack) + CSS transitions (directional slide)
**Confidence:** HIGH

## Summary

This phase adds browser-like back-navigation to the existing DetailPanelSlot. Every entity or cluster viewed is pushed onto a stack in the uiStore. Users navigate backwards via a breadcrumb row with directional slide transitions. Closing the panel clears the stack entirely.

The implementation is entirely frontend -- no server changes, no new dependencies. The core challenge is threading a `pushView` action through every call site that currently calls `selectEntity` / `setSelectedCluster`, and wrapping the panel content area in a directional CSS transition. The existing codebase uses plain Zustand (no immer), Tailwind v4 CSS-first configuration, and CSS `transition-transform` for the panel slide-in/out. The navigation stack follows the same patterns.

**Primary recommendation:** Add `navigationStack`, `pushView`, `goBack`, and `clearStack` to `uiStore.ts`. Create a `BreadcrumbRow` component and a `DirectionalTransition` wrapper inside `DetailPanelSlot`. Thread `pushView` through all 8 call sites that trigger entity/cluster selection.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- Universal history -- every entity click (map or panel drill-down) pushes onto the stack
- Map clicks push onto the existing stack (not a fresh start)
- Unlimited depth, no cap on stack entries
- Closing the panel (x button or Escape key) clears the entire stack
- No forward navigation button -- back-only with discard-forward-history (browser fork behavior)
- Stack entries are generic "panel view" objects -- no typed distinction between clusters, flights, ships, events, sites
- Lost-contact entities remain in the stack -- navigating back shows them with existing LOST CONTACT banner and grayscale treatment
- No depth count badge on the back button -- just a plain arrow
- Thin breadcrumb row below the header, appears on demand when stack has entries (not always reserved)
- Shows previous entries only -- current view is NOT shown in the breadcrumb (header already displays it)
- Labels: type + name, truncated with ellipsis for long names (e.g., "Cluster(5)" > "Airstrike" > "IR-721")
- Styling: uniform muted gray text, no entity color dots, `>` chevron separators
- Interaction: left-arrow goes back one step; last (most recent) segment is also clickable; deeper segments are display-only
- Overflow: horizontal scroll when breadcrumb exceeds panel width (360px)
- Entrance/exit: slides in/out vertically when first entry pushed / stack emptied
- Directional slide between views: drill-in slides content left (new from right), back slides content right (previous from left)
- Duration: 150ms (fast, snappy)
- Map fly-to and panel slide transition happen simultaneously (no waiting)

### Pattern 1: Stack Push Before Selection

**What:** Every call site that opens a new entity/cluster in the detail panel must push the *current* view onto the stack before changing `selectedEntityId` or `selectedCluster`.

**When to use:** Every time `selectEntity(id)` or `setSelectedCluster(cluster)` is called while the detail panel is already open with content.

**Example:**
```typescript
// In uiStore.ts
pushView: (view) => set((s) => ({
  navigationStack: [...s.navigationStack, view],
  slideDirection: 'forward',
})),
```

**Call site pattern (e.g., ThreatClusterDetail.tsx):**
```typescript
const handleEventClick = (eventId: string, lat: number, lng: number) => {
  // Push current cluster view onto stack BEFORE selecting new entity
  const { selectedCluster, isDetailPanelOpen, selectedEntityId } = useUIStore.getState();
  if (isDetailPanelOpen && (selectedCluster || selectedEntityId)) {
    const label = selectedCluster
      ? `Cluster(${selectedCluster.eventCount})`
      : deriveLabel(currentEntity);
    useUIStore.getState().pushView({
      entityId: selectedEntityId,
      cluster: selectedCluster,
      breadcrumbLabel: label,
    });
  }
  selectEntity(eventId);
  openDetailPanel();
  setFlyToTarget({ lng, lat, zoom: 10 });
};
```

### Pattern 2: GoBack Restores Previous View

**What:** `goBack` pops the last entry from the stack and restores either `selectedEntityId` or `selectedCluster` from it.

**Example:**
```typescript
goBack: () => set((s) => {
  const stack = [...s.navigationStack];
  const prev = stack.pop();
  if (!prev) return {};
  return {
    navigationStack: stack,
    selectedEntityId: prev.entityId,
    selectedCluster: prev.cluster,
    slideDirection: 'back',
  };
}),
```

### Pattern 3: Directional Slide Transition

**What:** Content area slides left (forward) or right (back) using CSS transitions driven by a `slideDirection` state value.

**When to use:** On every view change (entity selection or goBack).

**Example approach using key-based remount + CSS animation:**
```typescript
// In DetailPanelSlot.tsx
const slideDirection = useUIStore((s) => s.slideDirection);
const viewKey = selectedEntityId ?? selectedCluster?.id ?? 'empty';

// CSS classes applied to content wrapper
const enterClass = slideDirection === 'forward'

  ? 'animate-slide-in-left'
  : '';
```

### Pattern 4: Breadcrumb Row

**What:** A thin, horizontally scrollable row below the header showing previous stack entries as truncated labels with chevron separators.

**Example:**
```typescript
// BreadcrumbRow.tsx
function BreadcrumbRow({ stack, onBack, onClickLast }: Props) {
  if (stack.length === 0) return null;
  return (
    <div className="flex items-center gap-1 px-4 py-1.5 border-b border-border
                    overflow-x-auto text-[10px] text-text-muted whitespace-nowrap
                    animate-breadcrumb-enter">
      <button onClick={onBack} aria-label="Go back" className="...">
        <ArrowLeftIcon />
      </button>
      {stack.map((entry, i) => (
        <Fragment key={i}>
          {i > 0 && <span className="text-text-muted/50">&gt;</span>}
          {i === stack.length - 1 ? (
            <button onClick={onClickLast} className="hover:text-text-secondary truncate max-w-[120px]">
              {entry.breadcrumbLabel}
            </button>
          ) : (
            <span className="truncate max-w-[100px]">{entry.breadcrumbLabel}</span>
          )}
        </Fragment>
      ))}
    </div>
  );
}
```

### Anti-Patterns to Avoid
- **Storing entity data in the stack:** Only store `entityId` for entity views. The entity is looked up from stores when navigating back. Clusters are the exception since they're transient (not in any store).
- **Re-creating the panel open/close animation on back:** The panel stays open; only the *content* slides. The 300ms panel translate-x animation only fires on open/close.
- **Mutating the stack array:** Always spread-copy in Zustand set calls (`[...s.navigationStack, view]`).
- **Clearing stack on every new entity selection:** Stack only clears on panel close (dismiss or Escape). New map clicks push to the existing stack.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Breadcrumb overflow | Custom truncation-from-left logic | `overflow-x-auto` + `whitespace-nowrap` on container | CSS handles horizontal scroll natively; user scrolls to see older entries |
| Slide transitions | A full animation framework (framer-motion) | CSS `@keyframes` + React `key` prop remount | 150ms single-direction slide is trivially expressible in CSS; key changes trigger animation on mount |
| Label derivation | Complex type-switch per entity type | Extract existing `getTypeLabel` + `getEntityName` from `DetailPanelSlot.tsx` into a shared util | Already written -- just needs to be exported |

**Key insight:** The entire feature is a state management problem (stack + 8 wired call sites) with a thin UI layer (breadcrumb + CSS animation). No new dependencies needed.

## Common Pitfalls

### Pitfall 1: Mutual Exclusion Violation
**What goes wrong:** `selectEntity` clears `selectedCluster` and vice versa. If `goBack` sets both, or if `pushView` doesn't snapshot the right field, the mutual exclusion logic in `set` silently drops one.
**Why it happens:** The existing `selectEntity: (id) => set({ selectedEntityId: id, selectedCluster: null })` pattern force-clears the other field.
**How to avoid:** `goBack` must set BOTH fields atomically in a single `set()` call that bypasses the mutual-exclusion wrappers. Directly set `selectedEntityId` and `selectedCluster` rather than calling `selectEntity`/`setSelectedCluster`.
**Warning signs:** Navigating back to a cluster shows empty panel (cluster was nulled by `selectEntity`).

### Pitfall 2: Lost Contact on Back-Navigation
**What goes wrong:** User views Flight A -> clicks Event B -> Flight A disappears from store -> navigates back. `useSelectedEntity` returns `isLost: true` using its `useRef` last-known mechanism. But the ref was overwritten when Event B was selected.
**Why it happens:** `useSelectedEntity` holds only ONE last-known entity in its ref.
**How to avoid:** The stack stores `entityId` only. When `goBack` restores the ID, `useSelectedEntity` will find it in the store (if still alive) or show lost contact (if gone). The `useRef` in `useSelectedEntity` will capture it on the frame it's still found. If the entity was already gone before the user navigated away, the last-known ref already held it -- but then the stack was pushed and a new entity selected, overwriting the ref. **Solution:** The `PanelView` can optionally cache a snapshot label for the breadcrumb, but the actual entity lookup always goes through `useSelectedEntity` which handles lost-contact gracefully. If the entity is truly gone, the user sees "LOST CONTACT" -- this is the correct behavior per user decisions.
**Warning signs:** Back-navigating to a gone entity shows empty panel instead of LOST CONTACT banner.

### Pitfall 3: Re-render Storm from Stack Array
**What goes wrong:** Every component subscribing to `useUIStore` re-renders when the stack changes, even if they don't use it.
**Why it happens:** Zustand shallow equality check fails on new array reference.
**How to avoid:** Use granular selectors: `useUIStore((s) => s.navigationStack)` only in `BreadcrumbRow` and `DetailPanelSlot`. Other components should NOT select the stack. The `slideDirection` selector is similarly isolated.
**Warning signs:** Map layer re-renders on every entity click.

### Pitfall 4: Empty Stack on First View
**What goes wrong:** User clicks first entity on map. `pushView` is called but there's no "current view" to push.
**Why it happens:** The panel is closed (or has no entity/cluster) so there's nothing to push.
**How to avoid:** `pushView` should only be called when the panel is already open AND has an active entity or cluster. Guard: `if (isDetailPanelOpen && (selectedEntityId || selectedCluster))`.
**Warning signs:** Stack has a `null`/empty entry as first item.

### Pitfall 5: Stack Grows Unboundedly in Long Sessions
**What goes wrong:** User clicks 500 entities in a long session; stack array has 500 full ThreatCluster objects.
**Why it happens:** Unlimited depth (per user decision) + clusters can be large objects.
**How to avoid:** This is acceptable per the user's "unlimited depth, no cap" decision. Clusters with many cells could be moderately large but are transient. In practice, users are unlikely to drill through 100+ entities. No mitigation needed per spec, but could add a soft limit (e.g., 200) if memory becomes an issue.
**Warning signs:** None expected in practice.

### Pitfall 6: Breadcrumb Label Stale After goBack
**What goes wrong:** User goes back to a flight whose callsign changed since it was pushed.
**Why it happens:** The breadcrumb label is snapshotted at push time.
**How to avoid:** This is actually correct -- the label represents what the entity was when viewed. Don't try to make it reactive. The actual detail panel content IS reactive (via `useSelectedEntity`), so the header updates even if the breadcrumb shows the old label.
**Warning signs:** None -- snapshotted labels are the expected behavior.

### Pitfall 7: Escape Key Should Clear Stack
**What goes wrong:** User presses Escape, panel closes, but stack is preserved. Next time they open a new entity, the old stack is still there.
**Why it happens:** `closeDetailPanel` doesn't call `clearStack`.
**How to avoid:** Wire `clearStack` into the dismiss callback and the Escape key handler. Both `dismiss` in `DetailPanelSlot` and priority 5 in `useEscapeKeyHandler` must call `clearStack()`.
**Warning signs:** Opening a new entity shows stale breadcrumb from previous session.

## Code Examples

### Stack State in uiStore

```typescript
// In uiStore.ts - new state fields and actions
navigationStack: [] as PanelView[],
slideDirection: null as 'forward' | 'back' | null,

pushView: (view) => set((s) => ({
  navigationStack: [...s.navigationStack, view],
  slideDirection: 'forward',
})),

goBack: () => set((s) => {
  const stack = [...s.navigationStack];
  const prev = stack.pop();
  if (!prev) return {};
  return {
    navigationStack: stack,
    // Bypass mutual exclusion -- set both directly
    selectedEntityId: prev.entityId,
    selectedCluster: prev.cluster,
    slideDirection: 'back',
  };
}),

clearStack: () => set({
  navigationStack: [],
  slideDirection: null,
}),
```

### Helper: Derive Breadcrumb Label

```typescript
// Extract from DetailPanelSlot.tsx into shared util, e.g. src/lib/panelLabel.ts
export function deriveBreadcrumbLabel(
  entity: MapEntity | SiteEntity | null,
  cluster: ThreatCluster | null,
): string {
  if (cluster) return `Cluster(${cluster.eventCount})`;
  if (!entity) return '';
  const typeLabel = getTypeLabel(entity.type);
  const name = getEntityName(entity);
  return name ? `${typeLabel} ${name}` : typeLabel;
}
```

### Call Site Threading (BaseMap example)

```typescript

// AFTER:

  // Snapshot current view before overwriting

    entityId: state.selectedEntityId,
    cluster: state.selectedCluster,

```

```typescript

  setSelectedCluster(null);
```

## Open Questions

1. **Entity lookup for breadcrumb label at push time**
   - What we know: When pushing the current view to the stack, we need the current entity to derive a breadcrumb label. The entity is available in the component calling `pushView` (e.g., `useSelectedEntity` in `DetailPanelSlot`), but many call sites are outside the panel.
   - What's unclear: Whether to create a shared helper that looks up the entity from stores, or pass it from the component context.
   - Recommendation: Create `getCurrentPanelView(): PanelView | null` helper that reads from stores directly (similar to how `useEscapeKeyHandler` reads from `useUIStore.getState()`). This avoids prop-drilling.

2. **Fly-to on goBack**
   - What we know: User decisions say "Map fly-to and panel slide transition happen simultaneously." Forward navigation already flies to the new entity.
   - What's unclear: Should `goBack` also fly the map to the previous entity's coordinates?
   - Recommendation: Yes -- `goBack` should fly to the restored entity/cluster coordinates. For clusters, fly to `centroidLat/centroidLng`. For entities, look up from stores or use the stored coordinates. This matches the simultaneous behavior.
