# Phase 19.1: Advanced Search with Tag and Entity Type Filtering - Context

**Gathered:** 2026-03-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Enhance the existing Cmd+K search modal (Phase 19) to support structured tag-based queries with syntax highlighting, autocomplete, and bidirectional sync with sidebar filter controls. Search bar and sidebar filters are two UIs for the same underlying filter state.

</domain>

<decisions>
## Implementation Decisions

### Query Syntax
- Hybrid model: clickable type chips above the input that insert tags, plus direct tag typing in the input
- Tags rendered as syntax-highlighted colored text in the input (not pill tokens) — cursor behavior stays natural
- Entity type chips displayed above the search input in a horizontal row
- Default chip set: [Events] [Sites] [Iran] [US] (no dots, no bullet indicators)
- Recent tags (from last 5 searches) appended after the 4 default chips
- Plain text (no tag prefix) still works as freeform substring search (backward compatible with Phase 19)

### Tag Vocabulary
Full tag prefix list (all included except `source:`):

**Cross-entity:**
- `type:` — entity type (flight, ship, airstrike, ground_combat, site, etc.)
- `country:` — flight originCountry, GDELT actor countries, news sourceCountry
- `actor:` — GDELT actor1/actor2 names
- `location:` — GDELT locationName, site labels
- `severity:` — high/medium/low from Phase 17 scoring system
- `near:` — entities within proximity of a named site (reuse Phase 17 haversine, 50km)
- `since:` / `before:` — temporal filtering (since:6h, before:2026-03-20)
- `has:` — attribute presence check (has:callsign, has:actor2)

**Flight-specific:**
- `callsign:` — flight callsign/label
- `icao:` — ICAO24 hex code
- `altitude:` — barometric altitude (supports range operators >/<)
- `speed:` — ground speed (supports range operators, shared with ships)
- `squawk:` — transponder squawk code
- `ground:` — on-ground boolean (true/false)
- `vertical:` — vertical rate (climbing/descending)
- `unidentified:` — no-callsign flights (true/false)

**Ship-specific:**
- `mmsi:` — MMSI number
- `heading:` — ship heading (supports range operators)
- `shipname:` — vessel name

**Event-specific:**
- `cameo:` — GDELT CAMEO event code
- `goldstein:` — Goldstein conflict scale (supports range operators)
- `mentions:` — number of mentions (supports range operators)
- `date:` — event date

**Site-specific:**
- `site:` — site type (nuclear, naval, oil, airbase, desalination, port)
- `status:` — attack status (healthy, attacked)

### Negation
- `!` prefix on tag values for exclusion: `!country:iran`, `!type:ship`

### Filter Combination Logic
- User can use explicit `AND` / `OR` keywords (uppercase) between tag expressions
- Implicit operator (no keyword between tags) defaults to AND
- Parentheses supported for grouping: `type:flight AND (country:iran OR country:iraq)`
- Full boolean expression parsing required

### Search ↔ Sidebar Sync (Critical)
- Search bar and sidebar filter controls are TWO UIs for ONE filter state
- Typing `type:flight` in search bar activates Flights toggle in sidebar
- Toggling Flights ON in sidebar adds `type:flight` to search query
- Toggling a filter OFF removes the corresponding tag from search (no negation shown — only active filters appear as tags)
- Date range slider syncs bidirectionally with `since:`/`before:` tags
- Every search tag should have a corresponding sidebar filter control — if one doesn't exist, build it
- `since:6h` snaps the date slider; dragging slider updates search bar with `since:`/`before:`
- Clear search resets all filters to defaults

### Autocomplete & Discovery
- Two-stage autocomplete dropdown:
  - Stage 1: typing partial prefix suggests matching tag names (e.g., "co" → `country:`, `callsign:`)
  - Stage 2: after completing prefix (e.g., `country:`), suggests known values from current live entity data with counts (e.g., "iran (24)", "iraq (18)")
- Tab accepts highlighted suggestion and keeps cursor in input for more typing
- Enter runs the full search/filter (does not accept suggestion)
- Arrow keys navigate dropdown suggestions
- `?` icon in search bar opens a cheat sheet popover showing all available tags with examples
- Cheat sheet dismisses on click-away

### Claude's Discretion
- Exact autocomplete dropdown styling and positioning
- How to extract unique values from entity stores for Stage 2 suggestions
- Internal query parser implementation (AST, regex, or other approach)
- How to handle invalid/unknown tag prefixes (ignore, highlight red, suggest correction)
- Performance optimization for real-time autocomplete against live entity data

</decisions>

<specifics>
## Specific Ideas

- "Search like email" — Gmail-style tag prefix syntax is the mental model
- Default chips should be [Events] [Sites] [Iran] [US] — the most common filtering needs for this conflict dashboard
- Recent tags persist from last 5 searches as additional chips
- Full bidirectional sync between search bar and sidebar filters is the core architectural principle — they are not separate systems

</specifics>

## Existing Code Insights

### Reusable Assets
- `searchStore.ts` — query, isFilterMode, matchedIds state (needs extension for structured queries)
- `searchUtils.ts` — `getSearchableFields()` already indexes per-entity-type fields (foundation for tag matching)
- `SearchModal.tsx` — Cmd+K modal with input, results, Enter-to-filter (extend, not replace)
- `FilterChip.tsx` — existing pill component for active filter display
- `useSearchResults.ts` — cross-store entity search with useRef optimization
- `attackStatus.ts` — haversine distance calculation (reuse for `near:` tag)
- `useProximityAlerts.ts` — 50km proximity detection from Phase 17 (pattern for `near:`)
- `severity.ts` — severity scoring formula (reuse for `severity:` tag thresholds)
- `useCounterData.ts` — derives visible entity counts from stores (pattern for autocomplete counts)

### Established Patterns
- Zustand curried `create<T>()()` pattern for all stores
- `useEntityLayers.ts` — SEARCH_DIM_ALPHA=15 for non-matching entities (already wired)
- `filterStore.ts` — dateStart/dateEnd state with clearAll (extend for tag sync)
- `uiStore.ts` — toggle state (showFlights, showAirstrikes, etc.) with localStorage persistence

### Integration Points
- `searchStore` needs restructuring: replace plain `query` string with parsed query AST
- `searchUtils.ts` needs full rewrite: substring matching → tag-aware boolean expression evaluator
- `useEntityLayers.ts` already consumes `matchedIds` — no change needed for rendering pipeline
- `uiStore` toggles need bidirectional binding to tag state
- `filterStore` date range needs bidirectional binding to `since:`/`before:` tags
- `SearchModal.tsx` needs chip row, syntax highlighting overlay, autocomplete dropdown, and ? help popover
