# Phase 08.1: Add GDELT as Default Conflict Event Source - Research

**Researched:** 2026-03-17
**Domain:** GDELT v2 Events API, server-side adapter, CSV parsing, ZIP decompression
**Confidence:** HIGH

## Summary

This phase replaces the ACLED conflict event adapter with a GDELT v2 events adapter on the server side. GDELT v2 publishes a `lastupdate.txt` file every 15 minutes containing URLs to three ZIP archives (events, mentions, GKG). We download only the `.export.CSV.zip` file, decompress it in memory, parse the tab-delimited rows, filter by CAMEO root code (18/19/20) and FIPS country code (Middle East), then normalize to the existing `ConflictEventEntity` type.

The frontend is completely untouched -- `useEventPolling`, `eventStore`, `useEntityLayers`, and `StatusPanel` all consume `ConflictEventEntity[]` and do not care about the upstream source. The only frontend change is updating the polling interval from 300s to 900s to match GDELT's 15-minute cycle. The route handler (`server/routes/events.ts`) changes its import from `acled.ts` to `gdelt.ts`, the cache TTL updates to 900s, and no config/env changes are needed since GDELT requires no authentication.

**Primary recommendation:** Create `server/adapters/gdelt.ts` that fetches lastupdate.txt, downloads the export ZIP, parses with `adm-zip`, filters rows, and returns `ConflictEventEntity[]`. Update the events route to import from gdelt instead of acled. Update `CACHE_TTL.events` to 900_000 and `EVENT_POLL_INTERVAL` to 900_000.

## Standard Stack

### Core
| Library | Version | Purpose | Why Standard |
|---------|---------|---------|--------------|
| adm-zip | latest | In-memory ZIP decompression | Simple synchronous API, buffer-in/buffer-out, 86KB files trivial |
| @types/adm-zip | ^0.5.7 | TypeScript definitions | DefinitelyTyped, well-maintained |

### Supporting
No additional libraries needed. The tab-delimited CSV parsing is trivial string splitting -- no CSV library warranted for a headerless, quote-free, tab-delimited format with known column positions.

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| adm-zip | Node.js zlib | zlib cannot decompress ZIP archives (only gzip/deflate). ZIP is a container format with its own header structure. zlib WILL NOT WORK. |
| adm-zip | yauzl | More complex callback-based API, designed for streaming large ZIPs. Overkill for ~86KB. |
| adm-zip | fflate | Faster but more complex API with two modes. Unnecessary for small files. |

### Pattern 1: GDELT Adapter (follows existing adapter pattern)
**What:** Server-side adapter that fetches, decompresses, parses, filters, and normalizes GDELT data to `ConflictEventEntity[]`
**When to use:** Called by the events route handler on each request (cache miss)
**Example:**
```typescript

  'IR', 'IZ', 'SY', 'TU', 'SA', 'YM', 'MU', 'AE',

// CAMEO root codes for conflict events

  // 1. Fetch lastupdate.txt
  // 2. Parse first line to get export CSV URL
  // 3. Download ZIP, decompress with AdmZip
  // 4. Parse tab-delimited rows
  // 5. Filter by EventRootCode + ActionGeo_CountryCode
  // 6. Normalize to ConflictEventEntity[]
```

### Pattern 2: lastupdate.txt Parsing
**What:** Parse the 3-line lastupdate.txt to extract the export CSV URL
**When to use:** Every fetch cycle
**Example:**
```typescript
// lastupdate.txt format (3 lines, space-delimited):
// 150383 297a16b493de7cf6ca809a7cc31d0b93 http://data.gdeltproject.org/gdeltv2/20260317120000.export.CSV.zip
// 86743 abc123... http://data.gdeltproject.org/gdeltv2/20260317120000.mentions.CSV.zip
// 432876 def456... http://data.gdeltproject.org/gdeltv2/20260317120000.gkg.CSV.zip
//
// We want the line containing ".export.CSV.zip"
const lines = text.trim().split('\n');
const exportLine = lines.find(l => l.includes('.export.CSV.zip'));
const exportUrl = exportLine!.split(' ')[2]; // third field is URL
```

### Pattern 4: Event Classification (CAMEO to EntityType)
**What:** Map CAMEO root codes to missile/drone entity types
**When to use:** During normalization
**Example:**
```typescript
function classifyByCAMEO(eventRootCode: string, eventCode: string): 'missile' | 'drone' {
  // 18x = Assault (terrorism, hijacking, physical assault) -> drone
  if (eventRootCode === '18') return 'drone';
  // 19x = Fight (conventional military force) -> missile
  if (eventRootCode === '19') return 'missile';
  // 20x = Unconventional mass violence (CBR, nuclear) -> missile
  if (eventRootCode === '20') return 'missile';
  // Fallback (should not reach here with proper filtering)
  return 'missile';
}
```

### Pattern 5: Deduplication by GlobalEventID
**What:** GDELT may report the same event from multiple news sources within a 15-min window. Each has a unique `GlobalEventID`, so no dedup is needed at the row level. However, across 15-min windows, the same real-world event may generate new rows with different GlobalEventIDs.
**Recommendation:** Do NOT implement cross-window deduplication (deferred). Each 15-min export is a complete replacement (same as ACLED's full-replace pattern). The `GlobalEventID` is unique within an export.

### Anti-Patterns to Avoid
- **Using Node.js zlib for ZIP files:** zlib handles gzip/deflate compression, NOT ZIP archives. ZIP is a container format. `zlib.unzip()` will throw "incorrect header check" on `.zip` files.
- **Parsing CSV with regex or split on comma:** GDELT files are TAB-delimited despite the `.CSV` extension. Always split on `\t`.
- **Using ISO country codes:** GDELT's ActionGeo_CountryCode uses FIPS 10-4, NOT ISO 3166. Iraq = "IZ" (not "IQ"), Turkey = "TU" (not "TR"), etc.
- **Filtering on EventCode with startsWith:** Use `EventRootCode` (column 29, 0-indexed 28) which is the clean 2-digit root. Avoids string prefix matching on variable-length codes.
- **Requiring config/env vars for GDELT:** GDELT is fully public, no auth. Do NOT add env vars to `server/config.ts`. The adapter should work with zero configuration.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| ZIP decompression | Manual ZIP header parsing with zlib inflate | `adm-zip` | ZIP format has local file headers, central directory, compression method flags. Hand-rolling is error-prone. |
| CSV parsing | Full CSV parser library | `line.split('\t')` | GDELT is simple tab-delimited with no quoting, no embedded tabs, no headers. A split is sufficient and zero-dependency. |
| Date parsing | Moment.js / date-fns | Manual YYYYMMDD parse | GDELT dates are always YYYYMMDD format. `new Date(s.slice(0,4), s.slice(4,6)-1, s.slice(6,8))` is trivial. |

**Key insight:** The GDELT adapter is conceptually simple: HTTP fetch -> unzip -> split lines -> split tabs -> filter -> normalize. The only non-trivial part is ZIP decompression, which adm-zip handles in one line.

## Common Pitfalls

### Pitfall 1: FIPS vs ISO Country Codes
**What goes wrong:** Using ISO 3166 codes (IQ, TR, IL) instead of FIPS 10-4 (IZ, TU, IS) causes zero events to match.
**Why it happens:** Developers assume country codes are ISO. GDELT predates the widespread adoption of ISO and uses the older US government FIPS standard.
**How to avoid:** Use the exact FIPS code set from CONTEXT.md. Test with known GDELT data.
**Warning signs:** Zero events returned from GDELT despite the Middle East having active conflicts.

### Pitfall 2: Tab vs Comma Delimiter
**What goes wrong:** Splitting on comma produces garbage data because fields contain commas (e.g., place names "Tehran, Tehran, Iran").
**Why it happens:** Files have `.CSV` extension but are actually tab-delimited.
**How to avoid:** Always split on `\t`. This is documented in the GDELT codebook.
**Warning signs:** Parsed rows have wrong number of columns (way too many).

### Pitfall 3: HTTP (not HTTPS) for GDELT URLs
**What goes wrong:** GDELT's data server (`data.gdeltproject.org`) has TLS certificate issues. The `lastupdate.txt` endpoint and export URLs use HTTP, not HTTPS.
**Why it happens:** The GDELT data CDN does not have a properly configured TLS certificate (ERR_TLS_CERT_ALTNAME_INVALID verified during research).
**How to avoid:** Use `http://` URLs for all GDELT data fetches. Do NOT upgrade to HTTPS.
**Warning signs:** `ERR_TLS_CERT_ALTNAME_INVALID` or `UNABLE_TO_VERIFY_LEAF_SIGNATURE` errors.

### Pitfall 4: Missing Lat/Long in GDELT Rows
**What goes wrong:** Some GDELT events have empty ActionGeo_Lat/ActionGeo_Long fields, causing NaN coordinates.
**Why it happens:** Not all events can be geolocated. Country-level events may have empty coordinates.
**How to avoid:** Filter out rows where lat/lng are empty or parse to NaN. Add a `!isNaN(lat) && !isNaN(lng)` guard.
**Warning signs:** Map entities appearing at (0,0) or NaN positions causing deck.gl errors.

### Pitfall 5: Empty/Malformed Rows
**What goes wrong:** Last line of CSV may be empty. Some rows may have fewer than 61 columns.
**Why it happens:** Trailing newline, incomplete data.
**How to avoid:** Filter rows by `columns.length >= 61` before processing.
**Warning signs:** Array index out of bounds errors when accessing column 60 (SOURCEURL).

### Pitfall 6: ACLED Config Crash on Startup
**What goes wrong:** The current `server/config.ts` has `required('ACLED_EMAIL')` and `required('ACLED_PASSWORD')` which will throw if those env vars are not set, even though ACLED is no longer used.
**Why it happens:** Config validation runs eagerly on server start.
**How to avoid:** Make ACLED config optional (remove `required()` calls or guard with presence check) since GDELT needs no config.
**Warning signs:** Server crashes on startup with "Missing required env var: ACLED_EMAIL".

### GDELT lastupdate.txt Fetch and Parse
```typescript
// Source: GDELT project documentation + verified format
async function getExportUrl(): Promise<string> {
  const res = await fetch(GDELT_LASTUPDATE_URL);
  if (!res.ok) throw new Error(`GDELT lastupdate.txt failed: ${res.status}`);
  const text = await res.text();
  const lines = text.trim().split('\n');
  // Format: "size md5hash url" -- one line per file type (export, mentions, gkg)
  const exportLine = lines.find(l => l.includes('.export.CSV.zip'));
  if (!exportLine) throw new Error('No export URL found in lastupdate.txt');
  return exportLine.trim().split(' ')[2];
}
```

### ZIP Download and Decompress
```typescript
// Source: adm-zip documentation (https://github.com/cthackers/adm-zip)
async function downloadAndUnzip(url: string): Promise<string> {
  const res = await fetch(url);
  if (!res.ok) throw new Error(`GDELT export download failed: ${res.status}`);
  const buffer = Buffer.from(await res.arrayBuffer());
  const zip = new AdmZip(buffer);
  const entries = zip.getEntries();
  // Single CSV file inside the ZIP
  return entries[0].getData().toString('utf8');
}
```

### Row Parsing and Filtering
```typescript
// Source: GDELT v2 Event Codebook + verified column positions
function parseAndFilter(csv: string): ConflictEventEntity[] {
  const lines = csv.trim().split('\n');
  const events: ConflictEventEntity[] = [];

  for (const line of lines) {
    const cols = line.split('\t');
    if (cols.length < 61) continue;

    const eventRootCode = cols[COL.EventRootCode];
    const countryCode = cols[COL.ActionGeo_CountryCode];

    if (!CONFLICT_ROOT_CODES.has(eventRootCode)) continue;
    if (!MIDDLE_EAST_FIPS.has(countryCode)) continue;

    const lat = parseFloat(cols[COL.ActionGeo_Lat]);
    const lng = parseFloat(cols[COL.ActionGeo_Long]);
    if (isNaN(lat) || isNaN(lng)) continue;

    events.push(normalizeGdeltEvent(cols, lat, lng));
  }

  return events;
}
```

### Normalization to ConflictEventEntity
```typescript
// Source: existing ConflictEventEntity interface in server/types.ts
function normalizeGdeltEvent(
  cols: string[],
  lat: number,
  lng: number,
): ConflictEventEntity {
  const eventRootCode = cols[COL.EventRootCode];
  const eventCode = cols[COL.EventCode];
  const sqlDate = cols[COL.SQLDATE]; // YYYYMMDD

  return {
    id: `gdelt-${cols[COL.GLOBALEVENTID]}`,
    type: classifyByCAMEO(eventRootCode, eventCode),
    lat,
    lng,
    timestamp: parseSqlDate(sqlDate),
    label: `${cols[COL.ActionGeo_FullName]}: ${describeEvent(eventRootCode)}`,
    data: {
      eventType: describeEvent(eventRootCode),
      subEventType: `CAMEO ${eventCode}`,
      fatalities: 0, // GDELT does not track fatalities
      actor1: '', // Could extract from Actor1Name (col 7) if desired
      actor2: '', // Could extract from Actor2Name (col 17) if desired
      notes: '', // GDELT has no notes field
      source: cols[COL.SOURCEURL] ?? '',
    },
  };
}
```

## Open Questions

1. **ACLED config validation crash**
   - What we know: `server/config.ts` calls `required('ACLED_EMAIL')` which throws if not set
   - What's unclear: Whether server currently starts without ACLED env vars
   - Recommendation: Make ACLED config fields optional in config.ts to prevent startup crash. Use `process.env.ACLED_EMAIL ?? ''` pattern.

2. **GDELT data volume per 15-min window**
   - What we know: Export CSVs are ~86KB zipped. After filtering to Middle East conflict codes, expect 0-50 events per window.
   - What's unclear: Exact count varies by global activity level
   - Recommendation: Acceptable. Log count for monitoring.

3. **Duplicate GlobalEventIDs across windows**
   - What we know: Each 15-min export contains events from that window. Same real-world event may appear in multiple windows with different GlobalEventIDs.
   - What's unclear: How much overlap exists in practice
   - Recommendation: Full-replace pattern (already used by ACLED and flights) handles this naturally. Each poll replaces all events.
