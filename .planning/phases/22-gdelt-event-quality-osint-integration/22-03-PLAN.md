<task type="auto">
  <name>Task 1: Add audit-mode parseAndFilter variant to GDELT adapter</name>
  <files>server/adapters/gdelt.ts</files>
  <action>
    1. Add a new export function `parseAndFilterWithTrace(csv: string): AuditRecord[]` to `server/adapters/gdelt.ts`:
       - Import `AuditRecord`, `PipelineTrace`, `buildAuditRecord` from `../lib/eventAudit.js`
       - This function mirrors `parseAndFilter` logic but captures BOTH accepted AND rejected events with full pipeline trace
       - For each CSV row, build a PipelineTrace recording which Phase A checks passed/failed:
         - `passedRootCode`: CONFLICT_ROOT_CODES.has(eventRootCode)
         - `passedCameoExclusion`: !getConfig().eventExcludedCameo includes baseCode
         - `passedMiddleEast`: MIDDLE_EAST_FIPS.has(countryCode)
         - `passedGeoValid`: isGeoValid(fullName, countryCode)
         - `passedMinSources`: numSources >= getConfig().eventMinSources
         - `passedActorCountry`: actor1Country || actor2Country truthy
       - For rejected rows: create AuditRecord with status='rejected', rejectionReason string (e.g., 'non_conflict_root_code', 'excluded_cameo', 'non_middle_east', 'geo_invalid', 'single_source', 'no_actor_country', 'dedup_superseded', 'below_confidence_threshold')
       - For accepted rows: create AuditRecord with status='accepted', full entity, phaseB sub-scores from computeEventConfidence breakdown
       - Store raw GDELT columns as `rawGdeltColumns` — a Record with named keys: GLOBALEVENTID, SQLDATE, Actor1Name, etc. (the meaningful columns from COL)
       - Apply dispersion to accepted events before building audit records (so dispersion info is captured)
       - NOTE: Do NOT refactor parseAndFilter to call this internally — keep them separate. parseAndFilter is production-hot; parseAndFilterWithTrace is audit-only and slower.

    2. Also add a new export `backfillEventsWithTrace(days: number, samplesPerDay?: number): Promise<AuditRecord[]>`:
       - Same as `backfillEvents` but calls `parseAndFilterWithTrace` instead of `parseAndFilter`
       - Default `samplesPerDay` = 8 (every 3h) for audit completeness (vs 4/day for production)
       - Modify `generateBackfillUrls` to accept an optional `intervalMs` parameter (default 6h for production, 3h for audit)
       - Stream-friendly: write batches to a temp array, don't accumulate all in memory at once. For the --fresh mode covering 30+ days, this prevents OOM.
  </action>

  </verify>

    - parseAndFilterWithTrace returns AuditRecord[] with both accepted and rejected events
    - Each rejected record has a specific rejectionReason string
    - Each accepted record has full PipelineTrace with phaseA/phaseB checks and sub-scores
    - backfillEventsWithTrace supports configurable samplesPerDay (default 8)
    - Existing parseAndFilter behavior unchanged

         - Import `cacheGet` from `../server/cache/redis.js` (relative import)
         - Fetch cached events from `events:gdelt` Redis key
         - Since cached events are already filtered (accepted only), wrap each in an AuditRecord with status='accepted' and a minimal PipelineTrace (sub-scores reconstructed from entity.data.confidence if available, or marked 'cached — no trace')
         - Write to output file
       - **--fresh mode (full backfill):**
         - Import `backfillEventsWithTrace` from `../server/adapters/gdelt.js`
         - Calculate days since WAR_START (Feb 28, 2026)
         - Call `backfillEventsWithTrace(days, samplesPerDay)` 
         - Write results incrementally: use `fs.createWriteStream` with JSON array streaming (write `[`, then each record as `JSON.stringify(record)` + comma, then `]`)
         - Log progress: "Processing batch X/Y..." to stderr
       - **Output format:** JSON array of AuditRecord objects, pretty-printed (2-space indent for default mode, compact for --fresh to save disk)
       - Print summary to console: total records, accepted count, rejected count, rejection reason breakdown (count per reason)
       - Exit cleanly with `process.exit(0)` after writing
