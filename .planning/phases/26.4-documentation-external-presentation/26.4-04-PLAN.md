    - Test 3: robots.txt at public/robots.txt contains User-agent: * and Disallow: /api/
    - Test 4: Existing rate limit tests still pass after adding the public tier

  <action>
1. Read server/middleware/rateLimit.ts to understand the existing rateLimiters object structure (e.g., rateLimiters.flights, rateLimiters.events, etc. — established in Phase 26.3 with JSDoc per CLN-10).

2. Add a new rateLimiters.public tier with a tighter ceiling. Recommended values per research §"Pitfall 7":
   - max: 6 (requests)
   - windowMs: 60_000 (1 minute)
   - keyGenerator: per-IP (use the existing pattern from other rate limiters)
   - message: { error: 'Rate limited', code: 'RATE_LIMITED', message: 'Public demo tier — please be gentle' }
   - Add JSDoc explaining the tier exists for the live demo URL exposure mitigation
   - Use the same Upstash Ratelimit prefix pattern as other tiers; consider a "ratelimit:public" prefix to namespace separately

3. Wire rateLimiters.public into the route(s) that should use it. Discretion: either apply globally to /api/* with a check for missing API key (true public unauthenticated traffic), OR add it as an additional middleware before per-endpoint limiters when the request looks like scraper traffic (User-Agent matching). Recommendation: apply globally as a baseline, with per-endpoint limits running on top.

4. Add a test to server/__tests__/rateLimit.test.ts (or create a new server/__tests__/middleware/rateLimitPublic.test.ts) that:
   - Imports the new rateLimiters.public
   - Asserts it exists and has expected max/windowMs
   - Uses supertest or direct middleware invocation to send 7 requests in quick succession from the same IP
   - Asserts the 7th returns 429 (or whatever count exceeds the ceiling)
   - If existing rate limit tests use a mock/spy pattern, follow the same pattern for consistency

7. Run the rate-limit test suite: npx vitest run server/__tests__/rateLimit.test.ts (or whatever the test path is). MUST pass.

8. Run full vitest suite: npx vitest run. MUST pass.
