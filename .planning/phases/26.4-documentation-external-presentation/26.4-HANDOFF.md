**THE BLOCKER:** Run `npx vitest run server/__tests__/resilience/redis-death.test.ts` to reproduce. **2 of 10 tests FAIL:**

1. `/api/events` returns **500** when Redis throws on every call
2. `/api/sites` returns **500** when Redis throws

Both should return 200 with `degraded: true` (or 502/503 — anything but 500). The test is correct — it's exposing a real graceful degradation gap that the existing `cacheGetSafe` in `server/cache/redis.ts` doesn't fully cover for these two routes.

**Two paths forward (for the resuming agent to choose):**

**Path A — Fix the routes (recommended if scope is small):**

- Read `server/routes/events.ts` and `server/routes/sites.ts`
- Find Redis calls that bypass `cacheGetSafe` (likely backfill cooldown keys, lock keys, or direct `cacheGet`/`cacheSet` calls without try/catch)
- Wrap them in try/catch that swallows Redis errors gracefully
- Or update `cacheGetSafe` to swallow more error types
- Fix should be <50 LOC across these files
- Commit as `fix(26.4-03): close graceful degradation gap in events and sites routes`

**Path B — Document the gap as deferred:**

- Update the chaos test to allowlist events + sites with TODO comments
- Create `.planning/phases/26.4-documentation-external-presentation/26.4-03-DEFERRED.md` tracking the gap
- Acceptable IF Path A turns out to be a deeper refactor than the plan budget

**Try Path A first.** Switch to B only if it balloons.

**What's missing entirely (Task 4 — Zod response validation):**

- `server/middleware/validateResponse.ts` does NOT exist
- `server/schemas/cacheResponse.ts` does NOT exist (no `server/schemas/` directory)
- No `sendValidated` helper anywhere
- No route is wired to it

Per Plan 03 spec, Task 4 must:
