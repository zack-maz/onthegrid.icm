Structure:

```markdown
# ADR-0005: Phase 26.2 NLP Geolocation Approach Scrapped

**Status:** Superseded — reverted in Phase 26.3
**Date:** 2026-04-06 (decision to scrap), 2026-04-04 (original Phase 26.2 implementation)
**Deciders:** solo author

## Context

GDELT v2 events have known false-positive geolocation problems: many events arrive with country-centroid coordinates instead of actual incident locations, producing visual stacking artifacts and misleading cluster centroids on the map. Phase 22.1 partially addressed this with deterministic dispersion (concentric rings), but the underlying issue — that GDELT's geolocation is inherently unreliable for the Iran conflict region — remained.

Phase 26.2 attempted to solve this by adding an NLP pipeline:
- Title fetcher to retrieve article titles from GDELT URLs
- NLP extractor to parse location entities from titles
- Custom geo-validator with a hand-curated me-cities.json dataset (240 Middle East cities)
- Wired into the GDELT adapter's parseAndFilter pipeline

## Decision

Scrapped the entire NLP approach in Phase 26.3 cleanup. Reverted gdelt.ts, eventScoring.ts, eventAudit.ts, geoValidation.ts, and config.ts to their pre-26.2 state. Deleted nlpGeoValidator.ts, titleFetcher.ts, nlpExtractor.ts changes, me-cities.json, extract-geonames.ts, and the associated test files.

The hardcoded tables (CITY_CENTROIDS at 42 entries, FIPS country codes, CAMEO base codes) were preserved with `TODO(26.2)` flags pointing to a future GDELT redo phase.

## Consequences

### Positive
- Removed ~1500 lines of code that didn't deliver the promised value.
- Honest acknowledgment that we'd been patching bad geocoding with more code, not solving the underlying problem.
- Cleared the codebase for the production hardening work in Phase 26.3 (pino, Zod, OpenAPI, strict TS) without NLP code adding noise to the strict-mode sweep.
- Forced a re-think: the right solution is to source better-geolocated events upstream, not to NLP-extract location names from headlines that often don't include location names.

### Negative
- Lost time invested in the original implementation (~2 days of work).
- The original GDELT false-positive problem still exists — it is now flagged as `TODO(26.2)` tech debt awaiting a proper redo phase.
- Deferred phase 26.2 (now reframed as "GDELT redo on a clean foundation") depends on Phase 26.4.

### Neutral
- The dispersion algorithm from Phase 22.1 stays as the only mitigation in place today.
- The 42-entry CITY_CENTROIDS table is preserved as a fallback for the redo phase to build on.

## What I Learned

1. **Patching downstream of a bad signal compounds the problem.** The NLP layer was downstream of GDELT's geocoding. If the upstream signal is unreliable, no amount of post-processing makes it reliable — it just hides the unreliability behind an opaque layer.
2. **Spike before commit.** A two-day spike on real GDELT data with the proposed pipeline would have surfaced the "headlines often don't include location names" failure mode before the full implementation went in.
3. **Killing your darlings is a portfolio signal.** Documenting the scrap honestly (this ADR) is more valuable than pretending the project went smoothly. Reviewers are evaluating judgment, not just code.
4. **Cleanup phases are part of the product.** Phase 26.3 absorbing the scrap of 26.2 was the right call — it forced a hard look at what was actually shipping value vs what was decorative code.

## Alternatives Considered (post-scrap)

- **Keep the NLP code disabled but in the tree** — rejected: dead code in a portfolio repo is worse than no code.
- **Replace GDELT with a better source immediately** — rejected: no obvious better source exists (ACLED gated, UCDP too coarse), and the redo phase needs research.
- **Manual curation of recent events** — rejected: not scalable, contradicts "numbers over narratives" core value.

## References

- `.planning/phases/26.2-conflict-geolocation-improvement/26.2-CONTEXT.md` — original Phase 26.2 spec
- `.planning/phases/26.3-production-code-cleanup/26.3-01-SUMMARY.md` — the deletion commits
- STATE.md "Roadmap Evolution" — "Phase 26.2 SCRAPPED and deferred — NLP approach was wrong, patching bad geocoding with more code didn't work"
- README "What I Learned" section — public-facing version of this retrospective

Structure: one section per failure mode. Section template (adapted from Google SRE book):

```markdown
## {Failure mode name}

**Symptom:** What the user or monitor sees.
**Detection:** Which log line, metric, or alert surfaces this.
**Cause:** Root-level mechanism.
**Remediation:** Concrete steps to recover.
**Prevention:** What design decisions mitigate this (or what future work would).
```

Write all 9 failure modes. Ground each in real code paths:

1. **Upstash Redis unreachable**
   - Symptom: API responses slow but not failed (1-2s added latency).
   - Detection: pino log entries with `module: 'cache/redis'` and `err.message` containing `fetch failed` or `ETIMEDOUT`. `/health` endpoint reports `redis: false`.
   - Cause: Upstash REST endpoint transient error, network partition, or auth token expired.
   - Remediation: (1) Check Upstash dashboard for incident. (2) Verify UPSTASH_REDIS_REST_URL and UPSTASH_REDIS_REST_TOKEN in Vercel env. (3) Rotate token if compromised. (4) In-memory fallback (`cacheGetSafe` in server/cache/redis.ts) keeps the app serving stale data automatically — no user-facing outage unless the fallback cache was cold.
   - Prevention: cacheGetSafe wraps every get with try/catch and falls back to in-memory Map. Phase 26.4 Plan 03 added a chaos test that proves this path works.

2. **GDELT lastupdate.txt returns 404 or stale**
   - Symptom: /api/events returns the same events repeatedly or reports `stale: true`.
   - Detection: pino log entries with `module: 'adapters/gdelt'` and `err.message: 404` or `staleMs > 900_000`.
   - Cause: GDELT publishing pipeline paused or delayed (happens occasionally during holidays or maintenance).
   - Remediation: (1) Check http://data.gdeltproject.org/gdeltv2/lastupdate.txt manually in a browser. (2) If 404, wait — GDELT usually recovers within an hour. (3) If persistent, the `?backfill=true` query param on /api/events forces a direct URL construction from yesterday's data. (4) As last resort, restore cache from the last good snapshot.
   - Prevention: 1-hour backfill cooldown via `events:backfill-ts` Redis key prevents tight backfill retry loops.

3. **Overpass API timeout**
   - Symptom: /api/sites or /api/water first load hangs for >60s.
   - Detection: pino entries with `module: 'adapters/overpass'` and `err.message: timeout`. Existing cache returns `stale: true`.
   - Cause: Overpass query is expensive for the full Middle East bbox. Primary endpoint occasionally timeouts under load.
   - Remediation: (1) The adapter auto-retries against the `private.coffee` mirror — check pino logs for `trying mirror`. (2) If both timeout, existing cache stays served (24h TTL). (3) Manual: reduce bbox in overpass-water.ts query and re-run locally to warm cache.
   - Prevention: Tiered query strategy from Phase 26.1, mirror fallback, 24h cache — never re-runs during normal operation.

4. **AISStream WebSocket disconnect mid-collect**
   - Symptom: /api/ships returns fewer ships than expected, or empty array.
   - Detection: pino entries with `module: 'adapters/ship'` (or similar) and WebSocket close codes.
   - Cause: AISStream rate-limits on auth token, or WS handshake fails.
   - Remediation: (1) Verify AISSTREAM_API_KEY in Vercel env. (2) Rotate key if rate-limited. (3) The on-demand connect pattern means each request is isolated — one failed request doesn't poison subsequent ones.
   - Prevention: On-demand connect + collect for N ms + close pattern (no persistent WS).

5. **Yahoo Finance throttling**
   - Symptom: /api/markets returns stale or empty data.
   - Detection: pino entries with `module: 'adapters/yahoo-finance'` and HTTP 429 or 403.
   - Cause: Yahoo's unofficial API aggressively rate-limits repeat requests.
   - Remediation: (1) The 60s cache TTL should prevent this, but if breached, wait 5 minutes. (2) Check if User-Agent needs updating (Yahoo occasionally blocks outdated UA strings).
   - Prevention: Aggressive 60s cache, per-instrument batching.

6. **Vercel function timeout (10s limit)**
   - Symptom: 504 Gateway Timeout on specific endpoints, usually first load after cold start.
   - Detection: Vercel function logs show `ERROR: Task timed out after 10.00 seconds`.
   - Cause: Cold start + upstream API latency > 10s total. Most often /api/water or /api/sites on cold cache.
   - Remediation: (1) Retry the request — warm cache should respond in <1s. (2) Check if cache was evicted (Upstash command budget exhaustion wipes keys). (3) Use `/api/cron/warm` to pre-warm cache after deploys.
   - Prevention: Vercel cron keeps cache warm, s-maxage CDN headers serve most requests from edge without hitting functions.

7. **Upstash command budget exhausted (~500K commands/day)**

# Graceful Degradation Contract

This document describes what happens when each layer of the system fails or is unreachable. The app is designed to stay useful under partial failure — never a full outage from a single broken dependency.

## Cache Layer: Upstash Redis

**Normal:** Every cache operation hits Upstash REST. Hit = return cached value. Miss = fetch upstream, set cache, return fresh.

**Degraded — Upstash unreachable:** `cacheGetSafe` in `server/cache/redis.ts` catches the error and falls back to an in-memory `Map<string, CacheEntry>`. Subsequent requests in the same function instance benefit from the in-memory cache. New function instances start with empty in-memory cache but still work (cache-through behavior on first request per instance).

**Contract:**
- No user-facing 500s from Redis failures.
- Responses may be slower (upstream fetches on every request until in-memory warms).
- `/health` endpoint reports `redis: false`.
- Phase 26.4 Plan 03 chaos test proves this path.

## Data Source Layer: 8 upstream APIs

**Normal:** Each source polls on its own cadence. Fresh responses replace cached data.

**Degraded — source unreachable:**
- Cached response is served with `stale: true` flag set in the response envelope.
- `/api/{source}` returns 200 (never 5xx from upstream failure alone).
- Frontend displays stale indicator in StatusPanel connection dot (yellow = stale, red = error).
- Stale threshold per source (defined in constants, e.g., flights 60s, ships 120s, events no prune).

**Contract:**
- Users always see data, even if stale.
- Staleness is surfaced visually via connection health dots.
- No single source failure takes down the app.

## Response Layer: Zod validation (Plan 03)

**Normal:** API responses parsed through Zod schemas before `res.json()`. Mismatch = 500 in dev, log warn in prod.

**Degraded — schema mismatch in prod:** Response sent anyway, warning logged. This is intentional: a strict schema mismatch shouldn't take down production just because we drifted from the OpenAPI spec.

**Contract:**
- Dev: strict schema enforcement — fail loud.
- Prod: log mismatch for later fix, but serve the request.
- Drift between implementation and OpenAPI spec becomes a fixable bug, not an outage.

## Frontend Layer: Lost Contact

**Normal:** Entities update on each poll. Detail panel shows live data.

**Degraded — entity disappears from backend:** Detail panel shows "LOST CONTACT" banner with grayscale overlay (`src/hooks/useSelectedEntity.ts`).

**Contract:**
- Users never see a blank detail panel when an entity drops.
- Last known position persists until explicitly cleared.

`/health` is intentionally not rate-limited and not cached. Response shape:

  "redis": true | false,
  "uptime": 1234,

    ...
  }
}
```

Consumed by: Vercel cron `/api/cron/health`, external monitors (if added).

| Frontend | Entity dropped | Lost Contact banner, last known state |
