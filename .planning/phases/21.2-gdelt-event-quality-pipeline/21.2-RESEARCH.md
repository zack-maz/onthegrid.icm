# Phase 21.2: GDELT Event Quality Pipeline - Research

**Researched:** 2026-03-27
**Domain:** GDELT event data validation, CAMEO classification, composite confidence scoring
**Confidence:** HIGH

## Summary

This phase adds a four-stage quality pipeline to the existing GDELT event adapter: (1) geo-validation to discard misplaced events and detect city centroids, (2) expanded CAMEO base code classification, (3) Goldstein sanity checks with type reclassification, and (4) composite confidence scoring with configurable threshold filtering. All changes are server-side, concentrated in `server/adapters/gdelt.ts` and a new `server/lib/eventScoring.ts`. One minor client-side addition: dev-mode confidence score display in EventDetail.

The existing codebase provides strong foundations. The `parseAndFilter` function already handles FIPS filtering, root code gating, deduplication, and normalization. The `BASE_CODE_MAP` has 20 entries covering the most common codes. The `newsRelevanceThreshold` pattern in `AppConfig` provides the exact template for `eventConfidenceThreshold`. The design spec (`docs/superpowers/specs/2026-03-27-gdelt-event-quality-pipeline-design.md`) is comprehensive and covers all implementation details.

**Primary recommendation:** Refactor `parseAndFilter` into Phase A (raw row filtering + geo-validation + dedup) and Phase B (normalize + Goldstein sanity + centroid detection + confidence scoring + threshold filter). Create `server/lib/eventScoring.ts` for `computeEventConfidence()`. Extend `ConflictEventEntity.data` with optional `geoPrecision` and `confidence` fields.

### Pattern 1: Two-Phase Pipeline Architecture
**What:** Split `parseAndFilter` into Phase A (raw row filtering) and Phase B (normalize + score)
**When to use:** When scoring requires normalized entity objects but dedup requires raw columns
**Why:** The existing `parseAndFilter` deduplicates by raw column keys (date+EventCode+lat/lng) before normalization. Scoring needs entity objects (for type, actor names, etc). The pipeline naturally splits at the normalization boundary.

```typescript
// Phase A: Raw row filtering (operates on string[] columns)
// 1. Existing filters (root code, country code, actor presence, lat/lng validity)
// 2. NEW: Geo cross-validation (discard misplaced events)
// 3. Existing deduplication (date + EventCode + lat/lng, highest mentions wins)

// Phase B: Normalize and score (operates on ConflictEventEntity)
// 4. normalizeGdeltEvent() -- creates entity objects
// 5. CAMEO classification -- already runs inside normalizeGdeltEvent (expanded map)
// 6. NEW: Goldstein sanity check -- reclassify outliers
// 7. NEW: Centroid detection -- set geoPrecision
// 8. NEW: Confidence scoring -- compute and attach confidence
// 9. NEW: Threshold filter -- discard below configurable threshold
```

### Anti-Patterns to Avoid
- **Scanning entire FullName for country names:** Only parse the LAST comma-delimited segment. Scanning the whole string would falsely match actor references like "US forces in Baghdad, Iraq".
- **Modifying CONFLICT_ROOT_CODES:** The spec explicitly excludes expanding root codes beyond 18/19/20. Only expand the BASE_CODE_MAP within those root codes.
- **Adding new ConflictEventTypes:** The existing 11 types remain. Expanded CAMEO mapping provides finer sub-type distinctions within existing types only.
- **Filtering on geoPrecision:** Centroid detection sets a flag, it does NOT discard events. The flag feeds into confidence scoring.

## Common Pitfalls

### Pitfall 1: FullName Parsing Edge Cases
**What goes wrong:** `ActionGeo_FullName` format varies: "City, Region, Country", "City, Country", or just "City" (single segment)
**Why it happens:** GDELT geocoding inconsistency
**How to avoid:** Split on comma, take last segment, trim whitespace. Single-segment names (no comma) skip cross-validation entirely -- rely on existing FIPS filter.
**Warning signs:** Events with FullName like "Baghdad" (no country suffix) should pass through unchanged.

### Pitfall 2: FIPS Code vs Country Name Mismatch Table
**What goes wrong:** Building the FIPS-to-country reverse lookup with wrong FIPS codes (FIPS != ISO)
**Why it happens:** GDELT uses FIPS 10-4, not ISO 3166. Turkey = TU (not TR), Iraq = IZ (not IQ), Israel = IS (not IL).
**How to avoid:** Use the existing `MIDDLE_EAST_FIPS` set as the source of truth. The reverse lookup maps FIPS -> expected FullName country suffix(es).
**Warning signs:** Turkey events being incorrectly discarded because TU != "Turkey" in a lookup that used ISO "TR".

### Pitfall 3: Goldstein = 0 or Positive on Conflict Codes
**What goes wrong:** Attempting to reclassify events with Goldstein = 0 (missing/invalid) or positive Goldstein on conflict root codes.
**Why it happens:** GDELT sometimes has bad or missing Goldstein data for valid conflict events.
**How to avoid:** Per spec: Goldstein = 0 or positive on conflict root codes -> skip reclassification entirely. In confidence scoring, assign neutral 0.5 for the Goldstein consistency signal.
**Warning signs:** Legitimate conflict events being wrongly reclassified because of Goldstein = 0.

### Pitfall 4: parseInt for NumMentions/NumSources Producing NaN
**What goes wrong:** `parseInt("", 10)` returns NaN, not 0 or undefined. The existing code uses `parseInt(...) || undefined`.
**Why it happens:** Empty GDELT columns.
**How to avoid:** The confidence scorer must handle `undefined` numMentions/numSources. Default to 1 (single source assumed) per spec, producing a low but non-NaN contribution.
**Warning signs:** NaN propagation in confidence scores.

### Pitfall 5: Backfill Path Must Run Phase B
**What goes wrong:** `backfillEvents()` calls `parseAndFilter()` -- the refactored function must still return scored entities.
**Why it happens:** Both `fetchEvents()` and `backfillEvents()` call `parseAndFilter()`. The refactoring must not break the backfill path.
**How to avoid:** `parseAndFilter` remains the single entry point that runs both Phase A + Phase B internally. Both callers get scored entities automatically.
**Warning signs:** Backfilled events missing `geoPrecision` and `confidence` fields.

### Pitfall 6: Config Mock Must Include eventConfidenceThreshold
**What goes wrong:** Existing test mocks for `config` in `events.test.ts` don't include `eventConfidenceThreshold`, causing tests to fail.
**Why it happens:** The config mock is hardcoded and must be extended.
**How to avoid:** Add `eventConfidenceThreshold: 0.35` to all config mocks in test files.
**Warning signs:** TypeError or undefined when `gdelt.ts` tries to read `config.eventConfidenceThreshold`.

### Geo Cross-Validation
```typescript
// Source: Design spec section 1a + GDELT data patterns

// Non-ME country names that commonly appear as last segment of FullName
// when events are misattributed (FIPS says IS but FullName says "United States")
const NON_ME_FULLNAME_COUNTRIES = new Set([
  'United States', 'United Kingdom', 'Russia', 'China', 'France',
  'Germany', 'India', 'Japan', 'South Korea', 'North Korea',
  'Australia', 'Canada', 'Brazil', 'Italy', 'Spain',
  'Ukraine', 'Poland', 'Netherlands', 'Belgium', 'Sweden',
  'Norway', 'South Africa', 'Nigeria', 'Kenya', 'Ethiopia',
  'Indonesia', 'Thailand', 'Vietnam', 'Philippines', 'Mexico',
  // ~30 total based on most common non-ME ActionGeo values
]);

// FIPS -> expected country name suffix(es) for Middle East countries
const FIPS_TO_EXPECTED_COUNTRY: Record<string, string[]> = {
  'IR': ['Iran'],
  'IZ': ['Iraq'],
  'SY': ['Syria'],
  'TU': ['Turkey'],
  'SA': ['Saudi Arabia'],
  'YM': ['Yemen'],
  'MU': ['Oman'],
  'AE': ['United Arab Emirates', 'UAE'],
  'QA': ['Qatar'],
  'BA': ['Bahrain'],
  'KU': ['Kuwait'],
  'JO': ['Jordan'],
  'IS': ['Israel', 'West Bank', 'Gaza Strip', 'Palestinian Territory'],
  'LE': ['Lebanon'],
  'AF': ['Afghanistan'],
  'PK': ['Pakistan'],
};

function extractLastSegment(fullName: string): string | null {
  const parts = fullName.split(',');
  if (parts.length < 2) return null; // Single segment -> skip validation
  return parts[parts.length - 1].trim();
}

function isGeoValid(fullName: string, fipsCode: string): boolean {
  const lastSegment = extractLastSegment(fullName);
  if (!lastSegment) return true; // No comma -> skip, rely on FIPS filter

  // Check 1: FullName ends with a known non-ME country
  if (NON_ME_FULLNAME_COUNTRIES.has(lastSegment)) return false;

  // Check 2: FIPS says ME country but FullName contradicts
  const expected = FIPS_TO_EXPECTED_COUNTRY[fipsCode];
  if (expected && !expected.some(name => lastSegment.includes(name))) {
    // FullName country doesn't match any expected country for this FIPS code
    // Only discard if the last segment looks like a country name (has uppercase, no digits)
    if (/^[A-Z]/.test(lastSegment) && !/\d/.test(lastSegment)) {
      return false;
    }
  }

  return true;
}
```

### Goldstein Sanity Check
```typescript
// Source: Design spec section 2c + GDELT Goldstein scale reference

const GOLDSTEIN_CEILINGS: Record<ConflictEventType, { ceiling: number; downgrade: ConflictEventType | null }> = {
  mass_violence:       { ceiling: -7,  downgrade: 'assault' },
  wmd:                 { ceiling: -7,  downgrade: 'assault' },
  airstrike:           { ceiling: -5,  downgrade: 'shelling' },
  bombing:             { ceiling: -5,  downgrade: 'shelling' },
  ground_combat:       { ceiling: -4,  downgrade: 'assault' },
  shelling:            { ceiling: -4,  downgrade: 'assault' },
  assassination:       { ceiling: -3,  downgrade: 'assault' },
  abduction:           { ceiling: -3,  downgrade: 'assault' },
  assault:             { ceiling: -1,  downgrade: null },  // Lowest, no downgrade
  blockade:            { ceiling: -1,  downgrade: 'assault' },
  ceasefire_violation: { ceiling: -1,  downgrade: 'assault' },
};

function applyGoldsteinSanity(entity: ConflictEventEntity): ConflictEventEntity {
  const goldstein = entity.data.goldsteinScale;

  // Skip if Goldstein is 0 (missing/invalid) or positive (data error on conflict code)
  if (goldstein === 0 || goldstein > 0) return entity;

  const check = GOLDSTEIN_CEILINGS[entity.type];
  if (!check || !check.downgrade) return entity;

  // Reclassify if Goldstein exceeds ceiling by more than 3 points
  // (i.e., the score is LESS negative than expected)
  if (goldstein > check.ceiling + 3) {
    return {
      ...entity,
      type: check.downgrade,
    };
  }

  return entity;
}
```

### Composite Confidence Score
```typescript
// Source: Design spec section 3a

export function computeEventConfidence(
  entity: ConflictEventEntity,
  geoPrecision: 'precise' | 'centroid',
): number {
  const { numMentions, numSources, actor1, actor2, goldsteinScale } = entity.data;

  // Signal 1: Media coverage (weight 0.30)
  const mentions = numMentions ?? 1;
  const mediaCoverage = Math.min(1, Math.log2(mentions + 1) / Math.log2(50));

  // Signal 2: Source diversity (weight 0.20)
  const sources = numSources ?? 1;
  const sourceDiversity = Math.min(1, Math.log2(sources + 1) / Math.log2(15));

  // Signal 3: Actor specificity (weight 0.20)
  const actor1Present = actor1.length > 0 ? 1 : 0;
  const actor2Present = actor2.length > 0 ? 1 : 0;
  const actorSpecificity = (actor1Present + actor2Present) * 0.5;

  // Signal 4: Geo precision (weight 0.15)
  const geoSignal = geoPrecision === 'precise' ? 1.0 : 0.3;

  // Signal 5: Goldstein consistency (weight 0.15)
  let goldsteinConsistency = 0.5; // neutral default
  if (goldsteinScale !== 0 && goldsteinScale < 0) {
    const check = GOLDSTEIN_CEILINGS[entity.type];
    if (check) {
      const diff = goldsteinScale - check.ceiling;
      if (diff <= 0) {
        goldsteinConsistency = 1.0; // Within expected range
      } else {
        goldsteinConsistency = Math.max(0, 1.0 - diff / 6); // Linear decay
      }
    }
  }

  return (
    mediaCoverage * 0.30 +
    sourceDiversity * 0.20 +
    actorSpecificity * 0.20 +
    geoSignal * 0.15 +
    goldsteinConsistency * 0.15
  );
}
```

Codes 187-189 and 197-199 don't exist in the CAMEO spec, so they correctly fall to ROOT_FALLBACK. The success criterion "all CAMEO base codes in 180-200 range have explicit mappings" is **already satisfied** by the existing code.

**Recommendation:** No new entries needed in BASE_CODE_MAP for 3-digit codes. GDELT's EventBaseCode column can contain these 4-digit codes. Consider adding these for finer classification if they appear frequently, but this is low priority since GDELT primarily uses 3-digit base codes.

## Open Questions

1. **4-digit CAMEO sub-codes in GDELT EventBaseCode**
   - What we know: CAMEO defines 4-digit codes (1821, 1822, 1823, 1831-1834, 1951-1952, 2041-2042)
   - What's unclear: How often GDELT actually uses 4-digit codes vs 3-digit in the EventBaseCode column
   - Recommendation: The existing `classifyByBaseCode` already handles this gracefully -- 4-digit codes not in the map fall to ROOT_FALLBACK. Low priority for this phase.

2. **Confidence score distribution at 0.35 threshold**
   - What we know: User wants 0.35 threshold (aggressive) vs spec's 0.15 (no-op)
   - What's unclear: What percentage of current events would be filtered at 0.35
   - Recommendation: Pipeline logging will reveal the distribution. Threshold is env-var tunable. Ship at 0.35 and monitor.

3. **Geo-validation false positive rate**
   - What we know: The last-segment parsing approach is conservative (only checks last comma segment)
   - What's unclear: Whether some legitimate ME events have non-ME country names in their last segment
   - Recommendation: Log every discard with full context. Review logs after deployment to catch false positives.
