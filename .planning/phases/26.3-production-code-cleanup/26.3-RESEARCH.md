## Standard Stack

### Core
| Library | Version | Purpose | Why Standard |
|---------|---------|---------|--------------|
| pino | 10.3.1 | Structured JSON logging | 5x faster than Winston, JSON-native, child loggers, request context; industry standard for Node.js production logging |
| pino-http | 11.0.0 | Express request/response logging middleware | Auto-generates request IDs, creates child loggers per request, logs response time |
| pino-pretty | 13.1.3 | Dev-only human-readable log output | Colorized pretty-printing for local development; never used in production |
| zod | 4.3.6 | Runtime schema validation | TypeScript-first, zero dependencies, composable schemas; dominant validation library for TS projects |
| compression | 1.8.1 | gzip/brotli response compression middleware | Express official middleware; supports gzip, deflate, and brotli (Node.js 11.7+) |
| @vitest/coverage-v8 | 4.1.2 | Code coverage with V8 provider | Native V8 coverage, matches vitest ^4.1.0; AST-based remapping for accuracy |

### Supporting
| Library | Version | Purpose | When to Use |
|---------|---------|---------|-------------|
| Node.js crypto.randomUUID() | Built-in | X-Request-ID generation | Generate UUID per request; no external dependency needed (Node 19+) |

### Alternatives Considered
| Instead of | Could Use | Tradeoff |
|------------|-----------|----------|
| pino | winston | Winston is slower, more batteries-included; pino is leaner and faster for JSON logs |
| zod | joi/yup | Zod is TypeScript-native with zero deps; joi is larger, yup less maintained |
| compression middleware | Vercel CDN auto-compression | Vercel CDN handles gzip/brotli automatically at edge; middleware adds redundant compression in serverless BUT provides compression during local dev and for non-Vercel deployments |
| @vitest/coverage-v8 | @vitest/coverage-istanbul | V8 is default, lighter, and AST-based remapping now matches Istanbul accuracy |

**Installation:**
```bash
npm install pino pino-http zod compression
npm install -D pino-pretty @vitest/coverage-v8 @types/compression
```

**Important note on compression:** Vercel CDN automatically applies gzip/brotli compression at the edge layer. The `compression` middleware is still valuable for local development, but in production on Vercel it is redundant (and slightly wasteful of function CPU time). Consider conditionally enabling it only when NOT on Vercel (`!process.env.VERCEL`), or skip the middleware entirely and note Vercel handles it. The CONTEXT.md locks in compression middleware, so install it but gate it to non-production environments.

### Pattern 1: Pino Logger with Request Context
**What:** Replace custom `log()` wrapper with pino structured logger. Use pino-http middleware for automatic request logging with request IDs.
**When to use:** Every server-side log statement.
**Example:**
```typescript
// server/lib/logger.ts
import pino from 'pino';

export const logger = pino({
  level: process.env.LOG_LEVEL ?? 'info',
  // Redact sensitive fields
  redact: ['req.headers.authorization'],
  // Pretty-print in dev only
  ...(process.env.NODE_ENV !== 'production' && {
    transport: { target: 'pino-pretty' },
  }),
});

// Child logger usage in route/adapter:
// const log = logger.child({ module: 'events' });
// log.info({ count: 42 }, 'fetched events');
```

### Pattern 2: Pino-HTTP with Request ID Generation
**What:** Use pino-http as Express middleware; it auto-generates request IDs, logs request/response, and creates per-request child loggers.
**When to use:** Replace custom `requestLogger` middleware.
**Example:**
```typescript
// In createApp():
import pinoHttp from 'pino-http';
import { randomUUID } from 'node:crypto';
import { logger } from './lib/logger.js';

app.use(pinoHttp({
  logger,
  genReqId: (req) => req.headers['x-request-id'] as string ?? randomUUID(),
  customSuccessMessage: (req, res) => `${req.method} ${req.url} ${res.statusCode}`,
  customErrorMessage: (req, _res, err) => `${req.method} ${req.url} failed: ${err.message}`,
}));
```

### Pattern 3: Zod Validation Middleware
**What:** Generic Express middleware factory that validates `req.query` (and optionally `req.params`, `req.body`) against a Zod schema.
**When to use:** Every route that reads query params.
**Example:**
```typescript
// server/middleware/validate.ts
import type { Request, Response, NextFunction } from 'express';
import { z } from 'zod';

export function validateQuery<T extends z.ZodTypeAny>(schema: T) {
  return (req: Request, res: Response, next: NextFunction): void => {
    const result = schema.safeParse(req.query);
    if (!result.success) {
      res.status(400).json({
        error: 'Invalid query parameters',
        code: 'VALIDATION_ERROR',
        statusCode: 400,
        details: result.error.flatten().fieldErrors,
      });
      return;
    }
    // Replace raw query with parsed + validated values
    req.query = result.data;
    next();
  };
}
```

### Pattern 4: Consolidated Config with Zod Validation
**What:** Merge `server/config.ts` + `server/constants.ts` into a single module. Validate env vars at startup with Zod. Crash on missing required vars, warn on invalid optional vars.
**When to use:** Application startup.
**Example:**
```typescript
// server/config.ts (consolidated)
import { z } from 'zod';
import { logger } from './lib/logger.js';

const envSchema = z.object({
  // Required -- crash if missing
  UPSTASH_REDIS_REST_URL: z.string().url(),
  UPSTASH_REDIS_REST_TOKEN: z.string().min(1),
  // Optional -- defaults provided
  PORT: z.coerce.number().default(3001),
  NODE_ENV: z.enum(['development', 'production', 'test']).default('development'),
  CORS_ORIGIN: z.string().default('*'),
  LOG_LEVEL: z.enum(['trace', 'debug', 'info', 'warn', 'error', 'fatal']).default('info'),
  // API keys (optional -- graceful degradation)
  OPENSKY_CLIENT_ID: z.string().default(''),
  OPENSKY_CLIENT_SECRET: z.string().default(''),
  ADSB_EXCHANGE_API_KEY: z.string().default(''),
  AISSTREAM_API_KEY: z.string().default(''),
  // Tuning
  EVENT_CONFIDENCE_THRESHOLD: z.coerce.number().min(0).max(1).default(0.38),
  EVENT_MIN_SOURCES: z.coerce.number().int().min(1).default(2),
  EVENT_CENTROID_PENALTY: z.coerce.number().min(0).max(1).default(0.7),
  NEWS_RELEVANCE_THRESHOLD: z.coerce.number().min(0).max(1).default(0.7),
  // ... all other env vars
});

export const env = envSchema.parse(process.env);
// Plus all constants (WAR_START, IRAN_BBOX, CACHE_TTL, etc.)
```

### Pattern 5: Consistent JSON Error Shape
**What:** All error responses follow `{ error: string, code: string, statusCode: number }`. Dev mode adds `stack` property.
**When to use:** Error handler middleware + all inline error responses.
**Example:**
```typescript
// server/middleware/errorHandler.ts
export function errorHandler(err: Error, req: Request, res: Response, _next: NextFunction): void {
  const statusCode = 'statusCode' in err ? (err as any).statusCode : 500;
  const code = 'code' in err ? (err as any).code : 'INTERNAL_ERROR';

  const body: Record<string, unknown> = {
    error: err.message || 'Internal server error',
    code,
    statusCode,
    requestId: req.id, // from pino-http
  };

  if (process.env.NODE_ENV !== 'production') {
    body.stack = err.stack;
  }

  req.log?.error({ err, statusCode }, 'request error');
  res.status(statusCode).json(body);
}
```

### Pattern 6: Graceful Shutdown
**What:** SIGTERM handler that stops accepting new requests, waits for in-flight requests to complete, closes Redis connections, then exits.
**When to use:** Server startup (index.ts), NOT in Vercel entry (Vercel has its own 500ms SIGTERM handling).
**Example:**
```typescript
// In server/index.ts (local dev server only)
const server = app.listen(port, () => { ... });

process.on('SIGTERM', () => {
  logger.info('SIGTERM received, draining connections...');
  server.close(() => {
    logger.info('HTTP server closed');
    // Close Redis if needed (Upstash REST client is stateless, no persistent connection to close)
    process.exit(0);
  });
  // Force exit after 10s if graceful shutdown hangs
  setTimeout(() => process.exit(1), 10_000);
});
```

### Anti-Patterns to Avoid
- **Logging with console.log/console.error in production:** Always use pino's structured output; never `console.log` (breaks JSON log parsing)
- **Validating query params inline:** Always use the Zod middleware factory, never manually check `req.query.foo === 'true'`
- **Catching errors silently:** Every catch block must log with pino AND either re-throw or send a proper error response
- **Mixing config sources:** After consolidation, never import from both config.ts and constants.ts -- there should only be one file

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Request ID generation | Custom UUID middleware | pino-http `genReqId` option | Automatically attaches to every log line via child logger; handles X-Request-ID header passthrough |
| Request/response logging | Custom requestLogger middleware | pino-http middleware | Automatically logs method, url, status, duration, request ID; includes error serialization |
| Query param validation | Inline `if/else` checks | Zod schemas + `validateQuery()` middleware | Type-safe, composable, auto-generates error details; eliminates raw `req.query` access |
| Config validation | Manual `process.env.FOO ?? 'default'` | Zod `envSchema.parse(process.env)` | Validates all vars at startup, crashes fast on required vars, coerces types |
| JSON error responses | Inconsistent `res.status(X).json({...})` patterns | Centralized `errorHandler` + `AppError` class | Guarantees consistent shape, dev stack traces, request ID in every error |
| Response compression | Custom compression logic | `compression` npm package (or rely on Vercel CDN) | Handles content negotiation, streaming, threshold configuration |

**Key insight:** The current codebase has ~22 files importing a custom 24-line logger and inline query param validation across 9 routes. Replacing these with industry-standard libraries (pino + zod) is exactly the "portfolio-grade" signal the user wants -- it shows awareness of production tooling, not just functional code.

## Common Pitfalls

### Pitfall 1: Breaking Existing Log Consumers During Pino Migration
**What goes wrong:** Existing code uses `log({ level: 'info', message: '...' })` -- pino uses `logger.info({...}, 'message')` with different API.
**Why it happens:** Search-and-replace misses context; different call signatures.
**How to avoid:** Migrate logger.ts to export the same `log()` function signature initially (thin wrapper around pino), then do a second pass to convert callers to native pino API. Or migrate all 22 callers in one wave with a clear pattern.
**Warning signs:** Tests that mock `log()` will break if the import path changes.

### Pitfall 2: Zod Query Param Coercion
**What goes wrong:** Express delivers ALL query params as strings. `z.number()` fails on `"42"`.
**Why it happens:** Zod validates types literally unless you use `z.coerce.number()`.
**How to avoid:** Always use `z.coerce.number()` for numeric query params, `z.coerce.boolean()` for boolean params. For enum-like strings (`source=opensky`), use `z.enum([...]).default(...)`.
**Warning signs:** 400 errors on valid requests after adding validation.

### Pitfall 3: noUncheckedIndexedAccess TypeScript Breakage
**What goes wrong:** Enabling `noUncheckedIndexedAccess` in tsconfig adds `| undefined` to every array index and Record lookup, causing hundreds of new TS errors.
**Why it happens:** The codebase uses many `array[0]`, `map[key]` patterns that assume non-undefined results.
**How to avoid:** Enable the flag AFTER dead code removal and initial TS cleanup. Fix errors methodically with null checks, non-null assertions where safe, or destructuring with defaults.
**Warning signs:** If `tsc` reports 100+ new errors after enabling the flag, fix them in batches by file rather than all at once.

### Pitfall 4: Compression Middleware on Vercel (Redundant Work)
**What goes wrong:** Adding compression middleware in serverless functions that already have CDN-level compression doubles the compression work and wastes function CPU time.
**Why it happens:** Vercel CDN automatically handles gzip/brotli for all responses based on Accept-Encoding header.
**How to avoid:** Gate compression middleware: `if (!process.env.VERCEL) app.use(compression())`. This gives local dev compression without wasting serverless CPU.
**Warning signs:** Higher function execution times in Vercel dashboard after adding compression.

### Pitfall 5: Graceful Shutdown in Serverless Context
**What goes wrong:** Implementing full graceful shutdown (draining connections, closing pools) that never actually executes in Vercel serverless.
**Why it happens:** Vercel functions are stateless -- they get SIGTERM with 500ms to clean up, but there are no persistent connections to drain (Upstash Redis is REST-based, not a persistent TCP connection).
**How to avoid:** Keep the SIGTERM handler simple: log, flush pino (if using async transport), exit. Don't over-engineer connection draining for a REST-based Redis client.
**Warning signs:** Complex shutdown logic that never runs in production.

### Pitfall 6: Test Coverage Regression During Dead Code Removal
**What goes wrong:** Removing NLP test files drops overall test count; coverage percentage may shift unpredictably.
**Why it happens:** Dead code removal changes the denominator (total lines) and the numerator (covered lines).
**How to avoid:** Remove dead code FIRST, establish new coverage baseline, THEN set the 80% threshold.
**Warning signs:** Coverage threshold fails immediately after being set because the baseline was measured before dead code removal.

### Pitfall 7: Git Reversion Scope Creep
**What goes wrong:** Reverting 26.2 changes in shared files (gdelt.ts, eventScoring.ts) accidentally reverts non-26.2 changes made in the same files.
**Why it happens:** Files have changes from multiple phases (21.2, 22, 22.1, 26.2). A naive `git checkout <pre-26.2> -- file` may revert Phase 22 fixes.
**How to avoid:** Use `git diff` between the last pre-26.2 commit and HEAD for each file to identify ONLY 26.2-specific changes. Revert surgically, not by file-level checkout.
**Warning signs:** Tests from Phase 22 (dispersion, Bellingcat, audit) start failing after the revert.
