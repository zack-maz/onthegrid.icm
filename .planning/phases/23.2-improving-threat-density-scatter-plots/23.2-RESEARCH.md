# Phase 23.2: Improving Threat Density Scatter Plots - Research

**Researched:** 2026-04-02
**Domain:** deck.gl custom shaders, additive blending, reverse geocoding, cluster visualization
**Confidence:** HIGH

## Summary

This phase enhances the existing threat cluster ScatterplotLayer in `ThreatHeatmapOverlay.tsx` with four main improvements: (1) radial gradient rendering via a custom GLSL fragment shader injected through deck.gl's LayerExtension system, (2) additive blending via deck.gl v9's WebGPU-style `parameters` prop, (3) dual-dimension encoding (color=weight, radius=spread), and (4) enriched detail panel content with geographic context from site proximity and Nominatim reverse geocoding.

The existing codebase already has solid foundations: `computeThreatWeight`, `aggregateToGrid`, `mergeClusters`, `ThreatCluster` type, cluster pick/click/hover handling in BaseMap, and the `ThreatClusterDetail` panel. The changes are primarily to the visual rendering (shader extension + blending + sizing) and the detail panel content (type breakdown bars + geographic context). A new server endpoint for Nominatim reverse geocoding with Redis caching is also needed.

**Primary recommendation:** Use deck.gl's `LayerExtension` class with `fs:DECKGL_FILTER_COLOR` shader injection to apply a radial alpha falloff based on `geometry.uv` distance-from-center, combined with the `parameters` prop for additive blending. Keep the ScatterplotLayer (do not subclass), and add a `cellCount` field to ThreatCluster for spread-based radius.

### Pattern 1: LayerExtension for Radial Gradient

**What:** Inject custom GLSL into the ScatterplotLayer's fragment shader to apply radial alpha falloff.

**When to use:** When you need to modify rendering behavior of an existing deck.gl layer without subclassing.

**How it works in ScatterplotLayer:** The fragment shader has a varying `unitPosition` (aliased to `geometry.uv` in the fragment main) that spans [-1, 1] across the circle quad. `length(geometry.uv)` gives distance from center (0 at center, 1 at edge). The `DECKGL_FILTER_COLOR` injection point runs AFTER the standard fill/stroke logic, receiving `inout vec4 color` and `FragmentGeometry geometry`.

**Example:**
```typescript
// Source: deck.gl LayerExtension docs + ScatterplotLayer fragment shader source
import { LayerExtension } from '@deck.gl/core';

class RadialGradientExtension extends LayerExtension {
  static extensionName = 'RadialGradientExtension';

  getShaders() {
    return {
      inject: {
        'fs:DECKGL_FILTER_COLOR': `
          float dist = length(geometry.uv);
          // Smooth cubic falloff: full at center, zero at edge
          float falloff = 1.0 - smoothstep(0.0, 1.0, dist);
          color.a *= falloff;
        `,
      },
    };
  }
}

// Usage:
new ScatterplotLayer({
  id: 'threat-cluster-picker',
  extensions: [new RadialGradientExtension()],
  // ... other props
});
```

**Key insight:** The `smoothstep` GLSL function gives a smooth hermite interpolation from 0 to 1. Using `1.0 - smoothstep(0.0, 1.0, dist)` creates a natural falloff that is 1.0 at center and 0.0 at edge, with smooth acceleration. This produces the "hot center bleeding into dark map" effect. A cubic or quadratic falloff (`pow(1.0 - dist, 2.0)`) would give more concentrated centers; `smoothstep` is a good middle ground.

### Pattern 2: Additive Blending via Parameters Prop

**What:** Use deck.gl v9's `parameters` prop with WebGPU-style string constants to enable additive blending.

**Why:** Additive blending causes overlapping transparent circles to accumulate brightness, naturally creating "super-hot" zones without explicit merge logic.

**Example:**
```typescript
// Source: deck.gl v9 upgrade guide + GitHub discussion #9024
new ScatterplotLayer({
  parameters: {
    depthWriteEnabled: false,
    blendColorSrcFactor: 'src-alpha',
    blendAlphaSrcFactor: 'src-alpha',
    blendColorDstFactor: 'one',
    blendAlphaDstFactor: 'one-minus-src-alpha',
    blendColorOperation: 'add',
    blendAlphaOperation: 'add',
  },
  // ... other props
});
```

**Known issue (MEDIUM confidence):** There is a known issue in luma.gl v9 where blending may not be enabled by the blending props alone in WebGL mode. A workaround may be needed via `onBeforeRender` to manually call `device.gl.enable(device.gl.BLEND)`. This should be tested and the workaround applied only if needed.

**Blend formula explanation:** `SRC_ALPHA, ONE` means: `finalColor = srcColor * srcAlpha + dstColor * 1`. This is additive blending -- overlapping translucent circles add their color contributions together, getting brighter where they overlap. For the alpha channel, using `ONE_MINUS_SRC_ALPHA` prevents alpha from exceeding 1.0.

### Pattern 3: Zoom-Dependent Layer Ordering

**What:** Dynamically reorder the `threatLayers` relative to `entityLayers` in the DeckGLOverlay `layers` array based on current map zoom level.

**Implementation approach:** Add a `zoom` field to `mapStore` (updated via `onMove` or `onZoomEnd` on the Map component), then in BaseMap, conditionally order the layers:

```typescript
// In BaseMap.tsx:
const zoom = useMapStore((s) => s.zoom);

// Below zoom 9: threat layers ON TOP (after entities)
// Above zoom 9: threat layers BEHIND (before entities)
const orderedLayers = zoom < 9
  ? [...weatherLayers, ...entityLayers, ...threatLayers]   // threats on top
  : [...weatherLayers, ...threatLayers, ...entityLayers];   // threats behind
```

**Why this approach (not `beforeId`):** The deck.gl `MapboxOverlay` renders all layers in a single WebGL overlay canvas. Layer z-order is determined by array position -- later in the array = on top. The `beforeId` prop only applies to interleaving deck.gl layers with MapLibre layers, which is not what we need here (both threat and entity are deck.gl layers).

### Pattern 4: Hover Dimming for Clusters

**What:** When one cluster is hovered, dim all other clusters to 40% opacity while brightening the hovered one.

**Implementation:** Track hovered cluster ID in component state (not uiStore -- it's transient visual state). Pass it into the `getFillColor` accessor:

```typescript
const [hoveredClusterId, setHoveredClusterId] = useState<string | null>(null);

getFillColor: (d: ThreatCluster) => {
  const baseColor = getThermalColor(d.totalWeight, maxWeight);
  const isHovered = d.id === hoveredClusterId;
  const someHovered = hoveredClusterId != null;
  const alpha = isHovered ? 255 : (someHovered ? 102 : 180); // 40% = 102
  return [...baseColor, alpha];
},
```

**Note:** The existing hover handling in `BaseMap.handleDeckHover` already detects `info.layer?.id === 'threat-cluster-picker'` and shows the ThreatTooltip. The hovered cluster state for dimming should be managed inside `useThreatHeatmapLayers` to trigger re-render of the layer's `getFillColor`, not in uiStore.

### Anti-Patterns to Avoid
- **Subclassing ScatterplotLayer:** Use LayerExtension instead -- it's composable and doesn't require maintaining a custom layer class
- **Meter-based radius for clusters:** The decision specifies fixed pixel sizing. Don't use `radiusUnits: 'meters'` -- switch to `radiusUnits: 'pixels'`
- **Storing geocode results in client state:** Cache on server in Redis, not in browser. Nominatim has rate limits (1 req/sec for the public instance) -- always cache aggressively
- **Calling Nominatim directly from the browser:** Nominatim's usage policy requires a server-side intermediary for applications. Route through the Express server

## Common Pitfalls

### Pitfall 1: Additive Blending Not Enabling in luma.gl v9
**What goes wrong:** Setting `parameters` with blend factors but circles still render with default blending (not additive).
**Why it happens:** Known luma.gl v9 bug where WebGL blending state isn't enabled by the declarative props.
**How to avoid:** Test additive blending early. If it doesn't work, use `onBeforeRender` callback on the layer to manually `gl.enable(gl.BLEND)`. Alternatively, check if deck.gl 9.2.11 has fixed this issue.
**Warning signs:** Overlapping circles don't get brighter at intersections.

### Pitfall 2: Shader Variable Name Mismatch
**What goes wrong:** Using `unitPosition` directly in the `fs:DECKGL_FILTER_COLOR` injection instead of `geometry.uv`.
**Why it happens:** The ScatterplotLayer fragment shader assigns `geometry.uv = unitPosition` at the top of `main()`, but `unitPosition` is a varying, not available through the geometry struct in the injection point.
**How to avoid:** Always use `geometry.uv` in `DECKGL_FILTER_COLOR` injection. The struct is the stable API; varyings are implementation details.
**Warning signs:** GLSL compilation error about undeclared variable.

### Pitfall 3: Nominatim Rate Limiting
**What goes wrong:** Nominatim returns 429 or blocks the server IP.
**Why it happens:** Nominatim public instance allows max 1 request/second. Multiple simultaneous cluster detail opens could exceed this.
**How to avoid:** (a) Always check Redis cache first with long TTL (geographic names don't change). (b) Quantize lat/lng to reduce unique keys (e.g., round to 2 decimal places ~1.1km precision). (c) Include proper `User-Agent` header in requests. (d) Only call Nominatim when no site matches the cluster bounding box.
**Warning signs:** Geocode requests returning errors or empty results.

### Pitfall 4: Layer Extension Not Applied to Mock in Tests
**What goes wrong:** Tests fail because the `ScatterplotLayer` mock doesn't understand `extensions` prop.
**Why it happens:** The test mock in `deck-gl-layers.ts` is a minimal class that captures props but doesn't process extensions.
**How to avoid:** Tests should verify the `extensions` array is present and contains the right extension type. Don't test GLSL rendering in jsdom -- test the data flow (color mapping, radius calculation, hover state).
**Warning signs:** Test assertions on rendered pixels (which can't work in jsdom).

### Pitfall 5: Zoom State Causing Excessive Re-renders
**What goes wrong:** Tracking zoom in mapStore triggers re-renders of BaseMap and all child hooks on every zoom tick.
**Why it happens:** `onMove` fires continuously during zoom animation, updating zoom state ~60fps.
**How to avoid:** (a) Debounce zoom updates, or (b) only update when crossing the zoom-9 threshold (use a ref to track last zoom region), or (c) use `onZoomEnd` instead of `onMove`.
**Warning signs:** Performance degradation during zoom, layer flickering.

### Pitfall 6: Detail Panel Scroll Reset Not Working
**What goes wrong:** Scroll position persists between different clusters.
**Why it happens:** React reuses the DOM element if the component key doesn't change.
**How to avoid:** Add `key={cluster.id}` to the scrollable container in ThreatClusterDetail, forcing React to remount and reset scroll.
**Warning signs:** Opening a different cluster shows the scroll position from the previous cluster.

### Cell-Count-to-Pixel Radius Mapping
```typescript
// Linear interpolation from cell count to pixel radius
// Single cell: 12px, max observed cells: scale up to 100px
function clusterRadius(cluster: ThreatCluster): number {
  const cellCount = cluster.cells.length;
  const MIN_PX = 12;
  const MAX_PX = 100;
  // Linear scale: clamp between 1 and some max cell count (e.g., 20)
  const t = Math.min(1, (cellCount - 1) / 19);  // 0 for 1 cell, 1 for 20+ cells
  return MIN_PX + t * (MAX_PX - MIN_PX);
}
```

**Alternative (sqrt):** `Math.sqrt(cellCount / maxCellCount)` gives diminishing returns for very large clusters, preventing them from dominating the screen. This may be better for visual balance but makes it harder to distinguish medium clusters.

### Site Proximity Check
```typescript
// Source: existing pattern from src/lib/attackStatus.ts
import { useSiteStore } from '@/stores/siteStore';

function findSitesInBoundingBox(
  bbox: { minLat: number; maxLat: number; minLng: number; maxLng: number },
  sites: SiteEntity[],
  padding = 0.1, // degrees (~11km)
): SiteEntity[] {
  return sites.filter((s) =>
    s.lat >= bbox.minLat - padding &&
    s.lat <= bbox.maxLat + padding &&
    s.lng >= bbox.minLng - padding &&
    s.lng <= bbox.maxLng + padding
  );
}
```

### Type Breakdown Bars
```typescript
// Horizontal bars showing count per event type
// Data: derive from cluster.cells which have per-cell dominantType + eventCount
function getTypeBreakdown(cluster: ThreatCluster, events: ConflictEventEntity[]): Array<{type: string; count: number}> {
  const eventMap = new Map(events.map(e => [e.id, e]));
  const typeCounts = new Map<string, number>();
  for (const eventId of cluster.eventIds) {
    const event = eventMap.get(eventId);
    if (event) {
      typeCounts.set(event.type, (typeCounts.get(event.type) ?? 0) + 1);
    }
  }
  return [...typeCounts.entries()]
    .map(([type, count]) => ({ type, count }))
    .sort((a, b) => b.count - a.count);
}
```

### Nominatim Server Route
```typescript
// Source: Nominatim 5.2.0 docs (nominatim.org)
// server/routes/geocode.ts
import { Router } from 'express';
import { cacheGetSafe, cacheSetSafe } from '../cache/redis.js';

const GEOCODE_CACHE_PREFIX = 'geocode:';
const GEOCODE_TTL_MS = 30 * 24 * 60 * 60 * 1000; // 30 days logical TTL
const GEOCODE_REDIS_TTL_SEC = 90 * 24 * 60 * 60;  // 90 days hard TTL

export const geocodeRouter = Router();

geocodeRouter.get('/', async (req, res) => {
  const lat = parseFloat(req.query.lat as string);
  const lon = parseFloat(req.query.lon as string);
  if (isNaN(lat) || isNaN(lon)) return res.status(400).json({ error: 'lat and lon required' });

  // Quantize to ~1km precision to reduce cache key cardinality
  const qLat = Math.round(lat * 100) / 100;
  const qLon = Math.round(lon * 100) / 100;
  const cacheKey = `${GEOCODE_CACHE_PREFIX}${qLat},${qLon}`;

  // Check cache
  const cached = await cacheGetSafe<{ city?: string; country?: string; display?: string }>(cacheKey, GEOCODE_TTL_MS);
  if (cached) return res.json(cached);

  // Fetch from Nominatim
  const url = `https://nominatim.openstreetmap.org/reverse?format=jsonv2&lat=${qLat}&lon=${qLon}&zoom=10&accept-language=en`;
  const response = await fetch(url, {
    headers: { 'User-Agent': 'IranConflictMonitor/1.0 (personal project)' },
  });
  const data = await response.json();
  const result = {
    city: data.address?.city ?? data.address?.town ?? data.address?.village,
    country: data.address?.country,
    display: data.display_name,
  };

  await cacheSetSafe(cacheKey, result, GEOCODE_REDIS_TTL_SEC);
  res.json({ data: result, stale: false, lastFresh: Date.now() });
});
```

## Open Questions

1. **Additive blending luma.gl bug status**
   - What we know: GitHub discussion #9024 documents the issue. A workaround (`device.gl.enable(device.gl.BLEND)`) exists.
   - What's unclear: Whether deck.gl 9.2.11 (project's version) has fixed this bug.
   - Recommendation: Test blending in the browser early. If it doesn't work, apply the `onBeforeRender` workaround. This is a visual-only concern and won't affect tests.

2. **Zoom state update frequency**
   - What we know: `onMove` fires at ~60fps during zoom. `onZoomEnd` fires once per zoom action.
   - What's unclear: Whether `onZoomEnd` is available on the react-maplibre `<Map>` component (it may be `onZoom` instead).
   - Recommendation: Use threshold-crossing detection with a ref to avoid unnecessary re-renders. Only update mapStore zoom when it crosses the 9.0 threshold.

3. **Nominatim during development**
   - What we know: Public instance has 1 req/sec limit. No API key needed.
   - What's unclear: Whether development volume will trigger rate limiting.
   - Recommendation: Aggressive caching (30-day TTL). Quantize coordinates to 2 decimal places. For development, the Redis cache will quickly fill with common Middle East locations and subsequent requests will be cache hits.
