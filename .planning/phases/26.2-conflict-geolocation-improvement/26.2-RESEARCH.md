# Phase 26.2: Conflict Geolocation Improvement - Research

**Researched:** 2026-04-05
**Domain:** NLP-based geolocation cross-validation for GDELT conflict events
**Confidence:** HIGH

## Summary

This phase adds NLP-based cross-validation to the GDELT event pipeline to reduce misplaced conflict events on the map. The implementation involves three interconnected systems: (1) fetching article titles from GDELT SOURCEURL fields and caching them in Redis, (2) extending the existing compromise NLP library with a custom Middle East city lexicon to extract place names and actors from those titles, and (3) cross-validating extracted information against GDELT's geocoded coordinates to reject mismatches or relocate centroid events to specific cities.

The existing codebase already has all the foundational pieces. `server/lib/nlpExtractor.ts` uses compromise for actor-action-target extraction (extend with `.places()`). `server/lib/geoValidation.ts` has CITY_CENTROIDS (42 cities) and FIPS-based geo validation (expand to ~150-200 cities from GeoNames). `server/adapters/gdelt.ts` has a well-structured `parseAndFilter` pipeline with Phase A/B architecture and audit trace support. The Redis caching pattern in `server/cache/redis.ts` is directly reusable for title caching.

**Primary recommendation:** Extend the existing pipeline with a new Phase C (NLP validation) between Phase B scoring and the threshold filter. Use compromise's custom lexicon pass-through (`nlp(text, lexicon)`) to recognize ME-specific city names. Fetch titles concurrently with a 10-connection limit and cache in Redis with 7-day TTL.

### Pattern 1: Compromise Custom Lexicon for ME Cities

**What:** Pass a lexicon object to compromise so `.places()` recognizes Middle East city names that aren't in the default English lexicon.

**When to use:** Every NLP parse of an article title.

**Example:**
```typescript
// Source: compromise npm docs + verified WebSearch pattern
import nlp from 'compromise';

// Build lexicon from ME cities data (done once at module load)
const ME_CITY_LEXICON: Record<string, string> = {};
for (const city of ME_CITIES) {
  ME_CITY_LEXICON[city.name.toLowerCase()] = 'Place';
  // Also add alternate names if any
  if (city.asciiName && city.asciiName !== city.name) {
    ME_CITY_LEXICON[city.asciiName.toLowerCase()] = 'Place';
  }
}

// Extract places from title using custom lexicon
function extractPlaces(title: string): string[] {
  const doc = nlp(title, ME_CITY_LEXICON);
  return doc.places().out('array') as string[];
}
```

**Confidence:** HIGH -- verified from compromise docs and Microsoft CSE blog demonstrating identical pattern with custom locality lexicons.

### Pattern 3: Title Fetching with Concurrency + Redis Cache

**What:** Fetch article titles from SOURCEURL, parse og:title or `<title>` from first 10KB of HTML, cache URL->title in Redis.

**When to use:** During Phase C of the pipeline, before NLP validation.

**Example:**
```typescript
// Pattern follows existing open-meteo-precip.ts batch processing
const TITLE_CONCURRENCY = 10;
const TITLE_TIMEOUT_MS = 5000;
const TITLE_CACHE_KEY_PREFIX = 'title:';
const TITLE_LOGICAL_TTL_MS = 7 * 86_400_000;  // 7 days
const TITLE_REDIS_TTL_SEC = 30 * 86_400;       // 30 days hard TTL

// Regex-based title extraction (no HTML parser dependency needed)
function extractTitleFromHtml(html: string): string | null {
  // Try og:title first (most reliable for news articles)
  const ogMatch = html.match(
    /<meta[^>]+property=["']og:title["'][^>]+content=["']([^"']+)["']/i
  );
  if (ogMatch) return ogMatch[1].trim();

  // Fallback to <title> tag
  const titleMatch = html.match(/<title[^>]*>([^<]+)<\/title>/i);
  if (titleMatch) return titleMatch[1].trim();

  return null;
}
```

**Confidence:** HIGH -- regex pattern is well-established for meta tag extraction; Redis caching follows existing `cacheGet`/`cacheSet` pattern.

### Pattern 4: Pipeline Phase C Integration

**What:** Insert NLP validation as Phase C between existing Phase B (scoring) and the threshold filter in `parseAndFilter`.

**When to use:** Applied to all events that pass Phase A and Phase B normalization.

**Example:**
```typescript
// In parseAndFilter, after Phase B scoring:
// Phase C: NLP cross-validation (async -- requires title fetching)
// Events get one of:
//   - nlpVerified: true (title fetched, NLP confirms location)
//   - nlpMismatch: true (title fetched, NLP contradicts -> REJECT)
//   - nlpSkipped: true (title fetch failed -> penalize 0.7x)
//   - nlpRelocated: true (centroid event moved to NLP-extracted city)
```

**Note:** `parseAndFilter` is currently synchronous. Phase C requires async title fetching, so the function signature changes to `async parseAndFilter()`. The caller (`fetchEvents`) already awaits the CSV download, so this is a natural extension.

### Anti-Patterns to Avoid

- **Full HTML parsing:** Do NOT use cheerio, jsdom, or any DOM parser for title extraction. A simple regex on the first 10KB of HTML response body is sufficient and avoids dependency bloat in the serverless bundle.
- **Fetching full pages:** Do NOT download entire articles. Use `Range: bytes=0-10240` header or abort after 10KB to minimize bandwidth and latency.
- **Blocking on title fetch failures:** Do NOT let a single failed title fetch block the entire pipeline. Failed fetches should gracefully degrade (skip NLP, apply 0.7x penalty).
- **Parsing titles synchronously:** Do NOT fetch titles one-by-one. Batch with concurrency limit for throughput.
- **Over-relying on Actor1Name text:** GDELT Actor1Name can be country names (IRAN), leader names (KHAMENEI), or organization names (HEZBOLLAH). Use Actor1CountryCode (3-letter CAMEO) as the primary signal; Actor1Name as supplementary.
- **Trusting GDELT Actor1CountryCode blindly:** Empty Actor1CountryCode is common. When both actor country codes are empty, fall through to NLP-only validation from the title.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| NLP entity recognition | Custom regex tokenizer for places/actors | compromise `.places()` with custom lexicon | Edge cases in entity boundary detection, multi-word names, possessives |
| City coordinate lookup | In-memory linear search over 200 cities | Pre-built Map from GeoNames extraction | O(1) lookup vs O(n), important when processing batches of events |
| HTML meta tag parsing | Full DOM parser (cheerio/jsdom) | Regex on first 10KB of response | Title/og:title are in `<head>`, always near top of HTML; regex handles >95% of news sites |
| Batch concurrency control | Manual Promise queue | `Promise.allSettled` with chunked batches | Established pattern in this codebase (see `open-meteo-precip.ts`, `backfillEvents`) |
| Actor-to-country mapping | Runtime NLP inference of nationality | Static lookup table (CAMEO code -> FIPS) + keyword -> country map | Deterministic, fast, testable; NLP inference is unreliable for nationality |

**Key insight:** The pipeline already has sophisticated scoring, auditing, and batch processing patterns. This phase extends existing infrastructure rather than building from scratch. The only genuinely new capability is title fetching + NLP place extraction -- everything else is wiring.

## Common Pitfalls

### Pitfall 1: parseAndFilter Becomes Async
**What goes wrong:** `parseAndFilter` is currently synchronous. Adding title fetching makes it async. All callers need updating.
**Why it happens:** Title fetching requires HTTP requests with Redis cache lookups.
**How to avoid:** Change `parseAndFilter` signature to `async`, update `fetchEvents`, `backfillEvents`, and `parseAndFilterWithTrace`. The callers already use `await` for other async operations.
**Warning signs:** TypeScript will catch missing `await` at compile time if `strict` mode is on (it is).

### Pitfall 2: GDELT SOURCEURL Field Has Multiple URLs
**What goes wrong:** GDELT's SOURCEURL column (index 60) typically has a single URL, but some rows may have semicolon-separated or pipe-separated multiple URLs.
**Why it happens:** GDELT aggregates from multiple sources; the SOURCEURL may reference the first/primary source or occasionally multiple.
**How to avoid:** Parse SOURCEURL, take the first URL only. If it contains a separator, split and use the first valid URL. Cache per-URL so repeated URLs across events hit the cache.
**Warning signs:** URL parsing failures, non-URL strings in the field.

### Pitfall 3: Title Fetch Overwhelms Serverless Function Timeout
**What goes wrong:** Fetching titles for 50+ events sequentially or with too-high concurrency could exceed Vercel's 10-second serverless timeout.
**Why it happens:** Each title fetch is 100-500ms. 50 events * 300ms / 10 concurrency = 1.5s -- manageable. But slow/unresponsive news sites can block.
**How to avoid:** Use aggressive per-request timeout (5s), concurrency limit (10), and abort early on slow responses. Redis cache means subsequent requests for the same URL are instant.
**Warning signs:** Function timeout errors in Vercel logs; latency spikes on `/api/events`.

### Pitfall 4: Compromise Doesn't Recognize Multi-Word City Names
**What goes wrong:** City names like "Deir ez-Zor", "Mazar-i-Sharif", or "Tel Aviv" span multiple tokens. Compromise may not recognize them as single Place entities.
**Why it happens:** compromise tokenizes on whitespace; multi-word entries need special handling in the lexicon.
**How to avoid:** Register multi-word names in the lexicon as-is. compromise handles multi-word lexicon entries correctly when passed via the second argument (`nlp(text, {'deir ez-zor': 'Place'})`). Verified in Microsoft CSE blog example with Arabic multi-word place names.
**Warning signs:** NLP extracting partial city names ("Deir" instead of "Deir ez-Zor").

### Pitfall 5: False Rejection of Events About Cross-Border Conflicts
**What goes wrong:** "Israel strikes targets in Syria" -- actors mention Israel, but the geocoded location is in Syria. A naive actor-country check would reject this.
**Why it happens:** The actor (Israel) and the event location (Syria) are different countries. This is expected for cross-border military operations.
**How to avoid:** Only reject when ALL extracted actors point to a different country than the geocode AND no extracted place names match the geocoded region. If the title mentions both the actor's country and the target's country/city, the geocode should match the TARGET location, not the actor.
**Warning signs:** High rejection rate for airstrike events (which are inherently cross-border).

### Pitfall 6: CAMEO Code 182 and 190 Rejection Breaks Existing Fixtures
**What goes wrong:** The CONTEXT.md says to reject 180, 182, 190 entirely. But 190 ("Conventional military force NOS") currently maps to `ground_combat` and 182 to `assault`. Rejecting these changes the behavior of existing fixture tests.
**Why it happens:** These codes are currently in CAMEO_SPECIFICITY with 0.1x weight. Moving them to EXCLUDED_BASE_CODES changes from "low weight" to "hard reject".
**How to avoid:** Add 182 and 190 to the `eventExcludedCameo` config default (currently `['180', '192']`). Update fixture tests that use these codes.
**Warning signs:** Regression test failures in `gdelt-fixtures.test.ts`.

### Pitfall 7: GeoNames City Names Clash with Common English Words
**What goes wrong:** Some city names are common English words (e.g., "Sur" in Turkey, "Salt" in Jordan, "Hit" in Iraq). Tagging these as Place creates false NLP matches.
**Why it happens:** GeoNames includes all populated places regardless of name ambiguity.
**How to avoid:** Use a minimum population threshold (50k as CONTEXT.md suggests) to exclude small towns with ambiguous names. Manually exclude known problematic short/common names. Only apply the custom lexicon for names >= 4 characters.
**Warning signs:** NLP extracting common words as place names.

### Example 2: Title Fetcher with Redis Cache

```typescript
// Source: follows existing open-meteo-precip.ts batch pattern + redis.ts caching
import { cacheGet, cacheSet } from '../cache/redis.js';

const CONCURRENCY = 10;
const FETCH_TIMEOUT_MS = 5000;
const MAX_BYTES = 10240; // 10KB -- enough for <head> section
const CACHE_PREFIX = 'title:';
const LOGICAL_TTL_MS = 7 * 86_400_000;
const REDIS_TTL_SEC = 30 * 86_400;

// Regex patterns for title extraction (no DOM parser needed)
const OG_TITLE_RE = /<meta[^>]+property=["']og:title["'][^>]+content=["']([^"']+)["']/i;
const OG_TITLE_RE_ALT = /<meta[^>]+content=["']([^"']+)["'][^>]+property=["']og:title["']/i;
const TITLE_TAG_RE = /<title[^>]*>([^<]+)<\/title>/i;

function extractTitleFromHtml(html: string): string | null {
  const og = OG_TITLE_RE.exec(html) ?? OG_TITLE_RE_ALT.exec(html);
  if (og) return decodeHtmlEntities(og[1].trim());
  const title = TITLE_TAG_RE.exec(html);
  if (title) return decodeHtmlEntities(title[1].trim());
  return null;
}

function decodeHtmlEntities(text: string): string {
  return text
    .replace(/&amp;/g, '&')
    .replace(/&lt;/g, '<')
    .replace(/&gt;/g, '>')
    .replace(/&quot;/g, '"')
    .replace(/&#39;/g, "'");
}

/**
 * Fetch titles for multiple URLs with concurrency limit.
 * Returns Map<url, title | null>.
 */
export async function batchFetchTitles(
  urls: string[],
): Promise<Map<string, string | null>> {
  const results = new Map<string, string | null>();
  const uncached: string[] = [];

  // Check Redis cache first
  for (const url of urls) {
    const cached = await cacheGet<string>(CACHE_PREFIX + hashUrl(url), LOGICAL_TTL_MS);
    if (cached) {
      results.set(url, cached.data);
    } else {
      uncached.push(url);
    }
  }

  // Fetch uncached titles in batches
  for (let i = 0; i < uncached.length; i += CONCURRENCY) {
    const batch = uncached.slice(i, i + CONCURRENCY);
    const settled = await Promise.allSettled(
      batch.map(async (url) => {
        const res = await fetch(url, {
          signal: AbortSignal.timeout(FETCH_TIMEOUT_MS),
          headers: { 'User-Agent': 'IranConflictMonitor/1.0' },
        });
        if (!res.ok) return { url, title: null };
        // Read only first ~10KB
        const reader = res.body?.getReader();
        if (!reader) return { url, title: null };
        let html = '';
        while (html.length < MAX_BYTES) {
          const { done, value } = await reader.read();
          if (done) break;
          html += new TextDecoder().decode(value);
        }
        reader.cancel();
        return { url, title: extractTitleFromHtml(html) };
      }),
    );

    for (const result of settled) {
      if (result.status === 'fulfilled') {
        const { url, title } = result.value;
        results.set(url, title);
        if (title) {
          await cacheSet(CACHE_PREFIX + hashUrl(url), title, REDIS_TTL_SEC);
        }
      }
    }
  }

  return results;
}
```

## GDELT Data Patterns Critical for Implementation

### Actor Country Codes (CAMEO vs FIPS)

GDELT uses **two different code systems** that must not be confused:

| Field | Code System | Example Iran | Example Israel | Example Iraq |
|-------|-------------|-------------|---------------|-------------|
| Actor1CountryCode (col 7) | 3-letter CAMEO | IRN | ISR | IRQ |
| Actor2CountryCode (col 17) | 3-letter CAMEO | IRN | ISR | IRQ |
| ActionGeo_CountryCode (col 53) | 2-letter FIPS 10-4 | IR | IS | IZ |

**Confidence:** HIGH -- verified against official CAMEO.country.txt from GDELT documentation repository.

A CAMEO-to-FIPS mapping table is required for cross-validation. The full mapping for 16+1 ME countries:

```
IRN->IR, IRQ->IZ, ISR->IS, SYR->SY, TUR->TU, SAU->SA,
YEM->YM, LBN->LE, JOR->JO, KWT->KU, QAT->QA, ARE->AE,
BHR->BA, OMN->MU, AFG->AF, PAK->PK, EGY->EG
```

### Actor Name Patterns

GDELT Actor1Name contains UPPERCASE strings that can be:
- Country names: `IRAN`, `ISRAEL`, `IRAQ`
- Leader names: `KHAMENEI`, `NETANYAHU`
- Organization names: `HEZBOLLAH`, `HAMAS`, `IRGC`
- Military terms: `IRAN MILITARY`, `ISRAEL DEFENSE`
- Ethnic groups: `KURD`, `SUNNI`

The Actor1CountryCode (3-letter CAMEO) is the most reliable field for actor nationality. Actor1Name is supplementary.

### SOURCEURL Patterns

GDELT SOURCEURL (column 60) is typically a single URL to the source article. In rare cases it may be empty. The URL points to news sites like Reuters, BBC, Al Jazeera, etc. Most URLs will be HTTPS.

   - What we know: Some GDELT rows have empty or malformed SOURCEURL.
   - What's unclear: What percentage of ME conflict events lack a SOURCEURL.
   - Recommendation: Treat as "title fetch failure" -- apply 0.7x confidence penalty, skip NLP validation.

3. **Redis budget impact of title caching**
   - What we know: Redis commands at ~92% budget. Title cache adds URL->title entries with 7-day TTL.
   - What's unclear: How many unique URLs appear per GDELT update cycle (repeat rate).
   - Recommendation: Use URL hash (SHA-256 truncated, existing `hashUrl` pattern from newsClustering) as cache key to minimize key length. Monitor Redis usage after deployment.

4. **Performance impact of title fetching in Vercel serverless**
   - What we know: Vercel serverless has 10s default timeout (can be extended to 60s on Pro).
   - What's unclear: Whether batch title fetching completes within timeout for large event sets.
   - Recommendation: 5s per-fetch timeout, 10 concurrency, abort after 10KB. Worst case: 50 uncached URLs / 10 concurrency * 5s timeout = 25s. Need to verify Vercel timeout config.
