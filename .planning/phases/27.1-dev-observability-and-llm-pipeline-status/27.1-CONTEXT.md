# Phase 27.1: Dev Observability and LLM Pipeline Status

## Scope

Enhance the dev-only API status overlay (DevApiStatus) with real-time LLM pipeline progress tracking, per-source health metrics, error diagnostics, data quality indicators, and copy-to-clipboard diagnostics. Server-side progress exposed via dedicated endpoint.

## Decisions

### 1. LLM Pipeline Progress Granularity

**Decision:** Both stage-level AND batch-level.

- Stage-level: `idle → grouping → llm-processing → geocoding → done`
- Batch-level within each stage: batch 3/12, geocoded 15/45
- Total group count and per-stage completion counts

### 2. Exposure Mechanism

**Decision:** Dedicated `/api/events/llm-status` endpoint.

- Client polls separately (e.g. every 5s while processing, back off when idle)
- Returns current pipeline stage, batch progress, timing, errors
- Clean separation from `/api/events` response

### 3. Persistence Strategy

**Decision:** In-memory for live progress, Redis for final summary.

- Module-level `Map` or object in events route for live batch/stage tracking
- Write `events:llm-summary` to Redis on completion: lastRun, groupCount, batchCount, geocodeCount, duration, errors
- Status endpoint reads in-memory if available, falls back to Redis summary

### 4. DevApiStatus Panel Enhancements

**Decision:** Full metrics suite for all 8 API sources.

**New per-source metrics (all 8 stores):**

- Poll interval countdowns ("next poll in 12s")
- Success/failure rates (rolling window of last N fetches)
- Response times (avg + last, measured in polling hooks)

**Error diagnostics:**

- `lastError: string | null` added to each store
- Surface actual failure reason in panel ("AISStream: WebSocket timeout", "GDELT: HTTP 503")
- Stores update `lastError` on `setError()` calls, clear on success

**Data quality indicators:**

- Events: total, precision breakdown (exact/city/region), LLM-enriched count
- Flights: total, unidentified count, ground count
- Ships: total count (expand later if needed)
- Sites: total, attacked count
- Show quality breakdown inline per row or as expandable detail

**Copy diagnostics:**

- Single button copies entire panel state as JSON to clipboard
- Includes all metrics, errors, quality indicators, timestamps
- For bug reports and debugging

### 5. Store Changes Required

Each of the 8 data stores needs:

- `lastError: string | null` — set on error, cleared on success
- `nextPollAt: number | null` — set by polling hook after each fetch
- `recentFetches: Array<{ ok: boolean; durationMs: number; timestamp: number }>` — rolling window (last 10)

Polling hooks (useFlightPolling, useShipPolling, useEventPolling, useNewsPolling, useMarketPolling, useWeatherPolling, useWaterFetch, useWaterPrecipPolling) need to:

- Measure fetch duration (Date.now delta)
- Record success/failure in store
- Set nextPollAt after scheduling setTimeout

### 6. Server-Side LLM Progress Tracking

In `server/routes/events.ts`, the fire-and-forget LLM block needs:

- Module-level progress state object tracking current stage, batch index, total batches, geocode index, total geocodes, start time
- Update progress on each batch completion and geocode completion
- Clear/reset on pipeline start, mark done on completion
- New route `/api/events/llm-status` reads this state

### 7. Client-Side LLM Status Polling

New hook `useLLMStatusPolling`:

- Poll `/api/events/llm-status` every 5s when pipeline is active
- Back off to 30s when idle
- Feed data into DevApiStatus panel for the Events row's LLM progress display

## Deferred Ideas

- Pipeline waterfall visualization (horizontal timing bars per stage) — high effort, defer to future
- Configurable alert thresholds — hardcoded defaults sufficient for now
- Historical sparklines per source — nice but not essential for dev observability
