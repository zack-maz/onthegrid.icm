# Phase 27.1: Dev Observability and LLM Pipeline Status - Research

**Researched:** 2026-04-09
**Domain:** Client-side Zustand store instrumentation, server-side in-memory progress tracking, dev-only UI overlay
**Confidence:** HIGH

## Summary

This phase adds dev-only observability to an existing 8-source polling architecture. The codebase has a highly consistent pattern across all 8 Zustand stores and 8 polling hooks. Each store currently tracks `connectionStatus` and `lastFetchAt` but lacks `lastError`, `nextPollAt`, and `recentFetches`. Each polling hook follows the same recursive `setTimeout` + visibility awareness pattern, making the instrumentation work mechanical and uniform.

The server-side LLM pipeline progress tracking requires module-level state in the events route module. On Vercel with Fluid Compute (enabled by default since early 2025), module-level state persists across warm starts -- meaning a progress object initialized at module scope will survive between invocations within the same instance. However, cold starts reset this state, so the design correctly falls back to Redis for the last-run summary. The fire-and-forget LLM block in `server/routes/events.ts` (lines 294-348) already runs as a detached `void (async () => {...})()` after the response is sent, making it straightforward to inject progress callbacks without blocking the HTTP response.

**Primary recommendation:** Add 3 fields to each store (`lastError`, `nextPollAt`, `recentFetches`), wrap each hook's fetch call in a duration-measuring wrapper, create a module-level `LLMPipelineProgress` object in events.ts, expose it via `/api/events/llm-status`, and rebuild DevApiStatus.tsx to consume all new data.

## User Constraints (from CONTEXT.md)

### Locked Decisions

1. **LLM Pipeline Progress Granularity:** Both stage-level AND batch-level. Stage-level: `idle -> grouping -> llm-processing -> geocoding -> done`. Batch-level within each stage: batch 3/12, geocoded 15/45. Total group count and per-stage completion counts.
2. **Exposure Mechanism:** Dedicated `/api/events/llm-status` endpoint. Client polls separately (e.g. every 5s while processing, back off when idle). Returns current pipeline stage, batch progress, timing, errors. Clean separation from `/api/events` response.
3. **Persistence Strategy:** In-memory for live progress, Redis for final summary. Module-level `Map` or object in events route for live batch/stage tracking. Write `events:llm-summary` to Redis on completion: lastRun, groupCount, batchCount, geocodeCount, duration, errors. Status endpoint reads in-memory if available, falls back to Redis summary.
4. **DevApiStatus Panel Enhancements:** Full metrics suite for all 8 API sources. New per-source metrics: poll interval countdowns, success/failure rates (rolling window), response times (avg + last). Error diagnostics with lastError. Data quality indicators per source. Copy diagnostics button.
5. **Store Changes Required:** Each of the 8 data stores needs: `lastError: string | null`, `nextPollAt: number | null`, `recentFetches: Array<{ ok: boolean; durationMs: number; timestamp: number }>` (rolling window, last 10).
6. **Server-Side LLM Progress Tracking:** Module-level progress state object tracking current stage, batch index, total batches, geocode index, total geocodes, start time. Update on each batch/geocode completion. New route `/api/events/llm-status`.
7. **Client-Side LLM Status Polling:** New hook `useLLMStatusPolling`. Poll 5s when active, back off to 30s when idle.

### Recommended New Store Fields

Add to each store interface:

```typescript
// New observability fields
lastError: string | null;
nextPollAt: number | null;
recentFetches: Array<{ ok: boolean; durationMs: number; timestamp: number }>;

// New actions
recordFetch: (ok: boolean, durationMs: number) => void;
setNextPollAt: (ts: number | null) => void;
```

Implementation pattern:

```typescript
// Inside create<XxxState>()()
lastError: null,
nextPollAt: null,
recentFetches: [],

setXxxData: (response) =>
  set({
    // ... existing fields ...
    lastError: null,  // clear error on success
  }),

setError: (message?: string) =>
  set({
    connectionStatus: 'error',
    lastError: message ?? 'Unknown error',
  }),

recordFetch: (ok, durationMs) =>
  set((state) => ({
    recentFetches: [
      ...state.recentFetches.slice(-9),  // keep last 9
      { ok, durationMs, timestamp: Date.now() },
    ],
  })),

setNextPollAt: (ts) => set({ nextPollAt: ts }),
```

**Design choice: `setError` gets an optional message parameter.** This is backward-compatible -- existing callers pass no argument and get `'Unknown error'`. New instrumented callers pass the error message string. ### Recommended Hook Instrumentation Pattern

Wrap each hook's fetch function to measure duration:

```typescript
const fetchXxx = async (): Promise<void> => {
  if (cancelled) return;
  const start = Date.now();
  try {
    const res = await fetch(url);
    if (cancelled) return;
    if (!res.ok) {
      const errMsg = `Xxx API ${res.status}`;
      setError(errMsg);
      recordFetch(false, Date.now() - start);
      return;
    }
    const data = await res.json();
    setXxxData(data);
    recordFetch(true, Date.now() - start);
  } catch (err) {
    if (!cancelled) {
      const errMsg = err instanceof Error ? err.message : 'Network error';
      setError(errMsg);
      recordFetch(false, Date.now() - start);
    }
  }
};

const schedulePoll = (): void => {
  if (cancelled) return;
  const nextTs = Date.now() + POLL_INTERVAL;
  setNextPollAt(nextTs);
  timeoutRef.current = setTimeout(async () => {
    await fetchXxx();
    schedulePoll();
  }, POLL_INTERVAL);
};
```

## Common Pitfalls

### Pitfall 1: Module-Level State Loss on Vercel Cold Starts

**What goes wrong:** The LLM progress object resets to `idle` when Vercel spins up a new instance. A client polling for progress might see `idle` mid-pipeline if routed to a fresh instance.
**Why it happens:** Vercel Fluid Compute reuses instances (warm starts) but cold starts reset all module-level state. With Fluid Compute, cold starts affect <1% of requests, but they still happen.
**How to avoid:** The design already handles this: Redis summary fallback provides the last-completed-run data. For live progress, accept that cold starts may briefly show `idle` -- the next request to the original warm instance will return the real progress. Document this limitation.
**Warning signs:** DevApiStatus shows "idle" when you know the pipeline is running.
[CITED: https://vercel.com/docs/functions]

### Pitfall 2: Changing setError Signature Breaks Existing Callers

**What goes wrong:** Making `setError(message: string)` required would break all existing `setError()` calls with no arguments.
**Why it happens:** 18 call sites across 8 hooks and stores all call `setError()` with no args today.
**How to avoid:** Make message optional: `setError: (message?: string) => void`. Default to `'Unknown error'` when no message provided. Existing callers continue to work unchanged.
**Warning signs:** TypeScript compilation errors after changing the interface.

### Pitfall 3: recentFetches Array Growing Unbounded

**What goes wrong:** Without a cap, `recentFetches` grows forever over a long browser session (flights poll every 5s = 17,280 entries/day).
**Why it happens:** Easy to forget the `.slice(-9)` cap when implementing.
**How to avoid:** Always slice before push: `[...state.recentFetches.slice(-9), newEntry]`. Cap at 10 entries as specified in CONTEXT.md. The Zustand `set()` call produces a new array each time, so no mutation concerns.
**Warning signs:** DevApiStatus panel becomes sluggish after hours of use; memory usage climbs.

### Pitfall 4: useWaterPrecipPolling Has No Error Surface

**What goes wrong:** This hook silently swallows all errors (lines 33-34) and never calls `setError()`. Adding instrumentation naively might miss this hook entirely.
**Why it happens:** By design -- stale precipitation data is acceptable and the water layer gracefully handles missing precip. The hook was intentionally written to not disturb the water facilities data.
**How to avoid:** For observability purposes, still `recordFetch(false, ...)` on error but do NOT call `setError()` (preserving the existing design intent). The `recentFetches` array will show failures without changing the `connectionStatus`.
**Warning signs:** Water precip always shows 100% success rate despite actual failures.

### Pitfall 5: nextPollAt Drift on Tab Visibility Changes

**What goes wrong:** When the tab goes hidden, polling stops (timeout cleared). When it comes back, an immediate fetch happens then polling resumes. But `nextPollAt` from the old schedule is now stale.
**Why it happens:** The visibility change handler clears the timeout but doesn't clear `nextPollAt`. The DevApiStatus countdown would show a past timestamp.
**How to avoid:** In the visibility hidden handler, set `nextPollAt` to `null`. In the visibility visible handler, after scheduling the next poll, update `nextPollAt` normally.
**Warning signs:** Countdown shows negative time ("overdue") after tab focus returns.

### Pitfall 6: Concurrent LLM Pipeline Runs

**What goes wrong:** Two near-simultaneous `/api/events` requests could both pass the `shouldRunLLM()` check and both start the fire-and-forget pipeline, corrupting the shared progress object.
**Why it happens:** The `recordLLMTimestamp()` is called before spawning, but on Vercel with Fluid Compute, concurrent requests to the same instance share module-level state.
**How to avoid:** Add a guard in the progress object: `if (llmProgress.stage !== 'idle') return;` at the top of the fire-and-forget block. The existing `recordLLMTimestamp()` call prevents cross-instance races via Redis, but the in-memory guard prevents intra-instance races.
**Warning signs:** Progress jumps backward or shows impossible batch counts.
