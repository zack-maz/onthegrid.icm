# Phase 22.1: Fixing Dispersion - Context

**Gathered:** 2026-04-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix two bugs discovered after Phase 22 shipped: (1) event dispersion not visually working — events still stack at city centroids on the map, and (2) camera continuously re-centering when a `near:` search query is active instead of jumping once. Both are regressions/gaps in existing functionality, not new features.

</domain>

<decisions>
## Implementation Decisions

### Dispersion Root Cause — Data-First Diagnosis
- Run the existing audit script to dump actual GDELT lat/lng values and compare against CITY_CENTROIDS
- Determine how many events match vs miss the 0.01° tolerance window
- If GDELT coordinates are close but outside 0.01°: widen tolerance to 0.05° (~5.5km) — safe because no two CITY_CENTROIDS are within 50km of each other
- Ring radii (3/6/9 km) stay as-is for now — revisit after dispersion is visually working

### Backfill Merge — Re-Disperse After Merge
- Move `disperseEvents()` out of `parseAndFilter` and into the events route, AFTER all merging (cached + backfill + fresh) is complete
- Single dispersion pass over the final merged event set eliminates slot collisions across batches
- `parseAndFilter` returns undispersed events — audit script gets raw coordinates (more useful for debugging)
- Redis cache stores dispersed coordinates — dispersion only runs on cache miss/refresh, not every read

### Camera Fly-To — Once Per Unique Pin Location
- Camera should fly to a `near:` location once when the proximity pin coordinates change (new target)
- Should NOT re-fly if pin stays the same but other search terms or filters change
- Clearing `near:` resets the "already flown" state — re-entering `near:Tehran` flies to Tehran again (natural UX)
- Fix is scoped to `useQuerySync` near: handling only — notification fly-to behavior is fine and unchanged
- Implementation: track last-flown pin coordinates in a ref, compare before calling `setFlyToTarget`

### Claude's Discretion
- Exact ref tracking mechanism for camera fly-to deduplication
- Whether to add logging/metrics for centroid match rate
- Test structure for the fixes

</decisions>
