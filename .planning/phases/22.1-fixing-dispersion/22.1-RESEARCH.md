# Phase 22.1: Fixing Dispersion - Research

**Researched:** 2026-04-01
**Domain:** GDELT event dispersion pipeline bug + search camera fly-to UX bug
**Confidence:** HIGH

## Summary

This phase addresses two bugs discovered after Phase 22 shipped. Both are code-level integration issues in existing functionality, not new features or library adoption tasks.

**Bug 1: Dispersion not visually working.** The `disperseEvents()` function is called inside `parseAndFilter()` at line 342 of `server/adapters/gdelt.ts`. This means each GDELT export file (15-minute batch) gets its events dispersed independently. When `backfillEvents()` processes dozens of batches, and when the events route merges cached + backfill + fresh events, the per-batch dispersion assignments collide -- multiple batches independently assign slot 0 of Ring 0 for Tehran events, resulting in events stacking right back on top of each other. The fix is to move `disperseEvents()` out of `parseAndFilter` and into the events route, after all merging is complete.

**Bug 2: Camera continuously re-centering on `near:` queries.** In `useQuerySync.ts` lines 499-512, every time the Search-to-Sidebar effect fires (on any `parsedQuery` change), the code calls `setFlyToTarget` if a `near:` tag is present -- even if the pin coordinates have not changed. This means editing any other search term while `near:Tehran` is active causes the camera to re-fly to Tehran. The fix is a ref-based guard that tracks the last-flown coordinates and only triggers fly-to when the pin actually changes.

**Primary recommendation:** Two surgical fixes with clear root causes. No new dependencies, no architectural changes. Both fixes are testable with existing test infrastructure.

## Architecture Patterns

### Bug 1: Dispersion Relocation

**Current (broken) data flow:**
```
GDELT CSV -> parseAndFilter() -> disperseEvents() -> [per-batch dispersed events]
                                                           |
backfillEvents() merges per-batch results ----------------+
events route merges cached + backfill + fresh -> cache -> client
```
Problem: Each batch independently assigns ring slots. Batch A and Batch B both assign slot 0/Ring 0 for Tehran events, causing visual stacking.

**Fixed data flow:**
```
GDELT CSV -> parseAndFilter() -> [undispersed events with raw coordinates]
                                       |
events route merges cached + backfill + fresh
                                       |
                              disperseEvents() <- single pass over ALL events
                                       |
                              cache -> client
```

**Key change locations:**
1. `server/adapters/gdelt.ts` line 342: Remove `disperseEvents(results)` call, return `results` directly
2. `server/routes/events.ts` line 106-107: After `const merged = Array.from(eventMap.values())`, apply `const dispersed = disperseEvents(merged)`, then cache `dispersed`
3. `backfillEvents()` in gdelt.ts: No changes needed -- it calls `parseAndFilter` which will now return undispersed events automatically
4. `parseAndFilterWithTrace()` in gdelt.ts lines 658-679: Also remove its internal `disperseEvents` call and update audit record population accordingly

### Bug 2: Camera Fly-To Dedup

**Current (broken) behavior:**
```
useEffect([parsedQuery]) fires ->
  deriveFiltersFromAST extracts near: tag ->
    if proximityPin !== null -> setFlyToTarget() EVERY TIME
```

**Fixed behavior:**
```
useEffect([parsedQuery]) fires ->
  deriveFiltersFromAST extracts near: tag ->
    if proximityPin !== null AND coordinates differ from lastFlownRef ->
      setFlyToTarget() ONCE
      update lastFlownRef
    if proximityPin === null ->
      clear lastFlownRef (so re-entering near:Tehran works)
```

**Implementation pattern:**
```typescript
// Add ref at top of useQuerySync hook (alongside existing syncSourceRef)
const lastFlownPinRef = useRef<{ lat: number; lng: number } | null>(null);

// In the near: handling block (lines 499-512):
if (filterUpdates.proximityPin !== undefined) {
  useFilterStore.getState().setProximityPin(filterUpdates.proximityPin);
  hasFilterUpdates = true;
  if (filterUpdates.proximityPin !== null) {
    const prev = lastFlownPinRef.current;
    const pin = filterUpdates.proximityPin;
    const pinChanged = !prev || prev.lat !== pin.lat || prev.lng !== pin.lng;
    if (pinChanged) {
      useUIStore.setState({ isFiltersCollapsed: false });
      useNotificationStore.getState().setFlyToTarget({
        lng: pin.lng,
        lat: pin.lat,
        zoom: 8,
      });
      lastFlownPinRef.current = { lat: pin.lat, lng: pin.lng };
    }
  } else {
    // Clearing near: resets fly-to state
    lastFlownPinRef.current = null;
  }
}
```

### Anti-Patterns to Avoid
- **Re-dispersing on every cache read:** Dispersion should run once when writing to cache, not on every `/api/events` response. The decision specifies "Redis cache stores dispersed coordinates."
- **Deep equality for pin comparison:** Use simple `lat !== lat` comparison, not JSON.stringify. Pin objects are tiny (2 numbers).
- **Modifying parseAndFilterWithTrace without updating audit records:** The trace function has its own dispersion logic at lines 658-679 that also needs updating.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Coordinate comparison | Custom distance function | Direct `===` on lat/lng numbers | Pin coordinates come from the same lookup table, exact equality is correct |
| Tolerance widening | Fancy nearest-neighbor search | Simple constant change 0.01 -> 0.05 | User decision: safe because no two centroids are within 50km |

## Common Pitfalls

### Pitfall 1: Breaking parseAndFilterWithTrace When Fixing parseAndFilter
**What goes wrong:** The audit trace function at lines 658-679 has its own `disperseEvents()` call and audit record population. If you remove dispersion from `parseAndFilter` but forget `parseAndFilterWithTrace`, the audit script breaks.
**Why it happens:** Two parallel code paths that both call `disperseEvents`.
**How to avoid:** Update BOTH functions in the same commit. Remove dispersion from both, apply it at the route/script level.
**Warning signs:** Audit script tests fail, or audit records show dispersed coordinates when they should show raw ones.

### Pitfall 2: Cache Containing Mix of Dispersed and Undispersed Events
**What goes wrong:** During deployment transition, Redis cache may contain events that were dispersed by the old code. New events from the updated code would be undispersed before the route-level dispersion pass. After merging, re-dispersion of already-dispersed events would double-offset them.
**Why it happens:** `disperseEvents()` looks at lat/lng to match centroids. If an event was previously dispersed from Tehran centroid to 3km north, it will NOT match Tehran centroid anymore (outside 0.01 degree tolerance = ~1.1km), so it passes through unchanged. This is actually safe behavior.
**How to avoid:** No action needed -- dispersed events won't re-match centroids. But verify this assumption in tests.
**Warning signs:** Events appearing at ~6km from centroid (3km dispersion + 3km re-dispersion) -- but this should not happen due to tolerance check.

### Pitfall 3: Fly-To Ref Not Clearing on near: Removal
**What goes wrong:** If `lastFlownPinRef` is not cleared when `near:` is removed from the query, re-entering the same `near:Tehran` won't trigger a fly-to because the ref still holds the old Tehran coordinates.
**Why it happens:** Missing the `null` branch in the pin handling logic.
**How to avoid:** Explicitly set `lastFlownPinRef.current = null` when `filterUpdates.proximityPin` is `null` (i.e., near: tag was removed).
**Warning signs:** User types `near:Tehran`, removes it, types `near:Tehran` again -- camera doesn't move the second time.

### Pitfall 4: Tolerance Widening Without Verifying Data First
**What goes wrong:** Widening tolerance from 0.01 to 0.05 without first checking whether GDELT coordinates actually need it could cause false centroid matches.
**Why it happens:** Premature optimization before data diagnosis.
**How to avoid:** The user decision is clear: run the audit script FIRST to check match rates. Only widen if data shows near-misses. The 0.05 degree threshold is pre-approved as safe because no two CITY_CENTROIDS are within 50km.
**Warning signs:** Precise events at ~4km from a city center being incorrectly flagged as centroid events and dispersed.

## Open Questions

1. **Tolerance: 0.01 vs 0.05 -- which does the data need?**
   - What we know: Both `detectCentroid` and `findCentroidKey` use 0.01 degrees (~1.1km). GDELT may geocode to exact centroid coords or slightly offset ones.
   - What's unclear: The actual match rate of real GDELT data against the 0.01 window.
   - Recommendation: Run audit script first as prescribed. If match rate is high (>90%), keep 0.01. If significant near-misses exist, widen to 0.05. This is a data-driven decision, not a code decision.

2. **Should tolerance be a shared constant?**
   - What we know: Currently duplicated in `dispersion.ts:108` and `geoValidation.ts:176`.
   - Recommendation: Extract to a shared constant in `geoValidation.ts` and import in `dispersion.ts`. Prevents drift if tolerance is changed.
