# Phase 21.3: Multi-User Load Testing — Research

**Researched:** 2026-03-29
**Status:** Complete

## Codebase Findings

### Endpoints & Rate Limits

9 API endpoints to test, each with per-endpoint rate limiting via Upstash Redis sliding window:

| Endpoint | Rate Limit | Polling Interval | s-maxage (edge cache) |
|----------|-----------|------------------|----------------------|
| `/api/flights` | 120/60s | 5s | varies by source |
| `/api/ships` | 60/60s | 30s | ~30s |
| `/api/events` | 20/60s | 15min | ~900s |
| `/api/news` | 20/60s | 15min | ~900s |
| `/api/markets` | 30/60s | 60s | ~60s |
| `/api/weather` | 10/60s | 30min | ~1800s |
| `/api/sites` | 10/60s | once on mount | ~3600s |
| `/api/sources` | 30/60s | on demand | ~30s |
| `/health` | none | on demand | no-store |

### Rate Limiting Architecture

- `createRateLimiter` in `server/middleware/rateLimit.ts` uses `@upstash/ratelimit` with Redis-backed sliding window
- Identifier: `req.ip ?? x-forwarded-for ?? 'anonymous'`
- Rate limiting skipped in dev (`NODE_ENV !== 'production' && !process.env.VERCEL`)
- Returns 429 with `{ error: 'Too many requests' }` + standard `X-RateLimit-*` headers
- **Key insight:** All 100 k6 VUs from a single machine will share one IP, so rate limits apply collectively

### Edge Caching

- `cacheControl(sMaxAge, staleWhileRevalidate)` middleware sets `public, max-age=0, s-maxage=N, stale-while-revalidate=M`
- Vercel CDN caches responses at edge; `x-vercel-cache` header reports HIT/MISS/STALE
- **Key insight:** Edge caching is the primary scaling lever — under sustained polling, most requests should be cache HITs, meaning neither the serverless function nor Redis are invoked

### Redis Budget

- `/health` endpoint reports `estimatedDailyCommands: 15,282` (for 1 user)
- Upstash free tier: 10,000 commands/day
- **Key insight:** The estimate assumes 1 user. With edge caching, multiple users hitting cached responses don't generate additional Redis commands. The test should validate this by comparing `/health` snapshots before/after.

### Existing Test Infrastructure

- `scripts/smoke-test.ts` — sequential endpoint validation, checks status codes + expected JSON keys + cache headers
- No existing load test scripts
- k6 is the chosen tool (from CONTEXT.md decisions)

### Vercel Serverless Architecture

- All `/api/*` routes go through a single serverless function (`dist-server/vercel.cjs`)
- Cold starts affect first invocation; subsequent requests hit warm instances
- Vercel scales function instances automatically under load
- `vercel.json` rewrites: `/api/(.*)` → function, everything else → `index.html`

## Technical Research: k6

### k6 Scenario Design

k6 supports `scenarios` with different executor types:
- `ramping-vus` — gradually increase/decrease VUs over stages (ideal for ramp-up)
- Each VU runs its own iteration of the default function

### Realistic Polling Pattern

Each VU should simulate one browser session:
1. **Initial burst** (mount): Hit all endpoints once (flights, ships, events, news, markets, weather, sites, sources)
2. **Steady state** (polling): Only hit endpoints at their polling intervals

k6 approach: Use a single VU function with `sleep()` calls and conditional fetches based on elapsed time, or use multiple scenarios with different exec functions per endpoint.

**Recommended: Multiple scenarios** — one per polling frequency group:
- `flights_polling` — every 5s
- `ships_polling` — every 30s
- `markets_polling` — every 60s
- `events_polling` — every 900s (15min, won't trigger in 5min test)
- `weather_polling` — every 1800s (won't trigger in 5min test)
- `news_polling` — every 900s (won't trigger in 5min test)
- `sites_fetch` — once per VU
- `health_check` — periodic for Redis monitoring

### k6 Thresholds

```javascript
thresholds: {
  http_req_duration: ['p(95)<500'],        // p95 under 500ms
  http_req_failed: ['rate<0.01'],          // <1% non-429 errors
  'http_req_duration{endpoint:flights}': ['p(95)<300'],
  'http_req_duration{endpoint:ships}': ['p(95)<300'],
}
```

### k6 Tags & Custom Metrics

- Tag requests with `{ endpoint: 'flights' }` for per-endpoint analysis
- Use `Counter` for 429 responses (expected, tracked separately from errors)
- Use `Trend` for cold-start latency (first request per endpoint)

## Playwright Browser Validation

### Purpose

2-3 Playwright browser instances run alongside k6 to verify the React app handles degraded conditions:
- ConnectionStatus indicators stay 'connected' or transition to 'stale' (not error)
- UI renders without crashing
- Data updates visible in the dashboard

### Approach

- Use `@playwright/test` with `test.describe.parallel`
- Navigate to production URL
- Assert StatusPanel connection dots are green/yellow (not red)
- Assert entity counts update (flights > 0 at some point)
- Run for duration of k6 test

## Validation Architecture

### Dimension 1: Unit — k6 thresholds
Automated pass/fail via k6 built-in threshold checking.

### Dimension 2: Integration — Multi-endpoint correlation
Verify all endpoints respond correctly when hit concurrently (no resource contention between routes).

### Dimension 3: System — End-to-end browser validation
Playwright browsers verify the full stack (CDN → serverless → Redis → response → React rendering).

### Dimension 4: Performance — Latency & throughput metrics
k6 reports p50/p95/p99 latency, request rate, and error rate per endpoint.

### Dimension 5: Budget — Redis command sustainability
Compare `/health` snapshots before/after to measure actual Redis command consumption under load.

## Risk Assessment

1. **Rate limit exhaustion from single IP** — All 100 VUs share one IP. Flights endpoint (120/60s) will be hit 100 VUs × 1 req/5s = 20 req/s = 1200/min, far exceeding 120/min limit. Most requests will get 429s. This is expected and validates rate limiting works, but means k6 error thresholds must exclude 429s.

2. **Edge cache masking** — If Vercel edge caching is effective, most requests after warmup will be served from CDN without hitting the serverless function. This is good for production but means the test may not stress Redis/serverless as much as expected. Track `x-vercel-cache` HIT ratio to understand this.

3. **5-minute duration vs slow-poll endpoints** — Events (15min), news (15min), and weather (30min) won't complete a full polling cycle in 5 minutes. The initial request validates them, but sustained polling isn't tested. This is acceptable since those endpoints have high edge cache TTLs.
