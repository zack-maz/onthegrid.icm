  <name>Task 1: Write k6 load test script with multi-scenario realistic polling</name>
  <files>scripts/load-test.js, .gitignore</files>
  <action>
Create `scripts/load-test.js` — a k6 load test script that simulates 100 concurrent users with realistic polling patterns. k6 runs in its own Go-based JS runtime (NOT Node.js), so use k6's built-in modules (`k6`, `k6/http`, `k6/metrics`), NOT npm imports.

**Configuration:**
- `BASE_URL` from `__ENV.BASE_URL` with default `https://irt-monitoring.vercel.app`
- 5-minute test duration with ramp-up stages: 0->25 VUs over 30s, hold 25 for 30s, 25->100 over 60s, hold 100 for 120s, 100->0 over 60s

**Scenarios (use k6 `scenarios` with `ramping-vus` executor):**
- `flights_polling`: Each VU hits `/api/flights` every 5 seconds (the app's flight polling interval). This is the highest-frequency endpoint.
- `ships_polling`: Each VU hits `/api/ships` every 30 seconds.
- `markets_polling`: Each VU hits `/api/markets` every 60 seconds.
- `slow_polling`: Each VU hits `/api/events`, `/api/news`, `/api/weather` once on start (these have 15-30 min intervals so won't re-poll in 5 min).
- `static_fetch`: Each VU hits `/api/sites` and `/api/sources` once (one-time mount fetches).
- `health_monitor`: A single VU hits `/health` every 30 seconds to track Redis health throughout the test.

All scenarios share the same ramp-up stages so VU count grows together.

**Per-request tagging:**
- Tag every request with `{ tags: { endpoint: 'flights' } }` etc. for per-endpoint breakdown in results.

**Custom metrics (use k6 `Counter` and `Trend`):**
- `rate_limited` Counter: Increment on 429 responses (expected, not counted as errors).
- `cold_start_duration` Trend: Track first request per endpoint per VU (use a VU-scoped flag object).
- `cache_hits` Counter: Increment when `x-vercel-cache` header is `HIT`.
- `cache_misses` Counter: Increment when `x-vercel-cache` header is `MISS` or `STALE`.

**Response validation (in each scenario's exec function):**
- For non-429 responses: check `res.status === 200` using `check()` from k6.
- For 200 responses: check that the expected JSON key exists (e.g., `data` for most endpoints, `opensky` for `/api/sources`, `status` for `/health`).
- On 429: increment `rate_limited` counter, do NOT mark as failed (use `{ tags: { expected_status: '429' } }` and `http.expectedStatuses(200, 429)` or manually handle).

**Thresholds:**
```javascript
thresholds: {
  http_req_duration: ['p(95)<500'],
  'http_req_duration{endpoint:flights}': ['p(95)<300'],
  'http_req_duration{endpoint:ships}': ['p(95)<300'],
  'http_req_duration{endpoint:health}': ['p(95)<200'],
  http_req_failed: ['rate<0.05'],  // <5% — accounts for 429s in failed count
}
```

**Health snapshot:** In `setup()`, fetch `/health` and return the response as setup data. In `teardown(data)`, fetch `/health` again and log a comparison of Redis status + source freshness before/after.

**Helper function:** Create a `makeRequest(endpoint, path, expectedKey)` helper that handles tagging, 429 tracking, cache hit tracking, cold start tracking, and response validation — to avoid duplicating this logic in every scenario exec.

**Add to `.gitignore`:** Append `load-test-results/` if not already present — k6 JSON/CSV output will go here.

  <action>
Create `scripts/load-test.spec.ts` — a Playwright test that runs 2-3 browser instances against production to verify the React app stays functional during load testing.

**Important:** This is a standalone Playwright script, NOT part of the Vitest test suite. Use `@playwright/test` directly. The user will run it alongside k6 via `npx playwright test scripts/load-test.spec.ts`.

**Test structure:**
```typescript
import { test, expect } from '@playwright/test';
```

Use `test.describe('Production Load Validation')` with these tests:

1. **`test('app loads and renders map')`**
   - Navigate to `https://irt-monitoring.vercel.app`
   - Wait for the map container to be visible (selector: `[class*="maplibregl-map"]` or `canvas`)
   - Assert page title contains expected text
   - Take a screenshot for visual record

2. **`test('status indicators show healthy state')`**
   - Navigate to production URL
   - Wait for initial data load (~10s)
   - Look for StatusPanel/StatusDropdown indicators — assert no error states visible
   - Check that flight count text is visible and shows a number > 0 (selector for the flights counter in the sidebar)

3. **`test('app remains stable over 3 minutes')`**
   - Navigate to production URL
   - Wait for initial load
   - Every 30 seconds for 3 minutes (6 checks):
     - Assert page is still loaded (no crash/blank screen)
     - Assert no uncaught JS errors (listen for `page.on('pageerror')`)
     - Assert network requests are completing (no hanging spinners)
   - Take final screenshot

**Configuration at top of file:**
```typescript
const PROD_URL = process.env.PROD_URL ?? 'https://irt-monitoring.vercel.app';
```

**Playwright config:** Include `test.use({ ... })` with:
- `baseURL: PROD_URL`
- `timeout: 60_000` (longer timeout for production)
- `screenshot: 'on'`

**Error collection:** Use `const errors: string[] = []` and `page.on('pageerror', e => errors.push(e.message))` to collect JS errors, then assert `errors.length === 0` at end.
