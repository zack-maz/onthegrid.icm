## Decisions Made
- k6 ramping-vus executor used for all polling scenarios with shared ramp-up stages (0->25->100->0 over 5 minutes)
- 429 responses tracked as custom Counter metric rather than errors, since all 100 VUs share one IP and will collectively exceed per-endpoint rate limits
- Slow-poll endpoints (events/news/weather) fire once per VU then sleep, matching real app behavior where 15-30min polling intervals won't repeat in 5-min test
- Playwright stability test checks every 30 seconds for 3 minutes (6 checks) with pageerror event collection
- health_monitor uses constant-vus executor (1 VU) instead of ramping-vus since only one monitor is needed
