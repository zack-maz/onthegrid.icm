# Phase 21.3: Multi-User Load Testing - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Validate that the production deployment handles 100+ concurrent users without degradation. Write a reusable k6 load test script simulating realistic polling patterns, run it against live production, and verify latency, error rate, rate limiting, edge caching, and Redis budget sustainability. Fix minor issues inline; document major bottlenecks for a separate phase.

</domain>

<decisions>
## Implementation Decisions

### Concurrency Target & Tooling
- **Tool:** k6 (CLI, runs locally)
- **Concurrency:** 100+ virtual users
- **Pattern:** Realistic polling — each virtual user simulates the app's actual polling intervals (flights 5s, ships 30s, events 15min, markets 5min, weather 30min, news 15min, sites one-time)
- **Duration:** 5 minutes
- **Script location:** `scripts/load-test.js` committed alongside existing `scripts/smoke-test.ts`
- **Ramp-up:** k6 stages to gradually reach 100 users (not instant spike)

### Metrics & Pass/Fail Criteria
- **p95 latency:** Under 500ms across all endpoints
- **Error rate:** 0% non-rate-limit errors (5xx = failure; 429s are expected and tracked separately)
- **Redis commands:** Track via `/health` endpoint before and after test — calculate commands consumed, validate sustainability at Upstash free-tier scale
- **Cold starts:** Tag first request per endpoint as 'cold', report cold-start latency separately from steady-state
- **Vercel edge cache:** Track `x-vercel-cache` header (HIT/MISS) — expect high cache hit ratio under sustained polling

### Rate Limit Behavior
- **Single-IP testing:** Accept that all 100 VUs share one IP; 429s are expected and prove rate limiting works
- **No temporary limit changes:** Test with production rate limits as-is
- **No backoff:** k6 keeps polling at full rate even after 429s — validates rate limiter stability under sustained pressure
- **No thundering herd scenario:** Realistic staggered polling provides sufficient organic cache-expiry overlap

### Test Execution Environment
- **Target:** Live production (https://irt-monitoring.vercel.app)
- **Browser validation:** 2-3 Playwright browsers alongside k6 to verify React app handles stale/degraded responses correctly
- **Artifacts:** k6 script committed to `scripts/`, results output gitignored
- **Fix scope:** Minor issues (TTL tweaks, rate limit number adjustments) fixed inline; major bottlenecks documented for a separate phase

### Claude's Discretion
- k6 ramp-up profile (stages configuration)
- Exact k6 threshold syntax for pass/fail
- Playwright browser test assertions (what to verify in the UI)
- Results output format and directory structure

</decisions>

<specifics>
## Specific Ideas

- The existing `scripts/smoke-test.ts` validates individual endpoint health — the k6 script should simulate what happens when 100 users are all running those same endpoints on realistic polling schedules simultaneously
- Edge caching (`s-maxage`) is the primary scaling lever — the test should confirm most requests are cache HITs, proving the architecture scales without multiplying Redis or serverless function costs
- Rate limiter runs in-memory on Vercel serverless functions — under concurrent load from one IP, all 100 VUs should share the same rate limit window

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `scripts/smoke-test.ts`: Endpoint specs (paths, expected keys, cache checks) — k6 script can mirror these endpoints
- `server/middleware/rateLimit.ts`: `createRateLimiter` factory with per-endpoint limits (flights 120/min, ships 60/min, events 20/min, etc.) — these are the limits k6 should hit
- `server/cache/redis.ts`: `cacheGetSafe`/`cacheSetSafe` with in-memory fallback — k6 should verify this degrades gracefully
- `/health` endpoint: Already reports Redis latency, per-source freshness, command estimates — k6 can poll this for Redis budget tracking

### Established Patterns
- 6 polling hooks with different intervals: flights (5s), ships (30s), events (15min), markets (5min), weather (30min), news (15min) — k6 VU scenario should match these
- Sites are fetched once on mount (not polled) — k6 should hit `/api/sites` once per VU lifecycle
- `ConnectionStatus` type tracks health per data source — Playwright browsers should verify these stay 'connected' or transition to 'stale' gracefully

### Integration Points
- `vercel.json` rewrites all `/api/*` to serverless function — all endpoints share one function's cold start pool
- Vercel edge caching via `Cache-Control: max-age=0, s-maxage=N` — k6 should observe `x-vercel-cache` header
- `/api/sources` returns per-source configuration — useful for k6 to validate source health during test

</code_context>
