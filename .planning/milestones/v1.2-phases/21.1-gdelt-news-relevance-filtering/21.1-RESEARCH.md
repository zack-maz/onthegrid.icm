# Phase 21.1: GDELT News Relevance Filtering - Research

**Researched:** 2026-03-26
**Domain:** NLP-based news filtering, GDELT metadata enrichment, relevance scoring
**Confidence:** MEDIUM

## Summary

This phase replaces the current binary keyword filter in `server/lib/newsFilter.ts` with an NLP-based relevance scoring system. The core problem is false positives like "Tehran condemns airstrike in Yemen" being geolocated to Tehran. The solution combines three approaches: (1) keyword reclassification to a strict non-ambiguous set with ambiguous co-occurrence rules, (2) lightweight NLP extraction of actor-action-target triples using the `compromise` library, and (3) a 0-1 confidence score replacing the binary pass/fail filter.

A critical finding from this research is that the **GDELT DOC 2.0 API ArtList mode does NOT return V2Themes, V2Locations, V2Counts, or tone values in its JSON response**. The response contains only 8 fields: `url`, `url_mobile`, `title`, `seendate`, `socialimage`, `domain`, `language`, `sourcecountry`. The GDELT DOC API supports *filtering* by tone (e.g., `tone<-5`) but does not *return* the tone value per article. V2Themes, V2Locations, and V2Counts are properties of the GKG (Global Knowledge Graph) raw data files, not the DOC API. To access this metadata, one would need to download and parse the raw GKG CSV files (updated every 15 minutes, multi-MB compressed). This is an overweight approach for a news filtering pipeline and would significantly increase serverless function execution time and complexity. The research recommends adapting the decisions to compute tone-like signals from NLP analysis rather than relying on GDELT metadata.

**Primary recommendation:** Use `compromise` (v14.15.x, ~250KB minified) for lightweight NLP extraction; compute relevance score from NLP triple completeness + source reliability + headline negativity heuristic; skip GDELT GKG enrichment in favor of NLP-derived signals.

## Critical Finding: GDELT DOC API Metadata Limitation

**Confidence: HIGH** (verified via official GDELT documentation, Python client library, and multiple sources)

The GDELT DOC 2.0 API ArtList mode returns ONLY these fields per article:
| Field | Description |
|-------|-------------|
| `url` | Article URL |
| `url_mobile` | Mobile URL |
| `title` | Article title |
| `seendate` | Date in YYYYMMDDTHHmmssZ format |
| `socialimage` | Social sharing image URL |
| `domain` | Source domain |
| `language` | Article language |
| `sourcecountry` | Source country |

**NOT returned:** tone, V2Themes, V2Locations, V2Counts, or any GKG metadata.

These fields exist in the GDELT Global Knowledge Graph (GKG) raw CSV files, which are separate from the DOC API. The GKG CSV is tab-separated with 27 columns, including:
- Column 4: DocumentIdentifier (article URL)
- Column 8: V2Themes (semicolon-delimited theme;offset pairs)
- Column 10: V2Locations (hash-delimited with lat/lng)
- Column 15: V2Tone (comma-separated, first value = overall tone score)

To access this data, one would need to:
1. Download `lastupdate.txt` (same pattern as existing GDELT events adapter)
2. Parse the 3rd URL (GKG file, not events)
3. Download and decompress the GKG ZIP
4. Parse the multi-MB CSV
5. Match articles by DocumentIdentifier (URL) to DOC API results

**Assessment:** This adds significant complexity and serverless execution time. The GKG approach would require downloading, decompressing, and parsing multi-MB CSV files on every 15-minute poll cycle. The existing events adapter (`server/adapters/gdelt.ts`) already does something similar for events data, so the pattern is proven. However, the cost-benefit is questionable for a news filtering use case.

### Recommended Adaptation

Since the CONTEXT.md decisions assumed GDELT metadata was available via the DOC API, the implementation needs to adapt:

| Original Decision | Adaptation | Rationale |
|-------------------|------------|-----------|
| Feed GDELT tone score into formula | Compute negativity signal from NLP analysis of headline/summary | Compromise can assess sentence tone via verb analysis and conflict term density |
| V2Themes boost + hard reject | Apply theme-like classification using NLP: categorize article by extracted entities and action verbs | The same semantic signal can be derived from NLP without external data |
| V2Locations as primary geo source | Extract locations from NLP (compromise `.places()`) + fall back to keyword geo matching | Already available in the NLP pipeline |
| V2Counts for ranking | Use matched keyword count + NLP triple completeness for ranking | Proxies the same signal |

**Alternative: GKG enrichment as a future enhancement.** If the user decides the NLP-only approach is insufficient, a GKG enrichment layer can be added later using the established `lastupdate.txt` pattern from `server/adapters/gdelt.ts`. This would be a separate phase.

### Pattern 1: NLP Triple Extraction
**What:** Extract actor-action-target triples from headlines and summaries using compromise pattern matching
**When to use:** For every article before scoring

```typescript
// server/lib/nlpExtractor.ts
import nlp from 'compromise';

export interface ArticleTriple {
  actor: string | null;
  action: string | null;
  target: string | null;
}

export function extractTriple(text: string): ArticleTriple {
  const doc = nlp(text);

  // Strategy: Use compromise match patterns to find actor-verb-target structures
  // Pattern: [ProperNoun/Noun phrase] [Verb phrase] [ProperNoun/Noun phrase]
  const match = doc.match('[<actor>#ProperNoun+] [<action>#Verb+] [<target>#ProperNoun+]');

  if (match.found) {
    const groups = match.groups();
    return {
      actor: groups.actor?.text() || null,
      action: groups.action?.text() || null,
      target: groups.target?.text() || null,
    };
  }

  // Fallback: extract any available components
  const people = doc.people().out('array');
  const places = doc.places().out('array');
  const verbs = doc.verbs().out('array');

  return {
    actor: people[0] || places[0] || null,
    action: verbs[0] || null,
    target: people[1] || places[1] || null,
  };
}
```

### Pattern 2: Relevance Scoring
**What:** Compute 0-1 relevance score from multiple signals
**When to use:** After NLP extraction, before threshold filtering

```typescript
// server/lib/relevanceScorer.ts

interface ScoringInput {
  triple: ArticleTriple;
  source: string;
  title: string;
  summary?: string;
  keywords: string[];  // from reclassified keyword matching
}

const SOURCE_RELIABILITY: Record<string, number> = {
  'BBC': 1.0,
  'Al Jazeera': 0.95,
  'Reuters': 1.0,
  'Times of Israel': 0.9,
  'Tehran Times': 0.85,
  'Middle East Eye': 0.9,
  'GDELT': 0.7,  // Unknown GDELT sources get lower weight
};

export function computeRelevanceScore(input: ScoringInput): number {
  // Factor 1: NLP Triple Completeness (0-0.5)
  let tripleScore = 0;
  if (input.triple.actor) tripleScore += 0.15;
  if (input.triple.action) tripleScore += 0.2;
  if (input.triple.target) tripleScore += 0.15;
  // Full triple = 0.5, partial = proportional

  // Factor 2: Negativity/Conflict Signal (0-0.3)
  // Computed from NLP: presence of conflict verbs, negative modifiers
  const negativityScore = computeNegativity(input.title, input.summary);

  // Factor 3: Source Reliability (0-0.2)
  const reliability = SOURCE_RELIABILITY[input.source] ?? 0.6;
  const sourceScore = reliability * 0.2;

  return Math.min(1, tripleScore + negativityScore + sourceScore);
}
```

### Anti-Patterns to Avoid
- **Processing articles through NLP one at a time in a serial loop:** Compromise can process batches efficiently. Parse multiple articles in sequence but avoid creating a new `nlp()` instance per word -- pass the full text at once.
- **Downloading GKG raw files for metadata:** Adds multi-MB downloads, ZIP decompression, and CSV parsing per poll cycle. The NLP-first approach extracts equivalent signals from article text directly.
- **Trusting compromise for precise dependency parsing:** Compromise is "modest" NLP -- it excels at POS tagging and pattern matching but is not a dependency parser. Triple extraction will have gaps; the scoring formula must handle partial triples gracefully.
- **Blocking the response on NLP processing:** NLP enrichment should be part of the fetch-filter-cache pipeline, not a separate async step. The enriched data is cached, so NLP runs at most once per 15 minutes.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| POS tagging | Custom regex-based tagger | compromise `.match('#Verb')` | 83-tag grammar model, handles irregular forms |
| Named entity extraction | Keyword list of countries/orgs | compromise `.places()`, `.people()` | Handles variations, abbreviations, multi-word names |
| Sentence tokenization | Split on period/newline | compromise `.sentences()` | Handles abbreviations, decimal numbers, URLs |
| Text normalization | Custom lowercasing + stemming | compromise `.normalize()` | Handles contractions, unicode, possessives |

**Key insight:** The current `newsFilter.ts` is effectively a hand-rolled NLP system using regex word-boundary matching. Compromise replaces this with a proper linguistic analysis at minimal cost (~250KB).

## Common Pitfalls

### Pitfall 1: Headline-Only NLP Extraction
**What goes wrong:** Headlines are often compressed, missing articles/prepositions, using shorthand. NLP extraction fails on headline syntax like "Iran strikes back, missiles hit bases."
**Why it happens:** Headlines aren't grammatical sentences -- they use "headlinese" (dropped articles, copulas, tenseless verbs).
**How to avoid:** Always combine title + summary for NLP analysis. Process title first for quick extraction, fall back to summary for richer context. If both fail, rely on keyword reclassification score alone.
**Warning signs:** Low triple extraction rate (<30% of articles getting full triples).

### Pitfall 2: Over-Strict Threshold Causing Empty Feed
**What goes wrong:** With threshold at 0.7, too many legitimate articles get filtered out, leaving the news feed empty.
**Why it happens:** NLP extraction is imperfect; many real conflict articles score low because compromise can't parse the headline structure.
**How to avoid:** Log relevance scores in development to calibrate. The 0.7 threshold is configurable via env var for exactly this reason. Consider a "marginal" bucket (0.5-0.7) that can be shown in a debug mode.
**Warning signs:** News cluster count drops by >80% compared to current filter.

### Pitfall 3: Compromise Module Import in Serverless
**What goes wrong:** Compromise loads its English dictionary/grammar model synchronously on import. In serverless cold starts, this can add latency.
**Why it happens:** The ~250KB dictionary is parsed into memory on first require/import.
**How to avoid:** Import compromise at module level (top of file), not inside request handlers. The tsup bundle will include it. Test cold start time in Vercel preview deployment.
**Warning signs:** First request after deployment takes >3s.

### Pitfall 4: Type Widening of NewsArticle Breaks Downstream
**What goes wrong:** Adding `actor`, `action`, `target`, `relevanceScore` fields to `NewsArticle` breaks existing code that spreads or destructures the type.
**Why it happens:** The type is used across server and client (re-exported via `src/types/entities.ts`).
**How to avoid:** Make all new fields optional (`actor?: string`) for backward compatibility. Existing code ignores unknown fields when using TypeScript interfaces.
**Warning signs:** TypeScript errors in unrelated files after type change.

### Pitfall 5: RSS Articles Lacking Summary
**What goes wrong:** Some RSS feeds return articles without descriptions. NLP extraction from title-only text produces poor triples.
**Why it happens:** Not all RSS feeds include `<description>` elements for every item.
**How to avoid:** Track source-level extraction quality. For title-only articles, weight source reliability higher and triple completeness lower in the scoring formula.
**Warning signs:** RSS-sourced articles consistently score below threshold.

   - What's unclear: Whether the user considers NLP-derived signals an acceptable substitute for GDELT metadata fields, or if GKG enrichment is required.

2. **Triple Extraction Accuracy**
   - What we know: Compromise is "modest" NLP -- good at POS tagging, decent at pattern matching, not a dependency parser.
   - What's unclear: Extraction success rate on real GDELT headlines (headlinese style, non-standard grammar).
   - Recommendation: Build multiple extraction patterns (4-5 patterns for different headline structures), accept partial triples (actor-only or action-only), weight scoring accordingly. Log extraction rates in dev to calibrate.

3. **Score Calibration**
   - What we know: 0.7 threshold is deliberately strict per user preference. Formula weights are Claude's discretion.
   - What's unclear: How many articles the new pipeline will pass vs reject compared to current filter.
   - Recommendation: Run both old and new filters in parallel during development (log comparison). Tune weights based on real article sample before deploying.
