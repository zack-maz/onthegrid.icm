# Phase 20.2: Threat Heatmap Layer - Context

**Gathered:** 2026-03-23
**Status:** Ready for planning

<domain>
## Phase Boundary

A toggleable visualization layer that color-codes regions by GDELT conflict event density. Hot zones glow red, quiet areas stay dark. Uses existing event data from eventStore — no new API or data source. Plugs into the layer architecture built in Phase 20 (layerStore, MapLegend, LayerTogglesSlot).

</domain>

<decisions>
## Implementation Decisions

### Density Rendering
- Smooth Gaussian blur via Deck.gl HeatmapLayer — thermal signature / heat camera look
- 50km base influence radius per event, zoom-adaptive (tightens at local zoom, broadens at overview)
- Events blend into organic glowing blobs; high-density areas form bright cores fading to dark edges
- At overview (z4-6): broad regional threat zones visible; at local (z9+): precise hotspot clusters emerge

### Density Weighting
- Severity-weighted using type weights from `src/lib/severity.ts` — airstrikes/WMD (weight 10) burn brightest, minor events barely register
- Combined formula: typeWeight × recencyDecay per event, fed as HeatmapLayer weight
- Not raw event count — map reflects actual threat intensity, not just event frequency

### Color Scale & Intensity
- Dark-to-red monochrome ramp: transparent → #2d0000 (dark) → #8b1e1e (crimson) → #ef4444 (red) → #ff3b30 (bright) → #ff6b4a (white-hot core)
- Quiet areas are fully transparent (invisible), hot zones glow red
- 40-50% overall opacity — stronger than geographic/weather layers (20-30%) because threat data is primary intelligence
- Consistent with existing red entity colors (airstrikes #ff3b30, ground combat #ef4444)

### Layer Stacking Order
- Renders below entity icons, above geographic overlay
- Stack: basemap → geographic → **threat heatmap** → weather → entities → tooltips/UI
- Heatmap is environmental context; entities are foreground data points

### Legend
- Density gradient bar matching LEGEND_REGISTRY pattern (horizontal gradient, "Low" to "High" labels)
- Title: "Threat Density"
- Color stops: low (#2d0000) to high (#ff3b30)
- No numeric labels — density is relative

### Time Weighting
- Recency-decayed with 6h half-life (reuse exponential decay from `src/lib/severity.ts`)
- Fresh events (0h): full intensity; 6h: 50%; 12h: 25%; 24h+: barely visible
- Map always shows CURRENT threat, not historical accumulation
- Respects date range filter — when custom range is active, only events in that window feed the heatmap
- Recency decay still applies within the filtered window

### Live Reactivity
- Heatmap recomputes whenever eventStore.events changes (15-min poll cycle)
- Reactive derivation via useMemo — no extra polling needed
- Same pattern as useCounterData derives from stores

### Interaction Behavior
- Hover: zone summary tooltip showing event count, dominant event type, and latest event time
- Click: no action — clicks pass through to entity markers underneath
- Tooltip priority: entity > threat zone > weather (only one tooltip at a time)
- Consistent with weather layer pattern (hover tooltip, no click action)

### Claude's Discretion

- How to implement the zone tooltip picking (ScatterplotLayer grid or HeatmapLayer picking)
- Transition animation easing for layer toggle fade
- Exact zoom-adaptive radius scaling curve
- How to compute "dominant event type" in the tooltip (mode vs weighted)

## Specific Ideas

- Heatmap should feel like a thermal camera / heat signature overlay — organic red glow, not a data grid
- The visual distinction matters: geographic and weather layers are ambient context (20-30% opacity), threat heatmap is actionable intelligence (40-50% opacity)
- Zone tooltip is quick area intel — "12 events, mostly airstrikes, latest 2h ago" — not a deep analysis panel
