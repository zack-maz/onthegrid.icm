# Phase 20.2: Threat Heatmap Layer - Research

**Researched:** 2026-03-23
**Domain:** deck.gl HeatmapLayer aggregation, event density visualization, tooltip picking patterns
**Confidence:** HIGH

## Summary

This phase adds a toggleable threat density heatmap that renders GDELT conflict event data as a Gaussian-blurred heat overlay. The implementation sits firmly within established project patterns: deck.gl layer composition, layerStore gating, LEGEND_REGISTRY for inline legends, and the ScatterplotLayer-based tooltip picking strategy already proven in the weather layer.

The core rendering uses `HeatmapLayer` from `@deck.gl/aggregation-layers` (already installed as `^9.2.11`, already mocked for tests). Events are weighted using the existing `TYPE_WEIGHTS` from `src/lib/severity.ts` combined with a 6h half-life exponential decay. The key technical challenge is tooltip picking: HeatmapLayer is a continuous aggregation layer and does not provide per-object picking info (confirmed by deck.gl issue #6369). The established workaround in this codebase is using an invisible ScatterplotLayer grid for picking, exactly as WeatherOverlay.tsx does with its `weather-picker` layer.

**Primary recommendation:** Build a `ThreatHeatmapOverlay` component following the exact WeatherOverlay pattern -- a `useThreatHeatmapLayers()` hook returning HeatmapLayer + ScatterplotLayer picker, integrated into BaseMap's DeckGLOverlay layers array with tooltip priority: entity > threat > weather.

### Locked Decisions
- Smooth Gaussian blur via Deck.gl HeatmapLayer -- thermal signature / heat camera look
- 50km base influence radius per event, zoom-adaptive (tightens at local zoom, broadens at overview)

- Click: no action -- clicks pass through to entity markers underneath

### Pattern 1: Deck.gl Layer Hook (Follow WeatherOverlay.tsx)
**What:** A custom hook `useThreatHeatmapLayers()` that returns an array of deck.gl layers, gated by layerStore activation.
**When to use:** Every visualization layer in this project follows this pattern.
**Example:**
```typescript
// Source: WeatherOverlay.tsx pattern + deck.gl HeatmapLayer API
import { HeatmapLayer } from '@deck.gl/aggregation-layers';
import { ScatterplotLayer } from '@deck.gl/layers';

export function useThreatHeatmapLayers() {
  const events = useEventStore((s) => s.events);
  const isActive = useLayerStore((s) => s.activeLayers.has('threat'));
  const dateStart = useFilterStore((s) => s.dateStart);
  const dateEnd = useFilterStore((s) => s.dateEnd);

  return useMemo(() => {
    if (!isActive || events.length === 0) return [];

    // Filter events to date range, compute weights
    const weighted = events
      .filter(e => e.timestamp >= dateStart && e.timestamp <= dateEnd)
      .map(e => ({
        position: [e.lng, e.lat],
        weight: computeThreatWeight(e),
        ...e, // preserve for picker reference
      }));

    const heatmapLayer = new HeatmapLayer({
      id: 'threat-heatmap',
      data: weighted,
      getPosition: d => d.position,
      getWeight: d => d.weight,
      radiusPixels: 60,   // Claude's discretion
      colorRange: [...],   // dark-to-red locked ramp
      intensity: 1,
      threshold: 0.05,
      opacity: 0.45,
      aggregation: 'SUM',
      pickable: false,     // HeatmapLayer picking is unreliable
    });

    const pickerLayer = new ScatterplotLayer({
      id: 'threat-picker',
      data: gridPoints,    // Pre-aggregated zone centroids
      getPosition: d => [d.lng, d.lat],
      getRadius: 50000,
      radiusUnits: 'meters',
      getFillColor: [0, 0, 0, 0],  // invisible
      pickable: true,
    });

    return [heatmapLayer, pickerLayer];
  }, [isActive, events, dateStart, dateEnd]);
}
```

### Pattern 2: Tooltip Picker Grid (Follow weather-picker pattern)
**What:** An invisible ScatterplotLayer with pickable=true overlaid at regular grid intervals to provide tooltip data for the continuous heatmap.
**When to use:** When the primary visualization layer (HeatmapLayer) cannot provide per-object picking info.
**Implementation approach:**
- Generate a grid of aggregation cells covering the Middle East region (e.g., 1-degree spacing)
- For each cell, pre-compute: event count, dominant event type, latest event timestamp
- Feed these as ScatterplotLayer data with pickable=true
- On hover, BaseMap reads the aggregated cell data for tooltip content

### Anti-Patterns to Avoid
- **Using HeatmapLayer pickable for tooltips:** HeatmapLayer's `onHover` does NOT provide `info.object` data (deck.gl issue #6369). Always use a separate picker layer.
- **Re-fetching event data:** The heatmap must use `useEventStore.events` directly (or via `useFilteredEntities`), not create a new polling hook. Events are already polled every 15 minutes.
- **Putting heatmap in entity layers array:** The threat heatmap is a visualization layer separate from entity icons. It should be its own hook (like `useWeatherLayers`), not mixed into `useEntityLayers`.
- **Using computeSeverityScore for weights:** The locked decision specifies `typeWeight x recencyDecay` only, NOT the full severity formula which includes mentions/sources factors.

## Common Pitfalls

### Pitfall 1: HeatmapLayer radiusPixels vs. zoom-adaptive sizing
**What goes wrong:** Setting a fixed radiusPixels creates blobs that are too large at high zoom or too small at low zoom.
**Why it happens:** radiusPixels is in screen pixels, not meters. The locked decision specifies 50km base radius with zoom-adaptive behavior.
**How to avoid:** Use `radiusPixels` as a function of current zoom level, or use a reasonable compromise value (60-80px) and rely on HeatmapLayer's built-in zoom behavior. Note: HeatmapLayer recalculates density on viewport changes with `debounceTimeout` (default 500ms).
**Warning signs:** Heatmap looks like uniform soup at overview zoom, or tiny isolated dots at local zoom.

### Pitfall 2: Weight function mismatch with CONTEXT decisions
**What goes wrong:** Using `computeSeverityScore` (which includes mentions/sources/24h decay) instead of the locked formula (typeWeight x recencyDecay with 6h half-life).
**Why it happens:** severity.ts has a convenient exported function but it uses different decay parameters.
**How to avoid:** Create a dedicated `computeThreatWeight` function that uses TYPE_WEIGHTS but applies exponential decay with 6h half-life: `Math.pow(0.5, ageHours / 6)`. This is different from severity.ts's `1 / (1 + ageHours/24)` rational decay.
**Warning signs:** Old events have too much visual weight; heatmap doesn't feel "current threat."

### Pitfall 3: Tooltip picker grid alignment with heatmap visual
**What goes wrong:** Tooltip picker cells don't align with visual hot zones, causing confusing hover behavior.
**Why it happens:** HeatmapLayer uses Gaussian blur which smears beyond point locations; picker grid cells are fixed.
**How to avoid:** Use sufficiently large picker radius (50km matching influence radius) with reasonable grid spacing (1-2 degrees). Accept that tooltip zones are approximate, not pixel-perfect.
**Warning signs:** Hovering over visible red zone shows "0 events" tooltip.

### Pitfall 4: Layer ordering in DeckGLOverlay
**What goes wrong:** Heatmap renders above entity icons, obscuring clickable markers.
**Why it happens:** deck.gl layers render in array order (later = on top).
**How to avoid:** Threat heatmap layers must come BEFORE entity layers in the DeckGLOverlay layers array: `[...threatLayers, ...entityLayers, ...weatherLayers]`. Wait -- per locked stacking: basemap > geographic > threat heatmap > weather > entities. So: `[...threatLayers, ...weatherLayers, ...entityLayers]`.
**Warning signs:** Entity markers are invisible or unpickable when threat layer is active.

### Pitfall 5: Performance with large event datasets
**What goes wrong:** HeatmapLayer aggregation causes frame drops during map interaction.
**Why it happens:** Large datasets combined with large radiusPixels trigger expensive GPU aggregation.
**How to avoid:** Use `debounceTimeout: 500` (default) to delay re-aggregation during pan/zoom. The GDELT event dataset is typically hundreds to low thousands of events, which is well within HeatmapLayer's performance budget.
**Warning signs:** Map becomes laggy when panning with threat layer active.

### Pitfall 6: HeatmapLayer colorRange alpha channel
**What goes wrong:** Setting alpha in colorRange colors has no effect; the entire layer needs opacity control.
**Why it happens:** HeatmapLayer handles transparency via threshold and opacity props, not per-color alpha.
**How to avoid:** Use the `opacity` prop (0.4-0.5) for overall transparency. Use `threshold` (0.05) for fading low-weight areas to transparent. The `colorRange` should use fully opaque colors.
**Warning signs:** Quiet areas show faint color instead of being fully transparent.

## Code Examples

### Weight Computation Function
```typescript
// New function in ThreatHeatmapOverlay.tsx (not severity.ts)
// Uses TYPE_WEIGHTS from severity.ts but different decay

// TYPE_WEIGHTS must be exported from severity.ts (currently const, not exported)
const HALF_LIFE_HOURS = 6;

function computeThreatWeight(event: ConflictEventEntity): number {
  const typeWeight = TYPE_WEIGHTS[event.type] ?? 3;
  const ageMs = Math.max(0, Date.now() - event.timestamp);
  const ageHours = ageMs / (1000 * 60 * 60);
  // Exponential decay with 6h half-life (different from severity.ts rational decay)
  const recencyDecay = Math.pow(0.5, ageHours / HALF_LIFE_HOURS);
  return typeWeight * recencyDecay;
}
```

### Grid Aggregation for Tooltip Picker
```typescript
// Pre-aggregate events into grid cells for tooltip data
interface ThreatZoneData {
  lat: number;
  lng: number;
  eventCount: number;
  dominantType: string;
  latestTime: number;
}

function aggregateToGrid(
  events: ConflictEventEntity[],
  cellSize: number = 1, // degrees
): ThreatZoneData[] {
  const cells = new Map<string, {
    events: ConflictEventEntity[];
    lat: number;
    lng: number;
  }>();

  for (const e of events) {
    const cellLat = Math.floor(e.lat / cellSize) * cellSize + cellSize / 2;
    const cellLng = Math.floor(e.lng / cellSize) * cellSize + cellSize / 2;
    const key = `${cellLat},${cellLng}`;
    if (!cells.has(key)) {
      cells.set(key, { events: [], lat: cellLat, lng: cellLng });
    }
    cells.get(key)!.events.push(e);
  }

  return Array.from(cells.values())
    .filter(c => c.events.length > 0)
    .map(c => ({
      lat: c.lat,
      lng: c.lng,
      eventCount: c.events.length,
      dominantType: mode(c.events.map(e => e.type)),
      latestTime: Math.max(...c.events.map(e => e.timestamp)),
    }));
}
```

## Open Questions

1. **Exact radiusPixels value for 50km equivalent**
   - What we know: radiusPixels is in screen pixels, not meters. At zoom 5, 50km is roughly 50-80px. At zoom 10, 50km is ~1600px.
   - What's unclear: Whether to use a fixed value or zoom-responsive function. HeatmapLayer does NOT natively accept `radiusMinPixels`/`radiusMaxPixels` like IconLayer.
   - Recommendation: Use a fixed radiusPixels of 60-80 and let the Gaussian blur naturally create the zoom-adaptive effect. HeatmapLayer recalculates density on viewport changes. Test at z5 and z10 to validate visual appearance. If inadequate, use `useMap` to read current zoom and compute dynamic radiusPixels.

2. **TYPE_WEIGHTS export from severity.ts**
   - What we know: TYPE_WEIGHTS is currently a module-private `const` in severity.ts, not exported.
   - What's unclear: Whether to export it or duplicate the weights.
   - Recommendation: Export TYPE_WEIGHTS from severity.ts. This is the single source of truth per locked decision. Minor change, backward compatible.

3. **Grid cell size for tooltip picker**
   - What we know: Events are scattered across Middle East (lat 15-42, lng 30-70). A 1-degree grid produces ~1100 cells max. A 2-degree grid produces ~280 cells.
   - What's unclear: Whether finer or coarser grid provides better UX.
   - Recommendation: Start with 1-degree grid (good granularity for tooltip content). ScatterplotLayer handles thousands of points easily.
