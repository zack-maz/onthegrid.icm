# Phase 21.2: GDELT Event Quality Pipeline - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Improve GDELT event data accuracy with four server-side improvements: (1) geo-validation to discard misplaced events and detect city centroids, (2) expanded CAMEO base code classification, (3) Goldstein sanity checks with type reclassification, and (4) composite confidence scoring with configurable threshold filtering. All changes in `server/adapters/gdelt.ts` and new `server/lib/eventScoring.ts`. One minor client-side addition: dev-mode confidence score display in EventDetail.

Full design spec: `docs/superpowers/specs/2026-03-27-gdelt-event-quality-pipeline-design.md`

</domain>

<decisions>
## Implementation Decisions

### Pipeline observability
- **Per-event detail logging** for all pipeline stages: geo-validation discards, Goldstein reclassifications, and confidence threshold rejections
- Each log line includes event ID and reason: e.g., `"discarded gdelt-12345: FullName='New York, US' contradicts FIPS=IS"`
- **Pipeline summary line** at info level showing the full funnel: `"pipeline: 340 raw → 328 geo-valid → 5 reclassified → 290 above threshold → 290 final"`
- Uses existing structured `log()` from `server/lib/logger.ts`

### Confidence threshold
- **Default threshold: 0.35** (overrides spec's 0.15) — immediately filters events with almost no signal
- Configurable via `EVENT_CONFIDENCE_THRESHOLD` env var in `AppConfig`
- **Dev-mode score display** in EventDetail panel (`import.meta.env.DEV` guard, matching NotificationCard severity score pattern from Phase 17) — this is the one client-side change

### Non-ME country list
- Build from **most common non-ME ActionGeo_CountryCode values** in real GDELT data — Claude researches which countries appear most frequently in misplaced events
- Not a hardcoded shortlist of 5 — should be ~30 countries based on actual GDELT data patterns
- **FIPS reverse lookup is strict** — discard on contradiction (FullName country vs FIPS code mismatch), as spec describes

### Centroid detection
- Start with spec's 30 major city centroids
- **Claude's discretion** to expand with well-known conflict hotspot cities if they appear frequently in GDELT data
- **No country centroids** — cities only (country centroids are rarer and harder to distinguish from legitimate nearby events)
- ±0.01 degree tolerance (~1.1km) as spec describes

### Claude's Discretion
- Exact non-ME country list composition (~30 countries based on GDELT data analysis)
- Additional conflict hotspot cities for centroid table beyond the spec's 30
- Expanded CAMEO BASE_CODE_MAP entries (finer sub-type distinctions within existing 11 ConflictEventTypes, covering all codes in 180-200 range)
- Internal refactoring of `parseAndFilter` into Phase A / Phase B pipeline (implementation structure)

</decisions>

<specifics>
## Specific Ideas

- The core motivation mirrors Phase 21.1: "Tehran condemns airstrike in Yemen" should not place a marker on Tehran — but this time for EVENTS, not news articles. An assassination attempt in NYC on a pro-Palestinian activist gets pinned to Israel because Israel is the article's geopolitical subject.
- The 0.35 threshold is deliberately more aggressive than the spec's 0.15 — user prefers filtering junk immediately rather than shipping a no-op and tuning later. Can always tune down via env var if too aggressive.
- Per-event detail logging enables threshold tuning: monitor logs for a few days, adjust threshold based on real confidence distributions.
- Confidence score visible in dev mode only — consistent with "numbers over narratives" for the user, hidden from production.

</specifics>
