# Phase 20.1: Geographical & Weather Layers - Research

**Researched:** 2026-03-23
**Domain:** MapLibre terrain visualization, Deck.gl heatmap/icon layers, Open-Meteo API, geographic labeling
**Confidence:** HIGH

## Summary

This phase adds two visualization overlays to the existing dark basemap: (1) a geographic layer with elevation color tinting, contour lines, and feature labels, and (2) a weather layer with temperature heatmap, wind barbs, and weather tooltip. The geographic layer leverages MapLibre GL's native `color-relief` layer type (available since v5.20.0 -- the project has v5.20.1) combined with the `maplibre-contour` plugin for contour line generation, both using the already-configured AWS Terrarium DEM tiles. The weather layer uses Deck.gl's `HeatmapLayer` from `@deck.gl/aggregation-layers` (must be installed) for temperature visualization and `IconLayer` with programmatically generated SVG wind barb icons. Open-Meteo provides free, no-API-key weather data supporting up to 1000 locations per request with `current` snapshot parameters for temperature, wind speed, and wind direction.

The existing Phase 20 infrastructure (layerStore, MapLegend, LayerTogglesSlot) is fully ready for these layers. The `activeLayers` Set already includes `'geographic'` and `'weather'` IDs, toggle rows are wired, and `LEGEND_REGISTRY` is an empty array awaiting entries.

**Primary recommendation:** Use MapLibre native `color-relief` + `hillshade` layers for elevation (reusing existing terrain-dem source), `maplibre-contour` plugin for contour lines, Deck.gl `HeatmapLayer` for temperature, and Deck.gl `IconLayer` with SVG data URLs for wind barbs. Server-side Open-Meteo adapter fetches a 1-degree grid in 2 batched requests (split by latitude to stay under 1000-location API limit).

### Pattern 1: Geographic Overlay via MapLibre Native Layers
**What:** The geographic layer uses MapLibre's native `color-relief` layer type (reusing the existing `terrain-dem` raster-dem source) plus `maplibre-contour` for contour lines and a `symbol` layer with static GeoJSON for geographic labels.
**When to use:** When the overlay reads from the same DEM source that already powers 3D terrain.
**Implementation approach:**

The `color-relief` layer reuses the existing `terrain-dem` Source already defined in BaseMap.tsx. It uses the `['elevation']` expression for color mapping. The layer is added/removed imperatively via `useMap()` hook when the `geographic` layer is toggled, following the established pattern of imperative style customization in `onLoad`.

Alternatively, since the project uses `<Source>` and `<Layer>` JSX components from react-maplibre, the color-relief layer can be rendered declaratively inside the `<Map>` component, conditionally based on `layerStore.activeLayers.has('geographic')`. This is cleaner and follows the existing pattern in BaseMap (where `<Layer id="terrain-hillshade">` is already declared).

**Recommendation:** Use conditional JSX `<Layer>` components gated by layerStore state. This avoids imperative addLayer/removeLayer complexity and keeps the rendering declarative.

```typescript
// Inside <Map> component in BaseMap.tsx (or in GeographicOverlay child component)
{isGeographicActive && (
  <>
    <Layer
      id="elevation-tint"
      type="color-relief"
      source="terrain-dem"
      paint={{
        'color-relief-opacity': 0.25,
        'color-relief-color': [
          'interpolate', ['linear'], ['elevation'],
          0, '#1a1a2e',
          1500, '#334155',
          4000, '#94a3b8',
        ],
      }}
    />
    <Source id="contour-source" type="vector" tiles={[contourTileUrl]} />
    <Layer
      id="contour-lines"
      type="line"
      source="contour-source"
      sourceLayer="contours"
      paint={{
        'line-color': 'rgba(255, 255, 255, 0.18)',
        'line-width': ['case', ['>', ['get', 'level'], 0], 1.2, 0.6],
      }}
      minzoom={7}
    />
  </>
)}
```

### Pattern 2: Weather Layer via Deck.gl
**What:** Temperature heatmap uses `HeatmapLayer` from `@deck.gl/aggregation-layers`; wind barbs use `IconLayer` with programmatically generated SVG data URLs.
**When to use:** For GPU-accelerated overlay rendering that integrates with the existing DeckGLOverlay.
**Implementation approach:**

Weather layers are added to the `entityLayers` array returned by `useEntityLayers()` (or a separate `useWeatherLayers()` hook) and passed to `DeckGLOverlay`. They are conditionally included when `layerStore.activeLayers.has('weather')` is true.

```typescript
// HeatmapLayer for temperature
new HeatmapLayer({
  id: 'weather-temp-heatmap',
  data: weatherGrid,
  getPosition: (d) => [d.lng, d.lat],
  getWeight: (d) => d.temperature,
  radiusPixels: 60,
  intensity: 1,
  threshold: 0.05,
  colorRange: [
    [0, 0, 255, 70],    // cold blue
    [0, 200, 0, 70],    // green
    [255, 255, 0, 70],  // yellow
    [255, 0, 0, 70],    // hot red
  ],
  colorDomain: [-5, 45], // Celsius range for Middle East
  aggregation: 'MEAN',
  visible: isWeatherActive,
})
```

### Pattern 3: Programmatic Wind Barb SVG Generation
**What:** Generate SVG wind barb icons dynamically based on wind speed, encode as data URLs, and render via Deck.gl `IconLayer` with `getIcon` returning per-point URLs.
**When to use:** When each data point needs a unique icon (wind speed determines barb count/type).
**Implementation approach:**

The IconLayer supports dynamic icon packing -- each unique `getIcon` return value is loaded and packed into an internal atlas. For ~150-200 wind barbs with varying speeds, the number of unique SVG variants is bounded (0, 5, 10, 15, 20, 25, 30, 35, 40, 45, 50+ knots = ~12-15 unique icons at 5-knot granularity). Pre-generate these SVGs, cache as data URLs, and select by quantizing wind speed.

```typescript
// windBarbs.ts
function generateWindBarbSVG(speedKnots: number): string {
  // Build SVG with appropriate barbs/flags based on speed
  // Short barb = 5kn, long barb = 10kn, triangle = 50kn
  const svg = `<svg xmlns="http://www.w3.org/2000/svg" width="32" height="64">...</svg>`;
  return `data:image/svg+xml;charset=utf-8,${encodeURIComponent(svg)}`;
}

// Pre-generate cache for 5-knot buckets (0 to 100 knots)
const BARB_CACHE = new Map<number, string>();
for (let kn = 0; kn <= 100; kn += 5) {
  BARB_CACHE.set(kn, generateWindBarbSVG(kn));
}

export function getWindBarbIcon(speedKnots: number): string {
  const bucket = Math.round(speedKnots / 5) * 5;
  return BARB_CACHE.get(Math.min(bucket, 100))!;
}
```

### Pattern 4: Geographic Feature Labels via GeoJSON Symbol Layer
**What:** Static GeoJSON FeatureCollection with ~15 labeled point/line features, rendered as a MapLibre `symbol` layer with cartographic text styling.
**When to use:** For hardcoded geographic labels that don't change.
**Implementation approach:**

Create a static GeoJSON source with point features for each labeled geographic feature. Each feature has properties: `name` (letterspaced display text), `minzoom` (zoom tier), `category` (mountain, desert, water, river). The symbol layer uses `text-field: ['get', 'name']`, `text-letter-spacing: 0.3`, `text-transform: 'uppercase'`, and `minzoom` filter expressions.

### Pattern 5: Weather Tooltip Coexistence
**What:** Weather tooltip must not conflict with EntityTooltip.
**When to use:** When hovering over weather overlay area that has no entity.
**Recommendation:** Use Deck.gl `onHover` on the HeatmapLayer (or a transparent ScatterplotLayer at grid points) to detect weather-area hovers. Weather tooltip renders at a lower priority than entity tooltip -- if an entity is hovered, entity tooltip wins. If no entity is hovered and weather layer is active, weather tooltip shows. This can be managed in BaseMap's hover handler by checking whether the hovered object came from a weather layer or entity layer.

### Anti-Patterns to Avoid
- **Imperative addLayer/removeLayer for toggle:** Use conditional JSX `<Layer>` components instead; cleaner lifecycle management
- **Fetching full hourly forecast:** Use `current=` parameter for snapshot data, not `hourly=` which returns 168 data points per location
- **Single monolithic Open-Meteo request for 1000+ points:** Split into 2 requests by latitude band to stay under 1000-location limit
- **Custom canvas-based heatmap:** Deck.gl HeatmapLayer does GPU-accelerated Gaussian KDE natively
- **Modifying terrain exaggeration when geographic toggles:** 3D terrain exaggeration stays at 3.0x (locked decision)

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Contour line generation from DEM | Marching squares implementation | `maplibre-contour` plugin | Handles tile fetching, web worker, MVT encoding; battle-tested |
| Heatmap interpolation | Canvas-based kernel density estimation | Deck.gl `HeatmapLayer` | GPU-accelerated, handles viewport aggregation, color mapping |
| Elevation color mapping | Custom shader or tile recoloring | MapLibre `color-relief` layer type | Native layer type with interpolate expressions, uses existing DEM source |
| Wind barb SVG rendering | Complex canvas path drawing | Pre-generated SVG data URLs with Deck.gl IconLayer | Simpler, cacheable, leverages existing WebGL pipeline |
| Weather data grid API | Scraping weather services or manual interpolation | Open-Meteo API with comma-separated coordinates | Free, no API key, returns structured JSON, supports 1000 locations per request |

**Key insight:** Every visualization component in this phase has an established library solution. The geographic layer is entirely MapLibre-native (color-relief + hillshade + symbol layers). The weather layer is entirely Deck.gl-native (HeatmapLayer + IconLayer). The data source is a single free API with batch support.

## Common Pitfalls

### Pitfall 1: Open-Meteo 1000-Location Limit
**What goes wrong:** The 1-degree grid across 15-42N, 30-70E produces ~1,148 points, exceeding the 1000-location API limit.
**Why it happens:** The IRAN_BBOX spans 28 degrees latitude x 41 degrees longitude.
**How to avoid:** Split into 2 requests by latitude band (e.g., lat 15-28 = 574 points, lat 29-42 = 574 points). Combine results server-side before caching.
**Warning signs:** API returns error or truncated results.

### Pitfall 2: HeatmapLayer Requires @deck.gl/aggregation-layers
**What goes wrong:** `import { HeatmapLayer } from '@deck.gl/layers'` fails -- HeatmapLayer is NOT in the base layers package.
**Why it happens:** Deck.gl splits aggregation layers into a separate package to reduce bundle size.
**How to avoid:** Install `@deck.gl/aggregation-layers@^9.2.11` and import from there. Must also add test alias in `vite.config.ts`.
**Warning signs:** TypeScript compile error or runtime import failure.

### Pitfall 3: color-relief Layer Type Not Recognized by react-maplibre Types
**What goes wrong:** TypeScript may error on `type="color-relief"` if `@vis.gl/react-maplibre` types are behind maplibre-gl.
**Why it happens:** The `color-relief` layer type was added in maplibre-gl 5.20.0 and the style spec may not be typed in react-maplibre 8.1.0.
**How to avoid:** Cast the type prop if needed: `type={'color-relief' as any}`. Verify at runtime that the layer renders correctly.
**Warning signs:** TypeScript error on `<Layer type="color-relief">`.

### Pitfall 4: maplibre-contour Protocol Registration Timing
**What goes wrong:** Contour tiles fail to load if the custom protocol is registered after the Map component mounts.
**Why it happens:** `addProtocol` must be called before the Map instance tries to fetch tiles from the custom URL scheme.
**How to avoid:** Register the contour protocol in a top-level `useEffect` with `[]` dependency, or call it at module scope before React renders the Map. The `demSource.setupMaplibre(maplibregl)` call must happen once at app initialization.
**Warning signs:** 404 or failed tile requests in network tab; blank contour layer.

### Pitfall 5: HeatmapLayer iOS Safari Precision
**What goes wrong:** On iOS Safari, HeatmapLayer uses 8-bit precision fallback; weights must be integers not exceeding 255 per pixel.
**Why it happens:** WebGL constraints on mobile Safari.
**How to avoid:** Normalize temperature weights to 0-255 range using colorDomain. Set `colorDomain` explicitly rather than relying on auto-calculation.
**Warning signs:** Incorrect color mapping or visual artifacts on iOS.

### Pitfall 6: Weather Tooltip vs Entity Tooltip Conflict
**What goes wrong:** Both tooltips appear simultaneously or entity tooltip disappears when hovering weather-covered area.
**Why it happens:** Deck.gl hover events fire for all pickable layers; weather and entity layers may overlap.
**How to avoid:** Make HeatmapLayer `pickable: false`. Instead, use a transparent ScatterplotLayer at grid points for weather picking, layered below entity layers. In the hover handler, prioritize entity objects over weather objects.
**Warning signs:** Tooltip flickering, wrong tooltip shown, or tooltip disappearing unexpectedly.

### Pitfall 7: Contour Lines and 3D Terrain Interaction
**What goes wrong:** Contour lines render flat while terrain has 3D exaggeration, looking misaligned.
**Why it happens:** MapLibre symbol/line layers don't drape on terrain by default.
**How to avoid:** This is expected behavior -- contour lines render in screen space overlaid on the terrain. At the subtle opacity levels specified (15-20%), the slight misalignment is imperceptible. If it becomes an issue, reducing line opacity at high pitch angles is an option.
**Warning signs:** Contour lines visibly floating above or below terrain at steep pitch angles.

### Open-Meteo Adapter (Server)
```typescript
// server/adapters/open-meteo.ts
interface WeatherGridPoint {
  lat: number;
  lng: number;
  temperature: number;     // Celsius
  windSpeed: number;        // knots
  windDirection: number;    // degrees
}

interface OpenMeteoResponse {
  latitude: number;
  longitude: number;
  current: {
    temperature_2m: number;
    wind_speed_10m: number;
    wind_direction_10m: number;
  };
}

const BASE_URL = 'https://api.open-meteo.com/v1/forecast';
const GRID_STEP = 1; // degrees

function buildGrid(latMin: number, latMax: number, lngMin: number, lngMax: number) {
  const lats: number[] = [];
  const lngs: number[] = [];
  for (let lat = latMin; lat <= latMax; lat += GRID_STEP) lats.push(lat);
  for (let lng = lngMin; lng <= lngMax; lng += GRID_STEP) lngs.push(lng);

  const coords: { lat: number; lng: number }[] = [];
  for (const lat of lats) {
    for (const lng of lngs) {
      coords.push({ lat, lng });
    }
  }
  return coords;
}

export async function fetchWeather(): Promise<WeatherGridPoint[]> {
  // Split grid into 2 latitude bands to stay under 1000-location limit
  const band1 = buildGrid(15, 28, 30, 70); // ~574 points
  const band2 = buildGrid(29, 42, 30, 70); // ~574 points

  const results: WeatherGridPoint[] = [];

  for (const band of [band1, band2]) {
    const latStr = band.map(c => c.lat).join(',');
    const lngStr = band.map(c => c.lng).join(',');
    const url = `${BASE_URL}?latitude=${latStr}&longitude=${lngStr}&current=temperature_2m,wind_speed_10m,wind_direction_10m&wind_speed_unit=kn&forecast_days=1`;

    const res = await fetch(url, { signal: AbortSignal.timeout(30_000) });
    if (!res.ok) throw new Error(`Open-Meteo returned ${res.status}`);

    const data: OpenMeteoResponse[] = await res.json();
    for (const point of data) {
      results.push({
        lat: point.latitude,
        lng: point.longitude,
        temperature: point.current.temperature_2m,
        windSpeed: point.current.wind_speed_10m,
        windDirection: point.current.wind_direction_10m,
      });
    }
  }

  return results;
}
```

### Weather Route (Server)
```typescript
// server/routes/weather.ts
import { Router } from 'express';
import { cacheGet, cacheSet } from '../cache/redis.js';
import { fetchWeather } from '../adapters/open-meteo.js';

const WEATHER_KEY = 'weather:open-meteo';
const LOGICAL_TTL_MS = 1_800_000;    // 30 min
const REDIS_TTL_SEC = 18_000;        // 5h hard TTL (10x logical)

export const weatherRouter = Router();

weatherRouter.get('/', async (req, res) => {
  const cached = await cacheGet(WEATHER_KEY, LOGICAL_TTL_MS);
  if (cached && !cached.stale) return res.json(cached);

  try {
    const grid = await fetchWeather();
    await cacheSet(WEATHER_KEY, grid, REDIS_TTL_SEC);
    res.json({ data: grid, stale: false, lastFresh: Date.now() });
  } catch (err) {
    console.error('[weather] Open-Meteo error:', (err as Error).message);
    if (cached) {
      res.json({ data: cached.data, stale: true, lastFresh: cached.lastFresh });
    } else {
      throw err;
    }
  }
});
```

### Wind Barb SVG Generator
```typescript
// src/components/map/layers/windBarbs.ts
const BARB_WIDTH = 32;
const BARB_HEIGHT = 64;
const STAFF_LENGTH = 50;

export function generateWindBarbSVG(speedKnots: number): string {
  const paths: string[] = [];
  let remaining = Math.round(speedKnots / 5) * 5; // Round to nearest 5
  let y = 4; // Start from top of staff

  // Staff line
  paths.push(`<line x1="16" y1="4" x2="16" y2="${4 + STAFF_LENGTH}" stroke="white" stroke-width="1.5" stroke-opacity="0.6"/>`);

  // Triangles (50 kn each)
  while (remaining >= 50) {
    paths.push(`<polygon points="16,${y} 28,${y + 4} 16,${y + 8}" fill="white" fill-opacity="0.6"/>`);
    y += 10;
    remaining -= 50;
  }

  // Long barbs (10 kn each)
  while (remaining >= 10) {
    paths.push(`<line x1="16" y1="${y}" x2="28" y2="${y - 4}" stroke="white" stroke-width="1.5" stroke-opacity="0.6"/>`);
    y += 6;
    remaining -= 10;
  }

  // Short barb (5 kn)
  if (remaining >= 5) {
    paths.push(`<line x1="16" y1="${y}" x2="22" y2="${y - 3}" stroke="white" stroke-width="1.5" stroke-opacity="0.6"/>`);
  }

  const svg = `<svg xmlns="http://www.w3.org/2000/svg" width="${BARB_WIDTH}" height="${BARB_HEIGHT}" viewBox="0 0 ${BARB_WIDTH} ${BARB_HEIGHT}">${paths.join('')}</svg>`;
  return `data:image/svg+xml;charset=utf-8,${encodeURIComponent(svg)}`;
}

// Pre-generate cache for all 5-knot buckets
const BARB_ICON_CACHE = new Map<number, string>();
for (let kn = 0; kn <= 100; kn += 5) {
  BARB_ICON_CACHE.set(kn, generateWindBarbSVG(kn));
}

export function getWindBarbIcon(speedKnots: number): string {
  const bucket = Math.min(Math.round(speedKnots / 5) * 5, 100);
  return BARB_ICON_CACHE.get(bucket)!;
}
```

## State of the Art

## Open Questions

1. **color-relief TypeScript types in react-maplibre 8.1.0**
   - What we know: `color-relief` is a valid layer type in maplibre-gl 5.20.1
   - What's unclear: Whether `@vis.gl/react-maplibre` 8.1.0 has updated its TypeScript types to include `color-relief` as a valid `type` prop
   - Recommendation: Try it first; if types fail, use `type={'color-relief' as any}` and add a comment explaining why

2. **HeatmapLayer opacity control granularity**
   - What we know: HeatmapLayer controls opacity through `colorRange` alpha channel and `colorDomain`/`threshold`
   - What's unclear: Whether 20-30% opacity can be achieved precisely without visual artifacts
