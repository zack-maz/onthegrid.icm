# Phase 19.2: Counter Entity Dropdowns - Research

**Researched:** 2026-03-22
**Domain:** React UI expansion/accordion patterns, entity list rendering, fly-to map integration
**Confidence:** HIGH

## Summary

This phase transforms the existing `CounterRow` component from a static label+value display into an interactive expandable row that reveals a scrollable entity list. The technical challenge is moderate: extending `useCounterData` to expose entity arrays alongside counts, implementing accordion expand/collapse with CSS max-height transitions (pattern already established in `MarketRow`), adding per-category proximity sorting via the existing `haversineKm` utility, and wiring click-to-fly-to using the `setFlyToTarget` + `selectEntity` + `openDetailPanel` pattern from `SearchModal`.

The main complexity lies in two areas: (1) the dual render sites for counters -- both `CountersSlot.tsx` (standalone overlay, now unused but may still be imported) and `Sidebar.tsx` `CountersContent` -- must be kept in sync; and (2) the scroll range indicator ("Showing 8-16 of 142") requires scroll position tracking with an `onScroll` handler and arithmetic based on item height.

**Primary recommendation:** Extend `useCounterData` to return both counts and entity arrays, refactor `CounterRow` to accept an optional `entities` array prop and manage its own expand/collapse state (with accordion coordination from parent via controlled `expandedKey` prop), and reuse existing `haversineKm` + `setFlyToTarget` patterns verbatim.

### Pattern 1: Accordion with Controlled Expand State
**What:** Parent component (`CountersContent` in Sidebar) manages a single `expandedKey: string | null` state. Each `CounterRow` receives `isExpanded` and `onToggle` props. Toggling a row sets that key; toggling the same row sets null.
**When to use:** Accordion behavior where only one row can be expanded at a time.
**Example:**
```typescript
// Source: existing MarketRow pattern in MarketsSlot.tsx
const [expandedKey, setExpandedKey] = useState<string | null>(null);

// In render:
<CounterRow
  label="Airstrikes"
  value={counters.airstrikes}
  entities={counterEntities.airstrikeEntities}
  isExpanded={expandedKey === 'airstrikes'}
  onToggle={() => setExpandedKey(prev => prev === 'airstrikes' ? null : 'airstrikes')}
  onEntityClick={handleEntityClick}
/>
```

### Pattern 2: CSS max-height Transition (Accordion Slide)
**What:** The dropdown container uses `overflow-hidden` with `transition-[max-height]` and dynamically sets `maxHeight` between 0 and the target height (~200px).
**When to use:** Smooth expand/collapse animation without JavaScript measurement.
**Example:**
```typescript
// Source: MarketRow.tsx line 96-99
<div
  className="overflow-hidden transition-[max-height] duration-150 ease-in-out"
  style={{ maxHeight: isExpanded ? 200 : 0 }}
>
  {/* Entity list content */}
</div>
```

### Pattern 3: Fly-To + Select + Open Detail Panel
**What:** Clicking an entity in the dropdown triggers three actions: fly map to coordinates, select entity in uiStore, open detail panel.
**When to use:** Any UI element that should navigate to an entity on the map.
**Example:**
```typescript
// Source: SearchModal.tsx handleSelect (lines 42-56)
const handleEntityClick = useCallback((entity: MapEntity | SiteEntity) => {
  useNotificationStore.getState().setFlyToTarget({
    lng: entity.lng,
    lat: entity.lat,
    zoom: 10,
  });
  useUIStore.getState().selectEntity(entity.id);
  useUIStore.getState().openDetailPanel();
  // NOTE: do NOT close dropdown (locked decision)
}, []);
```

### Pattern 4: Extending useCounterData to Return Entity Arrays
**What:** `useCounterData` currently returns only counts. It must be extended to also return the filtered entity arrays that underpin those counts, so CounterRow can render the entity list.
**When to use:** When the dropdown needs the actual entity objects, not just their count.
**Example:**
```typescript
export interface CounterEntities {
  unidentifiedFlights: FlightEntity[];
  airstrikeEvents: ConflictEventEntity[];
  groundCombatEvents: ConflictEventEntity[];
  targetedEvents: ConflictEventEntity[];
  hitSitesByType: Record<SiteType, Array<SiteEntity & { attackCount: number }>>;
}
```

### Anti-Patterns to Avoid
- **Computing entity arrays in CounterRow:** Each CounterRow should NOT independently subscribe to stores and filter entities. This would create N separate subscriptions. Instead, compute once in `useCounterData` and pass down.
- **Using state for scroll range indicator:** The "Showing X-Y of Z" text should derive from scroll position + item height arithmetic in an `onScroll` handler, not stored in React state (avoids re-render cascade during scroll). Use a ref to update text content directly, or use lightweight state updated via `onScroll`.
- **Mounting entity list when collapsed:** Use conditional rendering (`isExpanded && <EntityList />`) in addition to CSS transition, to avoid rendering hundreds of hidden DOM nodes for all counter rows simultaneously. Mount on expand, unmount after collapse animation completes (or keep mounted while transition runs for smoothness).

## Common Pitfalls

### Pitfall 1: Dual Render Sites Out of Sync
**What goes wrong:** `CountersSlot.tsx` and `Sidebar.tsx`'s `CountersContent` both render counter rows. Changes to one without updating the other creates inconsistency.
**Why it happens:** `CountersSlot` was the original standalone overlay. The Sidebar refactored counters into `CountersContent`. Both still exist in the codebase.
**How to avoid:** Check if `CountersSlot` is still rendered anywhere. If so, extract shared counter content into a single component used by both. If `CountersSlot` is vestigial, consider removing or redirecting it.
**Warning signs:** Counter rows render differently in sidebar vs standalone overlay.

### Pitfall 2: Re-render Cascade from Entity Arrays
**What goes wrong:** `useCounterData` returning entity arrays means every poll cycle produces new array references, causing all CounterRows to re-render even if their specific entities haven't changed.
**Why it happens:** Zustand store updates replace the entire flights/events array atomically. `useMemo` in `useCounterData` would recompute on every store update.
**How to avoid:** The existing `useMemo` dependency array already gates recomputation on the filtered arrays from `useFilteredEntities`. As long as the new entity arrays are computed inside the same `useMemo` block, they'll only recompute when the inputs change. The entity arrays within a given category are small (typically <200 items), so sorting on each recomputation is cheap.
**Warning signs:** Visible jank or console profiler showing excessive CounterRow re-renders.

### Pitfall 3: max-height Transition with Dynamic Content
**What goes wrong:** CSS `max-height` transition with a fixed large value (e.g., 200px) means the collapse animation is too slow for lists shorter than the max, because CSS transitions over the full range.
**Why it happens:** CSS doesn't know the actual content height; it transitions between 0 and the specified max-height regardless of actual content.
**How to avoid:** Set max-height to exactly `200px` (8 items * 25px each) which is the capped display height. For lists with fewer items, the slight timing discrepancy is negligible at 150ms duration. Alternatively, use `grid-template-rows: 0fr/1fr` transition pattern for content-height-aware animation.
**Warning signs:** Collapse animation feels sluggish for rows with only 1-2 entities.

### Pitfall 4: Scroll Range Indicator Performance
**What goes wrong:** Updating "Showing X-Y of Z" on every scroll event with `setState` causes excessive re-renders.
**Why it happens:** Scroll events fire at high frequency (60+ times per second during smooth scroll).
**How to avoid:** Either (a) use a ref to directly update the DOM text content, or (b) throttle the scroll handler. Since this is a small scrollable area with discrete items, throttling to ~100ms is sufficient.
**Warning signs:** React DevTools shows rapid state updates during scroll.

### Pitfall 5: Site Attack Count Computation Cost
**What goes wrong:** For hit sites, computing `attackCount` requires cross-referencing each site against all events via `computeAttackStatus`. Doing this in a hot render path is expensive.
**Why it happens:** `computeAttackStatus` does haversine distance calculations for each site against all events.
**How to avoid:** This computation is already done in `useCounterData` for count purposes. Extend it to capture and cache the attack count alongside the filtered site, reusing the same pass rather than computing twice.
**Warning signs:** Sidebar counter expansion causing frame drops with many events.

### Proximity Sort Implementation
```typescript
// Source: src/lib/geo.ts haversineKm
import { haversineKm } from '@/lib/geo';

const TEHRAN = { lat: 35.69, lng: 51.39 };
const TEL_AVIV = { lat: 32.07, lng: 34.78 };
const STRAIT_HORMUZ = { lat: 26.56, lng: 56.25 };

function sortByProximity<T extends { lat: number; lng: number }>(
  entities: T[],
  primary: { lat: number; lng: number },
  secondary?: { lat: number; lng: number },
): T[] {
  return [...entities].sort((a, b) => {
    const dA = haversineKm(a.lat, a.lng, primary.lat, primary.lng);
    const dB = haversineKm(b.lat, b.lng, primary.lat, primary.lng);
    if (dA !== dB) return dA - dB;
    if (secondary) {
      return haversineKm(a.lat, a.lng, secondary.lat, secondary.lng) -
             haversineKm(b.lat, b.lng, secondary.lat, secondary.lng);
    }
    return 0;
  });
}
```

### Scroll Range Indicator
```typescript
// Scroll handler for "Showing X-Y of Z" range display
const ITEM_HEIGHT = 25; // px per entity row
const VISIBLE_ITEMS = 8;

function handleScroll(e: React.UIEvent<HTMLDivElement>, total: number) {
  const scrollTop = e.currentTarget.scrollTop;
  const firstVisible = Math.floor(scrollTop / ITEM_HEIGHT) + 1;
  const lastVisible = Math.min(firstVisible + VISIBLE_ITEMS - 1, total);
  // Update range display (via ref or throttled state)
  return { first: firstVisible, last: lastVisible, total };
}
```

### Entity Metric Formatting
```typescript
// Per-type metric extraction
function getEntityMetric(entity: MapEntity | SiteEntity, attackCount?: number): string {
  switch (entity.type) {
    case 'flight': {
      const alt = entity.data.altitude;
      return alt !== null ? `${Math.round(alt * 3.281).toLocaleString()} ft` : 'N/A';
    }
    case 'ship':
      return `${entity.data.speedOverGround.toFixed(1)} kn`;
    case 'site':
      return attackCount !== undefined ? `${attackCount} attack${attackCount !== 1 ? 's' : ''}` : '';
    default: // conflict events
      return `GS ${entity.data.goldsteinScale.toFixed(1)}`;
  }
}
```
