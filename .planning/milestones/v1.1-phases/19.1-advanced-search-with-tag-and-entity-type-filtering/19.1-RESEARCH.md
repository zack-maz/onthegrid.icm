# Phase 19.1: Advanced Search with Tag and Entity Type Filtering - Research

**Researched:** 2026-03-22
**Domain:** Query parsing, boolean expression evaluation, bidirectional state sync, autocomplete UI
**Confidence:** HIGH

## Summary

This phase transforms the existing simple substring search (Phase 19) into a structured tag-based query system with boolean expression parsing, two-stage autocomplete, bidirectional sync with sidebar filter controls, and syntax-highlighted inline tags. The core architectural challenge is unifying search bar and sidebar filters into a single filter state -- making them two UIs for one truth source.

The existing `searchStore` holds a plain `query` string and a `matchedIds` Set. This must evolve into a parsed query AST that can be bidirectionally synchronized with `uiStore` toggles and `filterStore` parameters. The existing `searchUtils.ts` substring matcher must be replaced by a tag-aware boolean expression evaluator. The rendering pipeline (`useEntityLayers` consuming `matchedIds`) remains unchanged, which is a significant architectural advantage.

**Primary recommendation:** Build a pure, testable query parser (`src/lib/queryParser.ts`) that produces a typed AST, and a separate evaluator (`src/lib/queryEvaluator.ts`) that walks the AST against entity data. Keep the AST as the canonical representation in searchStore, with serialization to/from display string. Wire bidirectional sync through a coordination layer that prevents infinite update loops.

### Pattern 1: Recursive Descent Parser for Query AST
**What:** A hand-rolled tokenizer and recursive descent parser that converts query strings like `type:flight AND (country:iran OR country:iraq)` into a typed AST.
**When to use:** Always -- this is the core query representation.
**Example:**
```typescript
// AST node types
type QueryNode =
  | { type: 'tag'; prefix: string; value: string; negated: boolean }
  | { type: 'text'; value: string }        // freeform substring
  | { type: 'and'; left: QueryNode; right: QueryNode }
  | { type: 'or'; left: QueryNode; right: QueryNode }
  | { type: 'not'; child: QueryNode };

// Grammar (informal):
// expr     = term (('AND' | implicit) term)*
// term     = factor ('OR' factor)*
// factor   = '!' factor | '(' expr ')' | tag | text
// tag      = PREFIX ':' VALUE
// text     = WORD+

function parse(input: string): QueryNode | null {
  const tokens = tokenize(input);
  if (tokens.length === 0) return null;
  // recursive descent...
}
```

### Pattern 3: Bidirectional Sync with Loop Prevention
**What:** A coordination hook (`useQuerySync`) that watches for changes in searchStore AST and propagates to uiStore/filterStore, and vice versa. Uses a "source of change" flag to prevent infinite update loops.
**When to use:** Mandatory for the search<->sidebar sync requirement.
**Example:**
```typescript
// In useQuerySync:
const syncSourceRef = useRef<'search' | 'sidebar' | null>(null);

// When search bar changes:
useEffect(() => {
  if (syncSourceRef.current === 'sidebar') {
    syncSourceRef.current = null;
    return; // Skip -- this was triggered by sidebar
  }
  syncSourceRef.current = 'search';
  // Parse AST, extract type: tags, update uiStore toggles
  // Extract country: tags, update filterStore countries
  // Extract since:/before: tags, update filterStore dateRange
  syncSourceRef.current = null;

// When sidebar toggles change:

    syncSourceRef.current = null;

  }
  syncSourceRef.current = 'sidebar';
  // Reconstruct query string from current toggle state + other tags
  syncSourceRef.current = null;

```

### Anti-Patterns to Avoid
- **Storing query as string AND parsed AST separately:** The AST is the source of truth. The display string is derived from it (for sidebar->search sync) or parsed into it (for search bar typing). Never store both and try to keep them in sync manually.
- **Mutating uiStore/filterStore from inside searchStore:** Use the coordination hook, not cross-store calls inside store actions. Store actions should be pure state transitions.
- **contentEditable for the input:** The locked decision says "cursor behavior stays natural" which means using a standard `<input>` element. contentEditable has IME, selection, and undo/redo issues.
- **Regex-only parsing:** The grammar includes parentheses and boolean operators which regex cannot handle correctly. A proper recursive descent parser is needed.

## Common Pitfalls

### Pitfall 1: Infinite Sync Loop Between Search Bar and Sidebar
**What goes wrong:** Typing `type:flight` updates searchStore, which triggers sync to uiStore (toggle Flights ON), which triggers sync back to searchStore (add `type:flight` tag), creating an infinite loop.
**Why it happens:** React effects fire synchronously in development mode and can cascade.
**How to avoid:** Use a `syncSourceRef` flag (Pattern 3 above) to track which UI initiated the change. The sync effect checks this flag and skips propagation when the change originated from the other direction.
**Warning signs:** Browser hanging, "Maximum update depth exceeded" errors.

### Pitfall 2: Input Cursor Jump During Syntax Highlighting
**What goes wrong:** When using a controlled input with overlay, re-rendering can reset the cursor position to the end of the input.
**Why it happens:** React's controlled input model resets selection on every render when `value` is updated.
**How to avoid:** Store and restore `selectionStart`/`selectionEnd` after each state update using a `useEffect` or `requestAnimationFrame` callback. Or use `onInput` events with uncontrolled mode and a ref.
**Warning signs:** Cursor jumping to end when typing in the middle of a query.

### Pitfall 3: Performance with Live Entity Data Autocomplete
**What goes wrong:** Computing unique values across all entity stores on every keystroke causes lag.
**Why it happens:** Flights update every 5-30s, entities can number in the hundreds. Iterating all entities for every autocomplete suggestion is expensive.
**How to avoid:** Use `useMemo` with entity arrays as deps to pre-compute available values once per data update, not per keystroke. Cache value extraction results in a ref. Debounce autocomplete computation (100-200ms).
**Warning signs:** Input lag, janky autocomplete dropdown updates.

### Pitfall 4: Query String Display Inconsistency During Sidebar Sync
**What goes wrong:** When sidebar toggles change, the reconstructed query string doesn't match what the user originally typed (whitespace, ordering, casing differences).
**Why it happens:** Serializing AST back to string loses original formatting.
**How to avoid:** When the user is actively typing (input focused), don't overwrite their text with serialized AST. Only update the display string when the change came from sidebar. Track input focus state.
**Warning signs:** User's cursor position changes unexpectedly, typed text rewrites itself.

### Pitfall 5: Squawk Tag Has No Data
**What goes wrong:** The tag vocabulary includes `squawk:` but `FlightEntity.data` does not include a squawk field -- it exists in raw OpenSky data but is not propagated through the adapter.
**Why it happens:** The `squawk` column (index 14) is present in OpenSky API responses but was never added to the FlightEntity interface or normalization pipeline.
**How to avoid:** Either add `squawk` to the FlightEntity data model (requires server-side change) or exclude `squawk:` from the tag registry in this phase. Recommend: register the tag but return no autocomplete values and always pass (no-op) until data model is extended.
**Warning signs:** `squawk:7700` queries match nothing even when emergency squawk flights exist.

### Pitfall 6: OR/AND Precedence Confusion
**What goes wrong:** Users expect `type:flight country:iran OR country:iraq` to mean "flights from Iran or Iraq" but if AND binds tighter, it parses as `(type:flight AND country:iran) OR country:iraq`.
**Why it happens:** Standard boolean precedence (AND > OR) may not match user intuition.
**How to avoid:** Use standard precedence (AND binds tighter than OR, matching the locked decision's implicit-AND behavior) but make parentheses easy to use. The cheat sheet should show grouping examples prominently.
**Warning signs:** Users getting unexpected results with mixed AND/OR queries.

## Code Examples

### Query Tokenizer
```typescript
// Pure function, fully testable
type TokenType = 'TAG' | 'TEXT' | 'AND' | 'OR' | 'NOT' | 'LPAREN' | 'RPAREN';

interface Token {
  type: TokenType;
  value: string;
  prefix?: string;  // For TAG tokens: the part before ':'
  tagValue?: string; // For TAG tokens: the part after ':'
  start: number;     // Character position (for syntax highlighting)
  end: number;
}

function tokenize(input: string): Token[] {
  const tokens: Token[] = [];
  let i = 0;
  while (i < input.length) {
    // Skip whitespace
    if (/\s/.test(input[i])) { i++; continue; }
    // Parentheses
    if (input[i] === '(') { tokens.push({ type: 'LPAREN', value: '(', start: i, end: i + 1 }); i++; continue; }
    if (input[i] === ')') { tokens.push({ type: 'RPAREN', value: ')', start: i, end: i + 1 }); i++; continue; }
    // Read word
    const wordStart = i;
    while (i < input.length && !/[\s()]/.test(input[i])) i++;
    const word = input.slice(wordStart, i);
    // Check for AND/OR keywords
    if (word === 'AND') { tokens.push({ type: 'AND', value: word, start: wordStart, end: i }); continue; }
    if (word === 'OR') { tokens.push({ type: 'OR', value: word, start: wordStart, end: i }); continue; }
    // Check for negation prefix
    const negated = word.startsWith('!');
    const cleanWord = negated ? word.slice(1) : word;
    // Check for tag (contains ':')
    const colonIdx = cleanWord.indexOf(':');
    if (colonIdx > 0 && colonIdx < cleanWord.length - 1) {
      const prefix = cleanWord.slice(0, colonIdx);
      const tagValue = cleanWord.slice(colonIdx + 1);
      tokens.push({
        type: negated ? 'NOT' : 'TAG',
        value: word,
        prefix,
        tagValue,
        start: wordStart,
        end: i,
      });
    } else {
      tokens.push({ type: 'TEXT', value: word, start: wordStart, end: i });
    }
  }
  return tokens;
}
```

### Entity Evaluation for a Single Tag
```typescript
function evaluateTag(
  entity: MapEntity | SiteEntity,
  prefix: string,
  value: string,
): boolean {
  switch (prefix) {
    case 'type':
      return entity.type === value;
    case 'country':
      if (entity.type === 'flight')
        return entity.data.originCountry.toLowerCase() === value.toLowerCase();
      if ('data' in entity && 'actor1' in (entity as any).data) {
        const e = entity as ConflictEventEntity;
        return e.data.actor1.toLowerCase().includes(value.toLowerCase())
            || e.data.actor2.toLowerCase().includes(value.toLowerCase());
      }
      return false;
    case 'callsign':
      return entity.type === 'flight' && entity.label.toLowerCase().includes(value.toLowerCase());
    case 'near': {
      // value is a site label -- find site, compute haversine
      const sites = useSiteStore.getState().sites;
      const site = sites.find(s => s.label.toLowerCase().includes(value.toLowerCase()));
      if (!site) return false;
      return haversineKm(entity.lat, entity.lng, site.lat, site.lng) <= 50;
    }
    case 'severity': {
      if (!isConflictEventType(entity.type)) return false;
      const score = computeSeverityScore(entity as ConflictEventEntity);
      if (value === 'high') return score > 50;
      if (value === 'medium') return score > 20;
      if (value === 'low') return score <= 20;
      return false;
    }
    // ... more tag evaluators
    default:
      return false;
  }
}
```

## Open Questions

1. **Severity thresholds for `severity:high/medium/low`**
   - What we know: `computeSeverityScore()` returns a positive number, formula is `typeWeight * log2(1+mentions) * log2(1+sources) * recencyDecay`
   - What's unclear: The exact thresholds for high/medium/low bucketing have no prior art in the codebase
   - Recommendation: Start with high > 50, medium > 15, low <= 15. Adjust based on real data distribution. These are Claude's discretion.

2. **Squawk tag data availability**
   - What we know: Raw OpenSky data includes squawk (column 14) but it's not in FlightEntity.data
   - What's unclear: Whether to extend the data model in this phase or defer
   - Recommendation: Register `squawk:` tag but mark it as no-op (always passes) with a TODO comment. Data model extension is out of scope for this phase.

3. **Recent tags persistence format**
   - What we know: Last 5 searches' tags should appear as additional chips
   - What's unclear: Exact localStorage key and serialization format
   - Recommendation: Store as JSON array of tag strings in `recentSearchTags` localStorage key, managed by searchStore.

4. **Performance at scale: full AST evaluation per entity**
   - What we know: Flights can be 200+, events 500+, sites 60+. Each needs AST evaluation.
   - What's unclear: Whether recursive AST walk per entity causes frame drops
   - Recommendation: Profile after implementation. The AST is shallow (typical depth 2-4) and evaluation is simple comparisons. Expected to be fine. If slow, add memoization by entity ID.
