    - cacheGet returns { data, stale: true, lastFresh } when data exceeds logical TTL

    2. Create `server/cache/redis.ts`:
       - Import `Redis` from `@upstash/redis`
       - Create module-scope client: `new Redis({ url: process.env.UPSTASH_REDIS_REST_URL!, token: process.env.UPSTASH_REDIS_REST_TOKEN! })`
       - Define `CacheEntry<T>` interface: `{ data: T; fetchedAt: number }`
       - Export `async function cacheGet<T>(key: string, logicalTtlMs: number): Promise<CacheResponse<T> | null>` -- calls `redis.get<CacheEntry<T>>(key)`, returns null if no entry, computes `stale` from `Date.now() - entry.fetchedAt > logicalTtlMs`, returns `{ data: entry.data, stale, lastFresh: entry.fetchedAt }`
       - Export `async function cacheSet<T>(key: string, data: T, redisTtlSec: number): Promise<void>` -- creates `CacheEntry<T>` with `fetchedAt: Date.now()`, calls `redis.set(key, entry, { ex: redisTtlSec })`
       - Export `redis` client for direct access if needed
       - Import `CacheResponse` from `../types.js`

    3. Create `server/__tests__/redis-cache.test.ts`:
       - Use `// @vitest-environment node` header

       - The mock `get<T>(key)` returns the stored value (already deserialized, as Upstash does auto-JSON)
       - The mock `set(key, value, opts?)` stores the value in the Map
       - Test cases per the `<behavior>` block above
       - Use `vi.useFakeTimers()` to control `Date.now()` for staleness tests

    4. Delete `server/__tests__/cache.test.ts` (tests the EntityCache class which will be deleted in Plan 03).
